# 给用过 AI Coding 的工程师：怎么不被 AI 坑

> 不是教你用 AI，是教你管住 AI

## 你肯定遇到过这些

- 让 AI 重构一个函数，结果它顺手改了三个不相关的文件
- Prompt 没写清楚，AI 生成了能跑但逻辑全错的代码
- Demo 阶段一切正常，上线后边界情况全崩
- 用了某个第三方 Agent/Skill，不知道它在后台干了什么
- 项目越来越大，AI 生成的代码越来越难审，最后只能"感觉没问题就合了"

这些不是你的问题，是你缺了一套**管 AI 的框架**。

---

## 核心矛盾：你只告诉 AI 能做什么

大多数人用 AI Coding 的方式是：

```
写 Prompt → AI 生成 → 跑一下看看 → 不对就再改
```

这个循环本身没问题，但有一个致命盲区：**你只定义了 AI 的能力，没有定义 AI 的禁区。**

类比一下：你让实习生"帮我整理这个项目"，但没说"不要动 production 配置"、"不要提交 .env"、"不要删现有测试"。结果可想而知。AI 也一样。

---

## 两件事必须同时做

### 1. 能力暴露（Capability）：AI 能干什么

这个大家都在做——写 Prompt、配置 Skill、定义 workflow。

```markdown
# skill.md
功能：生成销售报表
输入：CSV 数据
输出：PDF 文件
调用：agent.run("生成销售报表", data)
```

### 2. 禁止空间（Forbidden Zone）：AI 绝对不能干什么

这个几乎没人做，但它才是防止翻车的关键。

```markdown
# redline.md
- 不允许删除任何现有文件
- 不允许读取 .env 或任何配置文件
- 不允许调用 skill.md 以外的外部 API
- 不允许修改 /src/core 目录（除非明确指定）
- 不允许执行任意 shell 命令
```

**没有 redline 的 AI 系统 = 高能力、零护甲。**

---

## 实践方案：先写 TestPlan，再让 AI 干活

核心思路只有一句话：

> **先定义"什么叫做完了"，再让 AI 去实现。**

不是写单元测试（AI 自己会生成），而是写高层验收标准：

```markdown
# TestPlan.md

## Must Have（必须通过，否则不算完成）
- [ ] 输入有效 CSV，能生成 PDF
- [ ] 报表数据与输入数据一致，无计算错误
- [ ] 调用超限时返回明确错误，不静默失败

## Need Have（重要，需要大部分通过）
- [ ] 支持按日期筛选
- [ ] 支持 Excel 格式输出

## Should Have（可选，有时间再做）
- [ ] 支持邮件发送
- [ ] 支持报表加密

## 禁止操作（Redline）
- 不允许读取报表数据以外的本地文件
- 不允许向外部服务发送原始数据
- 不允许删除输入的 CSV 文件
```

把这个文件作为上下文喂给 AI，让它在实现时自己对照。

---

## 为什么这比"写完再测"更有效

| 方式          | 问题                     |
| ----------- | ---------------------- |
| 写完再测        | AI 生成了大量代码，不知道从哪开始审    |
| 边写边改        | 没有基准，AI 每次改动方向不一致      |
| TestPlan 先行 | AI 有明确目标，你有明确验收标准，双向可控 |

另外，TestPlan.md 还有一个隐藏好处：**它是给 AI 的约束，也是给你自己的约束。** 写 TestPlan 的过程会逼你想清楚需求，而不是"先让 AI 做做看"。

---

## 实际工作流

```
1. 新功能开始前，先写 TestPlan.md（10-15 分钟）
   - Must Have / Need Have / Should Have
   - Redline（禁止操作清单）
   - 异常路径（输入为空？超限？文件不存在？）

2. 把 TestPlan.md 作为上下文喂给 AI
   "请参考 TestPlan.md 实现以下功能..."

3. AI 生成后，对照 Must Have 逐条验收
   不要只跑 happy path

4. 第三方 Skill / Agent 必须在沙盒环境测试
   不要在主力机或生产环境直接跑
```

---

## 一句话总结

> AI 不缺能力，缺的是你给它的边界。
> `TestPlan.md` = 告诉 AI 做什么 + 告诉 AI 不能做什么。
