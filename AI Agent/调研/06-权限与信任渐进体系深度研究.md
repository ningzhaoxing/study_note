# Claude Code 权限与信任渐进体系深度研究

> 研究对象：Claude Code CLI（Anthropic 官方 Agent 产品）
> 研究视角：Agent 产品设计 / 信任工程
> 研究日期：2026-04-01

---

## 第一部分：研究概述

### 1.1 核心问题

Agent 产品面临一个根本性的信任困境：

> **用户凭什么让一个 AI 执行 `rm -rf`、修改生产配置、或者 `git push --force`？**

这不是一个技术问题，而是一个**信任工程**问题。Claude Code 为此设计了一套极其精细的权限体系，它的核心思想不是"一刀切地限制 Agent"，而是**构建一条从"完全不信任"到"完全信任"的渐进路径，让用户按自己的节奏走完这条路**。

本报告基于 Claude Code 源码（`permissions.ts`、`PermissionMode.ts`、`permissionsLoader.ts`、`yoloClassifier.ts`、`bashClassifier.ts`、`interactiveHandler.ts`、`pathValidation.ts`、`permissionSetup.ts` 等核心文件），完整拆解这套信任体系的设计逻辑。

## 第二部分：权限模式全景图——六级信任阶梯

### 2.1 六种权限模式

Claude Code 定义了六种权限模式，形成一个清晰的信任阶梯：

```
信任程度  ░░░░░░░░░░████████████████████████████████████
          │        │          │           │          │          │
      Plan Mode  Default  Accept     Auto       Bypass    dontAsk
      (规划)     (默认)   Edits     (AI判断)   (全放行)  (全拒绝)
                          (放行编辑)

信任方向：  ──────────────────────────────────────────►
             低信任                              高信任

安全方向：  ◄──────────────────────────────────────────
             低安全                              高安全
```

**源码定义**（`PermissionMode.ts`）：

```
┌───────────────┬──────────┬──────────┬──────────────────────────────────┐
│ 模式           │ 视觉标识  │ 颜色      │ 行为                             │
├───────────────┼──────────┼──────────┼──────────────────────────────────┤
│ plan          │ ⏸ PAUSE  │ planMode │ 只读模式，不执行任何写入操作       │
│ default       │ (无)     │ text     │ 每个写操作都弹窗审批              │
│ acceptEdits   │ ⏵⏵      │ autoAccept│ 自动批准工作目录内的文件编辑      │
│ auto          │ ⏵⏵      │ warning  │ AI 分类器自动判断是否安全         │
│ bypassPerm.   │ ⏵⏵      │ error    │ 跳过所有权限检查                  │
│ dontAsk       │ ⏵⏵      │ error    │ 所有 ask 自动转为 deny           │
└───────────────┴──────────┴──────────┴──────────────────────────────────┘
```

### 2.2 每种模式的适用场景

**Plan Mode**（信任级别：0）：
- 场景：首次使用、大型重构前的方案评审
- 行为：Agent 只能读取/搜索代码，不能修改任何文件或执行任何命令
- 用户角色：决策者——审阅 Agent 的方案

**Default Mode**（信任级别：1）：
- 场景：日常编程、bug 修复
- 行为：读取自动放行，写入/执行需要逐个审批
- 用户角色：审批者——每个操作过一眼

**Accept Edits**（信任级别：2）：
- 场景：确认 Agent 的编辑能力后，希望减少审批频率
- 行为：工作目录内的文件编辑自动放行，Bash 命令仍需审批
- 用户角色：监督者——只关注命令执行

**Auto Mode**（信任级别：3，内部功能）：
- 场景：Anthropic 内部使用，AI 分类器自动判断
- 行为：AI 分类器评估每个操作的安全性，安全则自动放行
- 用户角色：被动监督——只在分类器无法判断时才被打扰

**Bypass Permissions**（信任级别：4）：
- 场景：完全信任 Agent，或在沙箱环境中运行
- 行为：跳过所有权限检查，所有操作自动执行
- 用户角色：不参与——完全放手

**Don't Ask**（信任级别：-1，特殊模式）：
- 场景：只想让 Agent 分析但不执行任何操作
- 行为：所有需要审批的操作自动拒绝
- 用户角色：严格限制者——只允许已明确授权的操作

### 2.3 信任阶梯的升级路径

```
新用户入门                                     专家用户
    │                                              │
    ▼                                              ▼
Plan Mode ──→ Default ──→ Accept Edits ──→ Auto ──→ Bypass
    │            │              │            │         │
    │         每次审批       编辑免审批    AI判断    全放行
    │         建立基础信任   减少疲劳     智能化    完全信任
    │                                                 │
    └─── 任何时候都可以回退到更低信任级别 ────────────┘
```

**关键设计原则**：
1. **升级是用户主动的**：系统不会自动提升信任级别
2. **降级是即时的**：随时可以回退到更低级别
3. **每级之间有明确的价值差异**：不是为了多几个选项而增加复杂度

## 第三部分：七层权限决策引擎

这是 Claude Code 权限系统的核心——一个 **7 步决策流程**，决定每个工具调用是 `allow`（放行）、`deny`（拒绝）还是 `ask`（询问用户）。

### 3.1 完整决策流程

**源码位置**：`permissions.ts` 中的 `hasPermissionsToUseToolInner()` 函数

```
┌─────────────────────────────────────────────────────────────────┐
│                   权限决策引擎（7 步流程）                        │
│                                                                  │
│  工具调用请求                                                     │
│      │                                                           │
│      ▼                                                           │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Step 1: Deny 检查（Bypass-Immune，不可跳过！）            │   │
│  │                                                          │   │
│  │  1a. 工具整体被 deny 规则拒绝？                           │   │
│  │      → deny（即使在 bypass 模式也生效）                   │   │
│  │                                                          │   │
│  │  1b. 工具整体有 ask 规则？                                │   │
│  │      → ask（除非是沙箱 bash，转 1c）                     │   │
│  │                                                          │   │
│  │  1c. 工具特定的权限检查（子命令规则）                      │   │
│  │      → 委托给工具自身的 checkPermissions()                │   │
│  │      例：Bash(prefix:npm) → 只检查 npm 开头的命令        │   │
│  │                                                          │   │
│  │  1d. 工具实现层拒绝？                                     │   │
│  │      → deny                                              │   │
│  │                                                          │   │
│  │  1e. 工具需要用户交互？（requiresUserInteraction=true）    │   │
│  │      → ask（即使在 bypass 模式！AskUserQuestion 永远要问）│   │
│  │                                                          │   │
│  │  1f. 内容级 ask 规则？（如 Bash(npm publish:*)）          │   │
│  │      → ask（Bypass-Immune！发布操作永远要确认）           │   │
│  │                                                          │   │
│  │  1g. 安全检查？（.git/、.claude/、.vscode/、shell 配置）  │   │
│  │      → ask（Bypass-Immune！敏感路径永远要确认）           │   │
│  └──────────────────────────────────────────────────────────┘   │
│      │ 如果以上都没有匹配                                       │
│      ▼                                                           │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Step 2: Allow 检查                                        │   │
│  │                                                          │   │
│  │  2a. 当前是 bypass 模式？                                  │   │
│  │      → allow                                              │   │
│  │                                                          │   │
│  │  2b. 工具整体被 allow 规则允许？                           │   │
│  │      → allow                                              │   │
│  └──────────────────────────────────────────────────────────┘   │
│      │ 如果以上都没有匹配                                       │
│      ▼                                                           │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Step 3: Passthrough 转 Ask                                │   │
│  │                                                          │   │
│  │  工具返回了 passthrough？                                  │   │
│  │      → 转为 ask（交给用户决定）                            │   │
│  └──────────────────────────────────────────────────────────┘   │
│      │                                                           │
│      ▼                                                           │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Step 4: 模式变换（对 ask 结果做二次处理）                  │   │
│  │                                                          │   │
│  │  4a. dontAsk 模式？                                       │   │
│  │      → ask 转为 deny（不要问我，直接拒绝）                │   │
│  │                                                          │   │
│  │  4b. auto 模式？                                          │   │
│  │      → 交给 AI 分类器（详见第五部分）                     │   │
│  │                                                          │   │
│  │  4c. 无头/异步 Agent？                                    │   │
│  │      → 运行 PermissionRequest Hooks                       │   │
│  │      → Hook 无决定 → auto-deny                            │   │
│  └──────────────────────────────────────────────────────────┘   │
│      │                                                           │
│      ▼                                                           │
│  最终决策：allow / deny / ask（弹窗给用户）                      │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 Bypass-Immune 机制——"即使全放行也不能跳过的检查"

这是整个权限系统中最重要的设计决策之一：

**某些安全检查在任何权限模式下都不能被跳过**，包括：

1. **`requiresUserInteraction`**：AskUserQuestion 等需要用户输入的工具，即使在 bypass 模式也必须等用户回答
2. **内容级 ask 规则**：如 `Bash(npm publish:*)`，发布到 npm 这种不可逆操作永远需要确认
3. **敏感路径保护**：`.git/`、`.claude/`、`.vscode/`、shell 配置文件（`.bashrc`、`.zshrc`）的修改永远需要确认

**为什么要有 Bypass-Immune？**

```
场景：用户开启了 bypass 模式（全放行）

没有 Bypass-Immune 的情况：
  Agent 执行 `rm -rf .git/` → 自动放行 → git 历史全部丢失 😱
  Agent 执行 `echo 'alias rm=rm -rf' >> ~/.zshrc` → 自动放行 → shell 被污染 😱
  Agent 执行 `npm publish --access public` → 自动放行 → 代码意外发布 😱

有 Bypass-Immune 的情况：
  Agent 执行 `rm -rf .git/` → 安全检查拦截 → 弹窗询问 → 用户阻止 ✅
  Agent 执行修改 ~/.zshrc → 敏感路径拦截 → 弹窗询问 → 用户判断 ✅
  Agent 执行 npm publish → 内容级规则拦截 → 弹窗确认 → 用户决定 ✅
```

**产品设计洞察**：
> Bypass-Immune 的哲学是"信任不是绝对的"。即使用户选择了最高信任级别，系统仍然在最危险的操作前设置了"最后一道防线"。这不是不信任用户，而是保护用户不因为一时疏忽造成不可逆损失。这跟银行转账的二次确认是同一个逻辑。

## 第四部分：权限规则系统

### 4.1 规则匹配语法

Claude Code 设计了一套简洁但强大的规则匹配语法：

```
规则匹配模式：

工具级别匹配：
  "Bash"                    → 匹配 Bash 工具的所有调用
  "FileEdit"                → 匹配文件编辑工具的所有调用
  "mcp__server1"            → 匹配来自 MCP server1 的所有工具

内容级别匹配（精确控制）：
  "Bash(prefix:npm)"        → 匹配以 npm 开头的命令
  "Bash(prefix:git push)"   → 匹配 git push 命令
  "Bash(npm publish:*)"     → 匹配 npm publish 的所有变体

MCP 通配符：
  "mcp__server1__*"          → 匹配 server1 的所有工具

Agent 级别匹配：
  "Agent(Explore)"           → 匹配 Explore 类型的子 Agent
```

**规则行为**：每条规则有三种行为——`allow`（放行）、`deny`（拒绝）、`ask`（询问）

### 4.2 八层规则优先级

规则来源有八层，优先级从高到低：

```
┌─────────────────────────────────────────────────────────┐
│                八层规则优先级（从高到低）                  │
│                                                          │
│  Layer 1: policySettings（MDM 管控策略）                  │
│  ├─ 来源：/etc/claude-code/policy.json 或 MDM 下发       │
│  ├─ 可编辑：❌ 只读                                      │
│  └─ 特点：最高优先级，企业可以强制限制行为               │
│                                                          │
│  Layer 2: flagSettings（CLI 标志设置）                    │
│  ├─ 来源：命令行 --flag 参数                              │
│  ├─ 可编辑：❌ 只读                                      │
│  └─ 特点：启动时指定的临时规则                           │
│                                                          │
│  Layer 3: command（命令级规则）                           │
│  ├─ 来源：特定命令内置的规则                              │
│  ├─ 可编辑：❌ 只读                                      │
│  └─ 特点：某些命令可以附带权限约束                       │
│                                                          │
│  Layer 4: session（会话级规则）                           │
│  ├─ 来源：用户在对话中通过 "Always Allow" 添加            │
│  ├─ 可编辑：⚠️ 仅内存中（会话结束即清除）               │
│  └─ 特点：临时信任，不持久化                             │
│                                                          │
│  Layer 5: cliArg（命令行参数）                            │
│  ├─ 来源：启动命令的参数                                  │
│  ├─ 可编辑：⚠️ 仅内存中                                 │
│  └─ 特点：本次启动生效                                   │
│                                                          │
│  Layer 6: localSettings（本地项目设置）                   │
│  ├─ 来源：.claude/settings.json（项目目录下）             │
│  ├─ 可编辑：✅ 可编辑                                    │
│  └─ 特点：项目级规则，可加入 .gitignore                  │
│                                                          │
│  Layer 7: projectSettings（项目共享设置）                 │
│  ├─ 来源：.claude/settings.json（提交到 git 的版本）      │
│  ├─ 可编辑：✅ 可编辑                                    │
│  └─ 特点：团队共享的规则                                 │
│                                                          │
│  Layer 8: userSettings（用户全局设置）                    │
│  ├─ 来源：~/.claude/settings.json                         │
│  ├─ 可编辑：✅ 可编辑                                    │
│  └─ 特点：用户所有项目通用的规则                         │
│                                                          │
│  优先级解析：                                             │
│  ├─ 同层级内：deny > ask > allow                         │
│  └─ 跨层级：高层级规则覆盖低层级                         │
└─────────────────────────────────────────────────────────┘
```

### 4.3 规则设计的三个核心取舍

**取舍一：企业管控 vs 开发者自由**

```
policySettings（Layer 1）不可覆盖
  → 企业可以通过 MDM 强制禁止某些操作
  → 例如：deny Bash(prefix:rm -rf /) → 全公司禁止删除根目录
  → 开发者无法绕过这条规则，即使开启 bypass 模式

allowManagedPermissionRulesOnly 开关：
  → 启用后，只有 policySettings 的规则生效
  → 所有其他来源的规则被清空
  → "Always Allow" 选项从 UI 中隐藏
  → 适用于高安全要求的企业环境
```

**取舍二：持久化 vs 临时性**

```
持久化规则（存磁盘，跨会话生效）：
  ├─ userSettings → ~/.claude/settings.json
  ├─ projectSettings → .claude/settings.json
  └─ localSettings → .claude/settings.json（本地）

临时规则（仅内存，会话结束清除）：
  ├─ session → 用户点 "Always Allow" 生成
  └─ cliArg → 启动参数

设计意图：
  → 用户首次允许某操作时，默认添加 session 级规则（临时）
  → 如果用户确定长期需要，可以持久化到 settings.json
  → 避免"一时放行"变成"永久漏洞"
```

**取舍三：粒度 vs 简洁性**

```
粗粒度规则：
  "Bash" → allow（允许所有 bash 命令）
  → 简单但危险，等于完全放行 shell

细粒度规则：
  "Bash(prefix:npm install)" → allow（只允许 npm install）
  "Bash(prefix:git commit)" → allow（只允许 git commit）
  → 安全但繁琐，需要一条条配置

Claude Code 的折中：
  → 默认提供常见操作的 Suggestion（建议规则）
  → 用户点一下就能添加精确规则
  → 不需要手动写规则语法
```

## 第五部分：AI 行为分类器

### 5.1 分类器架构

Auto Mode 下，AI 分类器代替人类做权限判断。Claude Code 实现了两种分类器：

```
┌─────────────────────────────────────────────────────────┐
│                AI 分类器决策架构                          │
│                                                          │
│  权限请求（ask 结果）                                     │
│      │                                                   │
│      ▼                                                   │
│  ┌──────────────────────────────────────────────────┐   │
│  │ Fast Path 快速通道（跳过分类器）                    │   │
│  │                                                    │   │
│  │ 1. acceptEdits 会放行的操作 → 直接 allow           │   │
│  │ 2. 安全工具白名单：                                │   │
│  │    FileRead, Grep, Glob, LSP, ToolSearch,          │   │
│  │    TaskCreate/Get/List/Update,                      │   │
│  │    EnterPlanMode, ExitPlanMode                      │   │
│  │    → 直接 allow                                     │   │
│  └────────────────────┬─────────────────────────────┘   │
│                       │ 不在快速通道中                    │
│                       ▼                                   │
│  ┌──────────────────────────────────────────────────┐   │
│  │ YOLO 分类器（主分类器）                            │   │
│  │                                                    │   │
│  │ 两阶段评估：                                       │   │
│  │ Stage 1: Fast Stage                                │   │
│  │   └─ 快速规则匹配 → 明确安全/危险                  │   │
│  │                                                    │   │
│  │ Stage 2: Thinking Stage                            │   │
│  │   └─ XML 格式深度分析                              │   │
│  │   └─ 考虑对话上下文（transcript）                   │   │
│  │   └─ 输出：shouldBlock + reason                    │   │
│  └────────────────────┬─────────────────────────────┘   │
│                       │                                   │
│                       ▼                                   │
│  ┌──────────────────────────────────────────────────┐   │
│  │ Bash 分类器（命令专用）                            │   │
│  │                                                    │   │
│  │ 针对 Bash 命令的专用安全评估：                     │   │
│  │ ├─ 解析命令结构                                    │   │
│  │ ├─ 匹配危险模式（rm, chmod, kill, etc.）           │   │
│  │ ├─ 评估参数安全性                                  │   │
│  │ └─ 返回 allow/deny/ask                             │   │
│  └──────────────────────────────────────────────────┘   │
│                                                          │
│  分类器结果：                                            │
│  ├─ shouldBlock: false → allow                          │
│  ├─ shouldBlock: true → deny（附带原因）                │
│  └─ unavailable: true → 降级处理（见 5.3）              │
└─────────────────────────────────────────────────────────┘
```

### 5.2 分类器的自我保护机制

**Denial Limits（拒绝限制）**：

```typescript
const DENIAL_LIMITS = {
  maxConsecutive: 3,   // 连续 3 次拒绝 → 降级到手动审批
  maxTotal: 20,        // 累计 20 次拒绝 → 降级到手动审批
}
```

**设计逻辑**：如果分类器连续拒绝 3 次或累计拒绝 20 次，说明可能存在系统性问题（分类器过于严格，或者 Agent 在尝试做它不该做的事）。此时系统自动降级到手动审批模式，让人类介入判断。

**Fail-Closed 原则**：

```
分类器不可用时（网络问题、API 错误等）：

Feature Gate: tengu_iron_gate_closed（默认 = true）
  ├─ true（默认）: 分类器不可用 → deny（拒绝 + 告知用户）
  └─ false: 分类器不可用 → 降级到正常权限流程（弹窗询问）

设计意图：
  → "宁可误拒也不能误放"
  → 分类器的缺席不能成为安全漏洞
```

### 5.3 分类器的异步竞态设计

当用户在 Default Mode 下，分类器可以在后台异步运行：

```
权限请求
  ↓
弹出审批对话框（用户可见）
  ↓ 同时
启动分类器后台评估（用户不可见）
  │
  ├─ 分类器先完成，判定安全
  │   → 对话框自动消失 → allow
  │   → 用户体验："哦，它自己通过了"
  │
  ├─ 用户先做出决定
  │   → 分类器结果丢弃
  │   → 用户体验："我手动处理了"
  │
  └─ 分类器先完成，判定危险
      → 对话框保持（等用户决定）
      → 用户体验：看到风险提示后自行判断
```

**`pendingClassifierCheck`**：当分类器尚在评估中，权限对话框会显示一个"⏳ Classifier check pending..."指示器。这让用户知道系统正在自动评估，可以选择等一下或者直接手动决定。

## 第六部分：安全检查体系——不可妥协的底线

### 6.1 敏感路径保护

```
敏感路径分两类：

Type A: 分类器可审批的（classifierApprovable = true）
  ├─ .claude/         → Claude 自身的配置
  ├─ .git/            → Git 仓库数据
  ├─ .vscode/         → IDE 设置
  ├─ .bashrc          → Shell 配置
  ├─ .zshrc           → Shell 配置
  ├─ .profile         → Shell 配置
  └─ .ssh/config      → SSH 配置

  → 在 auto 模式下，分类器可以判断是否放行
  → 在其他模式下，必须弹窗询问

Type B: 不可审批的（classifierApprovable = false）
  ├─ Windows 路径绕过尝试
  └─ 跨机器桥接消息

  → 任何模式下都必须弹窗询问
  → 分类器不能代替人类判断
```

### 6.2 危险权限检测与剥离

**源码位置**：`permissionSetup.ts`

当用户从 Default Mode 切换到 Auto Mode 时，系统会自动检测并剥离"危险权限"：

```
危险权限定义：

Bash 相关：
  ├─ "Bash" → allow（允许所有命令，极度危险）
  ├─ "Bash(prefix:python)" → allow（允许执行任意 Python 代码）
  ├─ "Bash(prefix:node)" → allow（允许执行任意 Node 代码）
  └─ "Bash(prefix:ruby)" → allow（允许执行任意 Ruby 代码）

PowerShell 相关：
  ├─ "PowerShell" → allow
  ├─ "PowerShell(prefix:iex)" → allow（Invoke-Expression）
  └─ "PowerShell(prefix:Start-Process)" → allow

Task 相关：
  └─ "TaskStop(*)" → allow（允许终止任何任务）
```

**剥离逻辑**：

```
进入 Auto Mode
  ↓
扫描所有 allow 规则
  ↓
匹配危险权限模式
  ├─ 匹配到 → 从 allow 规则中移除
  │           → 存储到 strippedDangerousRules（可恢复）
  └─ 未匹配 → 保留
  ↓
退出 Auto Mode 时
  ↓
从 strippedDangerousRules 恢复被剥离的规则
```

**产品设计洞察**：
> 危险权限检测解决了一个实际问题：用户在 Default Mode 下可能为了方便添加了宽泛的 allow 规则（比如 "允许所有 Bash 命令"），这在有人审批时问题不大。但切换到 Auto Mode 后，这些宽泛规则就变成了安全漏洞——AI 分类器会直接放行所有命令。自动剥离这些规则，确保 Auto Mode 的安全底线不被人为降低。

## 第七部分：Hook 驱动的权限扩展

### 7.1 PreToolUse Hooks

```
PreToolUse Hook 可以返回三种决策：

1. allow：允许操作（可附带修改后的 input）
   → Agent 继续执行
   → 可选：附带 updatedPermissions（持久化新规则）

2. deny：拒绝操作（附带 systemMessage）
   → Agent 收到拒绝 + 原因
   → 自动调整策略

3. passthrough：不做决定，交给下一环节
   → 进入正常权限流程
```

### 7.2 PermissionRequest Hooks（无头模式专用）

```
无头模式下，无法弹窗给用户
  ↓
运行 PermissionRequest Hooks
  ↓
Hook 可以做：
  ├─ 查询外部审批系统（如 PagerDuty、Slack）
  ├─ 基于策略自动决定
  ├─ 记录审计日志
  └─ 返回 allow/deny

  如果所有 Hook 都返回 passthrough
    ↓
  auto-deny（安全降级）
```

**产品设计洞察**：
> Hook 机制让权限系统变得可扩展。企业可以接入自己的审批系统、安全策略引擎、审计日志系统，而不需要修改 Claude Code 的源码。这是"产品平台化"的关键能力。

## 第十一部分：产品设计启示

### 11.1 Agent 信任体系设计的五条原则

**原则一：信任是连续变量，不是布尔值**

不要只设计"开/关"两种模式。至少需要 3-5 个信任级别，覆盖从"完全不信任"到"完全信任"的全部场景。每个级别之间的差异应该是明确的、有产品价值的。

**原则二：即使在最高信任级别，也要有不可跳过的安全检查**

Bypass-Immune 不是对用户的不信任，而是对不可逆操作的保护。删除 git 历史、发布到公共仓库、修改 shell 配置——这些操作的后果可能是灾难性的，多一次确认的成本远低于事故恢复的成本。

**原则三：规则要可持久化、可分层、可共享**

- 可持久化：用户配置一次，永久生效（不用每次重新配置）
- 可分层：个人规则、项目规则、企业规则各有各的优先级
- 可共享：团队可以共享一套安全规则（通过 projectSettings）

**原则四：AI 分类器是"减少审批疲劳"的关键**

如果每个操作都要人工审批，用户很快就会养成"无脑点 Allow"的习惯——这比没有审批更危险。AI 分类器自动放行安全操作、只在危险操作时提醒，才能让审批机制真正发挥作用。

**原则五：权限决策必须有解释**

用户需要知道"为什么弹出这个对话框"（决策原因）和"这个操作有多危险"（风险等级）。没有解释的审批对话框只会让用户困惑和厌烦。

### 11.2 信任体系的建设优先级

```
P0 - 最小可行信任体系：
  ├─ Default Mode（逐个审批）
  ├─ Always Allow（会话级临时规则）
  └─ 基本的敏感路径保护

P1 - 减少审批疲劳：
  ├─ Accept Edits Mode（编辑自动放行）
  ├─ 持久化规则（settings.json）
  └─ 风险等级标注

P2 - 智能化：
  ├─ AI 分类器（自动判断安全性）
  ├─ 规则建议（Suggestions）
  └─ 异步竞态审批

P3 - 企业级：
  ├─ MDM 管控
  ├─ 八层规则优先级
  └─ 审计日志 + 分析
```

## 总结

Claude Code 的权限体系不是"一个审批弹窗"，而是一个**完整的信任工程系统**。它的设计深度体现在：

1. **六级信任阶梯**：从 Plan Mode 到 Bypass，覆盖了所有信任场景
2. **七步决策引擎**：每个权限请求经过严格的多步评估，而非简单的 if-else
3. **八层规则优先级**：从企业 MDM 到会话临时规则，每层有明确的定位
4. **Bypass-Immune 设计**：即使全放行，敏感操作仍需确认——这是"安全底线"思维
5. **AI 分类器 + 竞态审批**：减少审批疲劳，同时保持安全
6. **企业管控能力**：MDM、Managed Mode、审计日志——这是 B2B 场景的必要能力

这套体系的本质思想是：**让用户在安全的框架内，按自己的节奏建立对 Agent 的信任**。系统不强迫信任，也不阻止信任，而是提供一条清晰的、可回退的、有安全保障的信任升级路径。

> **一句话总结**：Claude Code 的权限体系是"渐进式信任 + 不可妥协的安全底线 + AI 辅助减负"三位一体的工业级信任工程实现。
