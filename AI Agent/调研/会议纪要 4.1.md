### 十、CLAUDE.md与Context体系搭建（核心讨论）

#### 10.1 Context隔离问题

王子阳提出：如果TARA和HARA在同一个仓库，AI可能会"犯之前犯过的错误"——同一个仓库里AI会沿用之前的（可能错误的）设计模式。

孙志东回应：每个app下增加自己的CLAUDE.md来限制。

王子阳补充CLAUDE.md的层级机制：

- 子文件夹的CLAUDE.md会覆盖上级CLAUDE.md（有冲突时子级优先）
    
- 应该把TARA专属的规则从根目录移到apps/tara/下
    
- 通用规则保留在全局CLAUDE.md
    
- HARA有自己独立的CLAUDE.md
    

孙志东确认："我应该把现在的CLAUDE.md放到TARA下面，把里面通用的东西放在全局，HARA有自己的CLAUDE.md。"

#### 10.2 Harness Engineering中的Context定位

王子阳阐述核心观点：

> "走Harness的话，Context是个非常重要的节点。在Harness Engineering概念里，不在Context里面的东西就等于没有。"
> 
> "AI不会去联网做任何research，AI所有的对业务的理解都要来自于Context仓库。他们提出的观点叫做**'仓库即真相'**。"

Context仓库的内容：

- 所有业务要求
    
- 预设假设（assumption）
    
- 架构设计
    
- 必须及时、跟上代码库
    
- AI不会去查Slack，不会查其他东西，所有项目业务上下文和技术方案都从这个仓库获取
    

#### 10.3 CLAUDE.md vs Context仓库的区别

王子阳澄清：

|      |                      |                                     |
| ---- | -------------------- | ----------------------------------- |
|      | CLAUDE.md            | Context仓库                           |
| 性质   | 代码规范、技术规则            | 业务上下文、需求、设计                         |
| 服务对象 | 代码研发Agent            | 所有Agent（PM Agent、技术方案Agent、研发Agent） |
| 内容   | 代码规范、测试规范、commit风格等  | PRD、技术方案、会议纪要、架构决策等                 |
| 价值   | 不怎么值钱（决定了Agent怎么写代码） | 真正值钱的（业务知识资产）                       |

王子阳原话：

> "CLAUDE.MD它其实不怎么值钱，它只是决定了Claude要怎么帮你写那些Context。真正值钱的是你的Context仓库以及对应弄好了的索引。"

#### 10.4 CLAUDE.md中应包含的维护规则

王子阳建议在CLAUDE.md中写入：

- 每次发完技术方案后，必须及时更新索引index的MD
    
- 规定项目代码结构：PRD放哪、补充的会议纪要放哪、其他资料放哪
    

#### 10.5 多Agent协作流程

王子阳描述的目标工作流：

```Plain
项目经理Agent → 写PRD → 放入Context仓库 → 更新索引
                                    ↓
技术方案Agent → 读PRD → 生成技术方案 → 放入Context仓库 → 更新索引
                                    ↓
代码研发Agent → 读PRD + 技术方案 → 开始开发
                （不在仓库里的不做，不联网research）
```

#### 10.6 人在Harness中的定位

李乐嘉总结：人的定位有两个：

1. Context工程师：维护和设计Context
    
2. 关键节点的审批与纠偏：在人机协作节点提供审批或纠偏
    

王子阳补充：

> "Harness英文叫马具，意思是把AI比作跑得很快的马，但你需要给它套上马具来控制它行进的方向。"
> 
> "以后人不要写任何代码，全让Agent写，人只负责把控方向以及提供约束。"

### 十一、知识库检索方案讨论

#### 11.1 Claude Code的检索策略

王子阳介绍Claude Code放弃传统RAG的原因：

传统RAG：

```Plain
文档 → chunk化 → 向量化 → 向量数据库 → 用户query → 语义匹配 → 结果 + query → 大模型生成
```

Claude Code采用的Just-in-time RAG / Agentic Search：

```Plain
大模型自己决定"我还缺什么信息" → 用工具（Grep、Glob、Read等）自己搜索 → 反思 → 继续
```

核心观点：随着模型越来越聪明，模型开始自己意识到需要什么信息，而不是让人替它决定。

#### 11.2 Claude Code的5个查询工具

- Glob（文件模式匹配）
    
- Grep（内容搜索）
    
- Read（文件读取）
    
- Agent/Explore（子Agent探索）
    
- Agentic Search（复杂自然语言搜索，组合前几个工具+反思）
    

王子阳发现：基本上是文字搜索而不是语义搜索。

曹颖杰补充：Agentic Search应该是先组合前面的工具（正则等），然后再通过获取到的内容反思一遍。

#### 11.3 工具复用建议

曹颖杰提出：

> "如果我们用Sonnet或Opus模型，要尽可能复用它的工具，因为这些工具会在它们的后训练阶段加入模型的能力上面。如果是我们外挂的能力，它的支持度没有原生的好——这是后训练里直接习得的能力。"

胡俊涛提到：目前TARA的知识库向量化和RAG是自己后来实现的，可以考虑改成大模型语义检索。

#### 11.4 分级检索方案

曹颖杰建议：可以做分级——大的目录或树状结构，通过结构去管理，不需要通过固定的语义相似度做匹配。

结论：这块由王子阳后续深入研究，明天晚上有专门的Claude Code讨论会。

### 十二、Harness Engineering三支柱讨论

#### 12.1 团队对三支柱的理解

|   |   |   |
|---|---|---|
|支柱|内容|团队掌握情况|
|Context Engineering|渐进式信息披露，仓库即真相|比较了解|
|Mechanical Constraints|机械约束（架构规则、CI/CD强制执行）|比较了解|
|Entropy Management|定期清理技术债，自动化回收重构|刚刚意识到重要性|

#### 12.2 第三支柱：熵管理的重要性

王子阳强调：

> "Harness文章提到要做定期清理技术债，不然AI会不断重复之前的技术债，让技术债越垒越多。定期清理非常重要。" "之前犯的一些错、技术债，AI很可能自动沿用，然后让技术债越来越大甩不掉。所以这个定期的自动化回收重构也是需要做的。"

李乐嘉回应：Context仓库已有一个Skill专门做Context维护（repo-maintain），但做得不是很好，需要加强。
