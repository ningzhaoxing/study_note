# Claude Code 人机协作交互模式深度研究

> 研究对象：Claude Code CLI（Anthropic 官方 Agent 产品）
> 研究视角：Agent 产品设计 / 人机协作范式
> 研究日期：2026-04-01

---

## 第一部分：研究概述与方法论

### 1.1 研究问题

Agent 产品设计中有一个核心矛盾：**自动化程度越高，用户越省力，但失控风险也越大；人工干预越多，越安全可控，但效率优势就消失了。**

传统的解法是二选一——要么做成"全自动脚本"，要么做成"人工审批流"。Claude Code 的做法完全不同：它设计了**五种人机协作模式**，形成一个从"完全人控"到"完全自治"的连续谱系，让用户根据场景和信任程度自由选择。

本报告基于 Claude Code 源码（`EnterPlanModeTool.ts`、`ExitPlanModeV2Tool.ts`、`AskUserQuestionTool.tsx`、`interactiveHandler.ts`、`REPL.tsx`、`coordinatorMode.ts`、`StreamingToolExecutor.ts` 等核心文件），深度拆解这套人机协作体系的设计逻辑。

### 1.2 研究方法

- **源码逆向分析**：阅读交互相关的全部核心文件，梳理状态转换和数据流
- **模式矩阵建模**：将五种协作模式抽象为统一的控制权分配矩阵
- **产品设计归因**：从"为什么这样设计"的角度分析每个决策背后的产品逻辑
- **竞品对比**：与 Cursor、GitHub Copilot Workspace、OpenAI Agents SDK 的交互模式对比

## 第二部分：五种人机协作模式的完整矩阵

Claude Code 在源码中实现了五种截然不同的协作模式，每种模式定义了不同的控制权分配方式：

```
┌─────────────────────────────────────────────────────────────────┐
│                    Claude Code 人机协作模式谱系                    │
│                                                                  │
│  人控程度  ████████████████████████████████████████░░░░░░  自治程度│
│            │          │          │          │          │          │
│        Plan Mode  Default   Background  Headless  Coordinator   │
│        (规划模式)  Mode      Mode        Mode      Mode         │
│                   (默认)    (后台)      (无头)    (协调者)       │
│                                                                  │
│  人的角色：决策者 → 审批者 → 监督者 → 不在场 → 项目经理         │
│  Agent角色：顾问 → 执行者 → 自主工作者 → 全自动 → 团队领导     │
└─────────────────────────────────────────────────────────────────┘
```

### 2.1 Plan Mode（规划模式）—— "先想清楚再动手"

**角色分配**：人 = 决策者，Agent = 方案设计师

**核心设计**：Agent 只能读取和探索代码，不能做任何修改。所有方案必须经人类审批后才能执行。

**源码实现**（`EnterPlanModeTool.ts`）：

```
进入 Plan Mode 的状态转换：

Agent 调用 EnterPlanMode
  ↓
checkPermissions() → { behavior: 'ask', message: 'Enter plan mode?' }
  ↓
用户确认
  ↓
handlePlanModeTransition(currentMode, 'plan')
  ↓
状态变更：
  ├─ toolPermissionContext.mode = 'plan'
  ├─ 存储 prePlanMode（退出时恢复）
  └─ 如果原模式是 auto → 激活分类器
  ↓
可用工具集收缩：
  ├─ ✅ 可用：Read, Glob, Grep, Agent(Explore), WebSearch, WebFetch
  ├─ ❌ 禁用：Write, Edit, Bash（有副作用的工具）
  └─ ❌ 禁用：NotebookEdit, PowerShell
```

**退出 Plan Mode 的设计**（`ExitPlanModeV2Tool.ts`）：

```
Agent 调用 ExitPlanMode(allowedPrompts?)
  ↓
读取方案：getPlan(agentId)
  ↓
方案来源判断：
  ├─ Web UI 编辑版（input.plan）→ 用户修改后的方案
  └─ 磁盘存储版 → Agent 原始方案
  ↓
团队模式检查：
  ├─ isPlanModeRequired() = true
  │   → 发送 plan_approval_request 到 team lead 邮箱
  │   → 等待审批
  └─ isPlanModeRequired() = false
      → 本地退出，无需审批
  ↓
权限恢复：
  ├─ 恢复 prePlanMode（default/auto/etc）
  ├─ auto 模式：剥离危险权限
  └─ 写入类工具重新启用
```

**`allowedPrompts` 机制**：退出 Plan Mode 时，Agent 可以声明自己需要哪些类别的操作权限（如 "run tests"、"install dependencies"），这不是具体命令，而是**语义级别的权限请求**。这让用户能在更高抽象层次上授权。

**产品设计洞察**：
> Plan Mode 解决了 Agent 产品的"信任冷启动"问题。用户第一次使用 Agent 时，不可能立即信任它去修改代码。Plan Mode 让用户先看到 Agent "怎么想"，确认思路对了再放手。这跟新员工入职先做方案、审批通过再执行是一个逻辑。

### 2.2 Default Mode（默认交互模式）—— "一事一审批"

**角色分配**：人 = 审批者，Agent = 执行者

**核心设计**：Agent 自由思考和规划，但每次执行有副作用的操作前必须请求用户审批。读取类操作自动放行。

**交互循环**：

```
┌──────────────────────────────────────────────────┐
│                Default Mode 交互循环              │
│                                                   │
│  用户输入任务                                      │
│    ↓                                               │
│  Agent 思考 + 规划                                 │
│    ↓                                               │
│  Agent 调用工具                                    │
│    ├─ Read/Glob/Grep → 自动放行 ✅                 │
│    ├─ Edit/Write → 弹出审批对话框 ⚠️              │
│    │   ├─ 用户 Allow → 执行                        │
│    │   ├─ 用户 Always Allow → 本次会话免审         │
│    │   └─ 用户 Deny → Agent 调整方案               │
│    └─ Bash → 弹出审批对话框 ⚠️                    │
│        ├─ 显示具体命令内容                          │
│        ├─ 风险等级标注（LOW/MEDIUM/HIGH）          │
│        └─ 建议的权限更新（Suggestions）             │
│    ↓                                               │
│  工具执行 → 结果返回                               │
│    ↓                                               │
│  Agent 继续或完成                                  │
└──────────────────────────────────────────────────┘
```

**权限审批的竞态设计**（`interactiveHandler.ts`）：

这是一个非常精妙的设计——审批对话框弹出的同时，后台 AI 分类器也在异步评估这个操作是否安全：

```
┌──────────────────────────────────────────────────┐
│              权限审批竞态机制                       │
│                                                   │
│  工具请求权限                                      │
│    ↓                                               │
│  创建 resolveOnce 守卫（防止重复决策）             │
│    ↓                                               │
│  并行启动两个赛道：                                │
│                                                   │
│  赛道 A（用户交互）      赛道 B（AI 分类器）       │
│  ┌─────────────────┐    ┌─────────────────┐       │
│  │ 弹出审批对话框   │    │ 异步运行分类器   │       │
│  │ 显示工具详情     │    │ 评估操作安全性   │       │
│  │ 等待用户决定     │    │ 返回 allow/deny  │       │
│  └────────┬────────┘    └────────┬────────┘       │
│           │                      │                 │
│           └──── 先到先得 ────────┘                 │
│                    ↓                               │
│           resolveOnce 锁定结果                     │
│                    ↓                               │
│           200ms 防抖（防止误触）                    │
│                    ↓                               │
│           执行最终决策                              │
└──────────────────────────────────────────────────┘
```

**关键细节**：
- `resolveOnce` 守卫确保不管哪个赛道先完成，决策只执行一次
- 200ms 防抖防止用户在对话框刚弹出时误按键盘
- 如果分类器在用户响应前判定安全 → 对话框自动消失，操作自动执行
- 如果用户在分类器完成前做出决定 → 用户决定优先，分类器结果丢弃

**产品设计洞察**：
> 这个竞态机制的妙处在于：如果你是熟练用户，分类器帮你自动放行，减少审批疲劳；如果你是谨慎用户，你的手动决定永远优先。同一个机制适配了两种用户心态，而且两种路径之间的切换对用户完全透明。

### 2.3 Background Mode（后台自治模式）—— "你忙你的，我忙我的"

**角色分配**：人 = 监督者，Agent = 独立工作者

**核心设计**：Agent 在后台独立工作，完成后通知用户。用户可以同时进行其他工作。

**实现机制**（`AgentTool` 的 `run_in_background` 参数）：

```
用户发起任务
  ↓
Agent 调用 AgentTool(run_in_background=true)
  ↓
创建后台子 Agent：
  ├─ 独立的 context window（不与主 Agent 共享）
  ├─ 独立的 abort controller
  ├─ 可选：worktree 隔离（git worktree 独立分支）
  └─ 可选：isolation="worktree"（完全隔离的代码副本）
  ↓
主 Agent 继续响应用户
  ↓
后台 Agent 独立工作...
  ├─ 可以读取文件、执行命令
  ├─ 权限检查：shouldAvoidPermissionPrompts = true
  │   → 尝试 PermissionRequest Hooks
  │   → 失败则 auto-deny（不会弹对话框打断用户）
  └─ 完成后返回结果
  ↓
用户收到通知：
  "Agent 'xxx' completed"
  ↓
用户可以查看结果或继续对话
```

**多 Agent 并行**：

```
用户："帮我同时做这三件事"
  ↓
主 Agent 并行启动 3 个后台 Agent：
  ├─ Agent A: 研究 API 设计（Explore 模式）
  ├─ Agent B: 修复 bug（worktree 隔离）
  └─ Agent C: 编写测试（worktree 隔离）
  ↓
三个 Agent 在各自独立的上下文中并行工作
  ↓
陆续完成，逐个通知用户
  ↓
主 Agent 汇总结果
```

**Worktree 隔离的意义**：

当多个 Agent 需要同时修改代码时，如果都在同一个工作目录操作会产生冲突。Worktree 隔离（`isolation: "worktree"`）为每个 Agent 创建独立的 git worktree，每个 Agent 在自己的分支上工作，完成后合并。这是一个极其务实的工程设计。

**产品设计洞察**：
> Background Mode 解决了 Agent 产品的"阻塞焦虑"——用户不想干等 Agent 完成一个长任务。这跟 IDE 的后台编译、CI/CD 的异步构建是同一个产品逻辑：让用户在等待中保持生产力。Worktree 隔离则是从 git 工作流中借鉴的思路，非常适合并行开发场景。

### 2.4 Headless Mode（无头模式）—— "人不在，Agent 自己干"

**角色分配**：人 = 不在场，Agent = 全自动

**核心设计**：没有终端 UI，没有人可以交互。Agent 必须自主完成所有决策，包括权限判断。

**适用场景**：CI/CD 集成、定时任务、API 调用、GitHub Actions

**源码实现**（`isNonInteractiveSession` 标志）：

```
Headless Mode 的行为差异：

┌─────────────────────────────────────────────────┐
│              功能         │ 交互模式 │ 无头模式 │
├─────────────────────────────────────────────────┤
│ AskUserQuestion           │    ✅    │    ❌    │
│ 权限审批对话框             │    ✅    │    ❌    │
│ Ctrl+C 中断               │    ✅    │    ❌    │
│ 外部编辑器（Ctrl+G）      │    ✅    │    ❌    │
│ 终端 UI 渲染              │    ✅    │    ❌    │
│ AI 分类器自动放行          │    ✅    │    ✅    │
│ Channel Relay（飞书/Slack）│    ❌    │    ✅    │
│ PermissionRequest Hooks   │    ❌    │    ✅    │
│ Persistent Retry（无限重试）│    ❌    │    ✅    │
└─────────────────────────────────────────────────┘
```

**权限决策的降级链**：

```
Headless 模式下，权限请求的处理：

权限请求
  ↓
shouldAvoidPermissionPrompts = true
  ↓
① 检查预设规则（settings.json 中的 allow 规则）
  ├─ 匹配 → allow
  └─ 不匹配 → 继续
  ↓
② AI 分类器评估（如果可用）
  ├─ 安全 → allow
  └─ 不安全/不可用 → 继续
  ↓
③ PermissionRequest Hooks
  ├─ Hook 返回 allow → allow
  └─ Hook 返回 deny 或无 Hook → 继续
  ↓
④ Channel Relay（如果配置了 Telegram/Discord/Slack）
  ├─ 发送权限请求到 Channel
  ├─ 等待远程审批
  └─ 超时 → deny
  ↓
⑤ 最终降级：auto-deny
  └─ Agent 收到 deny，调整策略或报告失败
```

**Channel Relay 机制**：

Headless 模式下最有趣的设计是 Channel Relay——当 Agent 遇到需要人类决策的情况时，它可以通过 Telegram、Discord 等渠道发送消息给人类，等待远程审批。这让"无人值守"的 Agent 在关键时刻仍能获得人类输入。

**产品设计洞察**：
> Headless Mode 不是简单地"去掉 UI"。它重新定义了权限决策的整条链路，确保 Agent 在无人环境下仍然有安全的降级策略。Channel Relay 是一个绝妙的折中——在"完全自动"和"完全阻塞等人"之间开辟了第三条路：通过异步消息获得远程人类的关键决策。

### 2.5 Coordinator Mode（协调者模式）—— "Agent 当项目经理"

**角色分配**：人 = 项目发起者，Agent = 项目经理 + 团队

**核心设计**：一个主 Agent（Coordinator）接收用户任务，分解为子任务，分配给多个 Sub-Agent 并行执行，最后汇总结果。

**源码规模**：`coordinatorMode.ts` 有 **19,000+ 行**，是整个代码库最复杂的模块之一。

**协作架构**：

```
┌─────────────────────────────────────────────────────┐
│                  Coordinator Mode                    │
│                                                      │
│  用户 ───→ Coordinator Agent（主 Agent）              │
│              │                                       │
│              ├─ 任务分解                              │
│              │   ├─ Task 1: "重构 API 层"            │
│              │   ├─ Task 2: "更新测试"               │
│              │   └─ Task 3: "更新文档"               │
│              │                                       │
│              ├─ 分配执行                              │
│              │   ├─ Sub-Agent A ← Task 1             │
│              │   │   └─ Worktree: feature/api        │
│              │   ├─ Sub-Agent B ← Task 2             │
│              │   │   └─ Worktree: feature/tests      │
│              │   └─ Sub-Agent C ← Task 3             │
│              │       └─ Worktree: feature/docs       │
│              │                                       │
│              ├─ 进度追踪                              │
│              │   ├─ TaskCreate/TaskUpdate 管理状态    │
│              │   ├─ 依赖关系：Task 2 blockedBy Task 1│
│              │   └─ 实时进度对用户可见               │
│              │                                       │
│              ├─ 结果汇总                              │
│              │   ├─ 收集各 Sub-Agent 的工作成果       │
│              │   ├─ 合并 Worktree 分支               │
│              │   └─ 解决冲突                          │
│              │                                       │
│              └─ 返回给用户                            │
│                  "任务完成，以下是变更摘要..."        │
└─────────────────────────────────────────────────────┘
```

**Plan Mode 在 Coordinator 中的特殊行为**：

当 Sub-Agent 处于 Plan Mode 时，它不是向用户请求审批，而是向 Coordinator Agent 请求：

```
Sub-Agent 进入 Plan Mode
  ↓
isPlanModeRequired() = true（由 Coordinator 设置）
  ↓
Sub-Agent 调用 ExitPlanMode
  ↓
发送 plan_approval_request 到 Coordinator 的 mailbox
  ↓
Coordinator 审阅方案
  ├─ 批准 → Sub-Agent 开始执行
  └─ 修改/驳回 → Sub-Agent 调整方案
```

**Task 依赖管理**：

```typescript
// Coordinator 创建任务时设置依赖
TaskCreate({ subject: "更新测试", description: "..." })
TaskUpdate({ taskId: "2", addBlockedBy: ["1"] })
// Task 2 被 Task 1 阻塞，直到 Task 1 完成

// Task 1 完成后
TaskUpdate({ taskId: "1", status: "completed" })
// Task 2 自动解除阻塞，可以开始执行
```

**产品设计洞察**：
> Coordinator Mode 是 Agent 产品的终局形态之一。它不再是"人指挥一个 Agent"，而是"人指挥一个 Agent 团队"。关键的设计选择是让 Coordinator Agent 既是用户的代理人（理解用户意图），又是 Sub-Agent 的管理者（分解任务、审批方案、解决冲突）。这本质上是把"技术经理"的角色交给了 AI。

## 第三部分：控制权分配机制的设计哲学

### 3.1 控制权谱系

将五种模式放在一个统一的控制权矩阵中：

| 维度 | Plan Mode | Default Mode | Background Mode | Headless Mode | Coordinator Mode |
|------|-----------|-------------|----------------|---------------|-----------------|
| **人的参与频率** | 每步（审方案） | 每个写操作 | 仅启动和结束 | 不参与 | 仅启动 |
| **Agent 自主度** | 最低（只能看） | 中等（需审批） | 高（独立执行） | 最高（全自动） | 高（团队管理） |
| **适用信任级别** | 陌生/不确定 | 日常使用 | 熟悉后 | 完全信任 | 完全信任 |
| **中断能力** | 随时退出 | Ctrl+C/Esc | 可 abort | 无（或 Channel） | 可取消子任务 |
| **典型场景** | 大型重构前 | 日常编码 | 并行开发 | CI/CD | 跨模块大任务 |
| **权限处理** | N/A（不执行） | 弹窗审批 | auto-deny | 预设规则+分类器 | Coordinator 审批 |
| **上下文隔离** | 共享主 context | 共享主 context | 独立 context | 独立 context | 每 Agent 独立 |

### 3.2 渐进式放权的产品逻辑

Claude Code 设计了一条清晰的"信任升级路径"：

```
新用户首次使用
  ↓
Plan Mode（"让我先看看你的方案"）
  ↓ 几次交互后，用户觉得 Agent 思路靠谱
Default Mode（"你可以执行了，但每步我看一下"）
  ↓ 用户发现 Agent 执行质量稳定
Always Allow 规则（"这类操作不用再问我了"）
  ↓ 用户完全信任
Accept Edits / Bypass Mode（"全部自动执行"）
  ↓ 场景升级
Background Mode / Headless Mode（"你自己干，完了告诉我"）
  ↓ 团队协作需求
Coordinator Mode（"管理一个团队帮我干"）
```

**每一步升级都是用户主动选择的，不是系统强制的。**

这个设计遵循了一个重要的产品原则：**信任是赢得的，不是给予的**。系统不会主动提升 Agent 的权限，而是通过稳定的执行表现让用户自愿放权。

### 3.3 "永远可以回退"的安全网

无论用户处于哪种模式，都有回退机制：

- **Plan Mode → Default Mode**：随时退出 Plan Mode
- **Default Mode → Plan Mode**：输入 `/plan` 随时进入
- **Background Agent**：可以 abort 取消
- **Headless Mode**：Channel Relay 保留远程干预能力
- **Coordinator Mode**：可以取消单个子任务或整个任务

**产品设计洞察**：
> 渐进式放权 + 随时可回退 = 用户心理安全感。这跟 SaaS 产品的"免费试用 → 基础版 → 专业版"升级路径是同一个产品逻辑，只不过升级的不是功能，而是"信任级别"。

## 第四部分：交互原语设计

Claude Code 在五种模式之上，设计了三个核心交互原语，它们是所有人机交互的基本构建块。

### 4.1 AskUserQuestion —— 结构化提问

**设计意图**：Agent 不是在闲聊，而是在做决策。当 Agent 需要用户输入时，应该给出**结构化的选项**而非开放式提问。

**源码实现**（`AskUserQuestionTool.tsx`）：

```
AskUserQuestion Schema：

questions: [1-4 个问题]
  ├─ question: "使用哪个数据库？"（明确的问句）
  ├─ header: "Database"（≤20字符，显示为标签芯片）
  ├─ multiSelect: false（单选 vs 多选）
  └─ options: [2-4 个选项]
       ├─ label: "PostgreSQL"（1-5 个词，选项名称）
       ├─ description: "关系型数据库，适合事务场景"
       └─ preview: "CREATE TABLE..."（可选，代码预览）

返回值：
  answers: { "使用哪个数据库？": "PostgreSQL" }
  annotations: { 用户附加的备注和预览选择 }
```

**Preview 机制**：当选项涉及具体的代码实现或 UI 布局时，每个选项可以附带一个 preview（支持 Markdown 渲染），用户可以在选择前直观比较不同方案。

```
┌─────────────────────────────────────────────┐
│  选项列表          │  Preview 预览区         │
│                    │                         │
│  ○ 方案 A          │  ```                    │
│    简洁实现         │  function handler() {   │
│                    │    return db.query(...)  │
│  ● 方案 B ←────────│  }                      │
│    完整实现         │  ```                    │
│                    │                         │
│  ○ 方案 C          │                         │
│    渐进式实现       │                         │
└─────────────────────────────────────────────┘
```

**关键约束**：
- `requiresUserInteraction = true`：即使在 Bypass 模式下也必须等用户回答
- `--channels` 模式下禁用（Telegram/Discord 无法展示复杂 UI）
- 用户永远可以选 "Other" 输入自定义答案

**产品设计洞察**：
> AskUserQuestion 的核心思想是"降低用户的认知负荷"。不是问"你想怎么做？"（开放式，需要用户自己想），而是问"这三个方案你选哪个？"（闭合式，用户只需判断）。这是将 Agent 从"执行者"升级为"顾问"的关键设计。

### 4.2 Permission Dialog —— 权限审批对话框

**设计意图**：让用户在最少信息量下做出正确的权限决策。

```
┌─────────────────────────────────────────────────┐
│  🔧 Bash                                        │
│                                                  │
│  npm install lodash                              │
│                                                  │
│  Risk: LOW                                       │
│  Reason: Installs a well-known npm package       │
│                                                  │
│  ┌──────────────────────────────────────────┐   │
│  │  [Allow]  [Always Allow npm install]     │   │
│  │  [Deny]   [Always Deny]                  │   │
│  └──────────────────────────────────────────┘   │
│                                                  │
│  Suggestions:                                    │
│  • Add rule: Bash(prefix:npm install) → Allow   │
│  ────────────────────────────────────────────    │
│  ⏳ Classifier check pending...                  │
└─────────────────────────────────────────────────┘
```

**关键设计元素**：

1. **风险等级（Risk Level）**：LOW / MEDIUM / HIGH，由 Permission Explainer 生成
2. **决策建议（Suggestions）**：系统建议用户添加什么样的永久规则
3. **分类器指示器**：显示 AI 分类器是否正在后台评估
4. **Always Allow**：一键创建会话级或永久级的 allow 规则
5. **200ms 防抖**：防止用户在对话框弹出时误触

### 4.3 Task Progress —— 任务进度追踪

**设计意图**：让用户知道 Agent 在做什么、做到哪了、还有多少没做。

```
TaskCreate Schema:
  subject: "修复认证模块"         ← 任务标题（祈使句）
  description: "需要修改 auth.ts"  ← 任务详情
  activeForm: "修复认证模块中"     ← Spinner 显示文本（进行时）

TaskUpdate Schema:
  taskId: "1"
  status: "in_progress" | "completed"
  addBlockedBy: ["2"]              ← 依赖关系
  addBlocks: ["3"]                 ← 被依赖关系
```

**视觉呈现**：

```
任务列表：
  ✅ #1 读取项目结构                    [completed]
  🔄 #2 修复认证模块 ← 当前正在执行     [in_progress]
  ⏸  #3 编写测试（等待 #2 完成）        [blocked]
  ○  #4 更新文档                        [pending]

Spinner 区域：
  ⠋ 修复认证模块中...
```

**依赖关系的产品价值**：Task 系统支持 `blockedBy` 和 `blocks` 依赖关系，这不只是技术实现，更是给用户提供**工作流可视化**——用户能看到任务之间的依赖关系，理解为什么某些任务还没开始。

## 第五部分：中断与恢复机制

### 5.1 三级中断体系

Claude Code 设计了从轻到重的三级中断：

```
┌──────────────────────────────────────────────────┐
│                三级中断体系                        │
│                                                   │
│  Level 1: Esc（轻度中断）                         │
│  ├─ 效果：关闭当前权限对话框                       │
│  ├─ 影响范围：仅当前交互                           │
│  └─ Agent 行为：收到 deny，调整策略继续工作        │
│                                                   │
│  Level 2: Ctrl+C（中度中断）                      │
│  ├─ 效果：取消当前 Agent 轮次                      │
│  ├─ 影响范围：当前正在执行的所有工具               │
│  ├─ Agent 行为：                                   │
│  │   ├─ cancel 类工具：收到 synthetic error        │
│  │   └─ block 类工具：等待完成后再中断             │
│  └─ 用户可以输入新指令                             │
│                                                   │
│  Level 3: Ctrl+C × 2（重度中断）                  │
│  ├─ 效果：强制终止所有操作                         │
│  ├─ 影响范围：整个 Agent 进程                      │
│  └─ Agent 行为：立即停止                           │
└──────────────────────────────────────────────────┘
```

### 5.2 工具中断行为分类

每个工具通过 `interruptBehavior()` 声明自己被中断时的行为：

| 中断类型 | 行为 | 适用工具 | 原因 |
|----------|------|---------|------|
| `cancel` | 立即终止，返回错误 | Read, Grep, Glob, WebFetch | 只读操作，随时可中断 |
| `block` | 等待完成后再响应中断 | Edit, Write, Bash（运行中） | 半途中断可能破坏文件 |

### 5.3 嵌套 Abort Controller

```
查询级 AbortController（最外层）
  └─ 批次级 siblingAbortController
      └─ 工具级 toolAbortController

中断传播规则：
- 用户 Ctrl+C → 查询级 abort → 所有子级 abort
- Bash 执行错误 → siblingAbortController abort → 同批次兄弟取消
  └─ 但不影响查询级（Agent 继续工作）
- 单工具超时 → toolAbortController abort → 仅取消该工具
  └─ 不影响批次和查询
```

**产品设计洞察**：
> 中断机制的粒度决定了用户的控制感。如果只有"全部停止"一个选项，用户会犹豫是否要中断（怕丢失进度）。三级中断让用户可以精确控制：想关个对话框？Esc。想换个方向？Ctrl+C。想完全停止？双击 Ctrl+C。这种粒度化的控制是"用户安全感"的基础。

## 第六部分：实时反馈与进度可视化

### 6.1 流式信息架构

Claude Code 的 UI 信息流不是"等完了再显示"，而是实时流式渲染：

```
信息类型层级：

┌─ StreamEvent（token 级）
│  └─ 模型生成的每个 token 实时显示
│
├─ ProgressMessage（工具执行级）
│  └─ 工具执行过程中的中间进度
│     例如："Reading file src/auth.ts..."
│     例如："Running npm test... (3/10 passed)"
│
├─ SpinnerWithVerb（操作级）
│  └─ 当前正在做什么的一句话描述
│     例如："⠋ Fixing authentication bug..."
│     显示在 Task 的 activeForm 中
│
├─ Token Budget Display（会话级）
│  └─ 顶部栏显示当前 token 使用量和预算
│
└─ Cost Threshold Warning（费用级）
   └─ 当累计费用接近阈值时弹出警告
```

### 6.2 进度信息的设计原则

1. **及时性**：信息在产生时立即推送，不积攒
2. **层次性**：不同粒度的信息在不同位置展示
3. **可忽略性**：进度信息不阻塞工作流，用户可以不看
4. **成本透明**：token 消耗和费用始终可见

**产品设计洞察**：
> 实时反馈解决的核心问题是"AI 在做什么？"这个疑问。用户看不到 Agent 的工作过程时会焦虑（"它是不是卡住了？""它在做我要的吗？"）。流式渲染 + Spinner + 进度消息三层反馈彻底消除了这种焦虑。

## 第七部分：Hooks 作为交互扩展点

### 7.1 Hook 事件全景

Claude Code 提供了 **18+ 种 Hook 事件**，覆盖了 Agent 生命周期的每个关键节点：

```
会话生命周期：
  SessionStart → ... → SessionEnd

每轮交互：
  PromptSubmit → PreToolUse → PostToolUse → Stop

权限相关：
  PermissionRequest（无头模式权限委托）

Plan 相关：
  PlanApproval（Coordinator 模式方案审批）

特殊事件：
  InstructionsLoaded（CLAUDE.md / Memory 加载后）
  TeammateIdle（子 Agent 空闲时）
  TaskCompleted（任务完成时）
  ConfigChange（配置变更时）
  WorktreeCreate / WorktreeRemove
```

### 7.2 Hooks 的三种交互扩展模式

**模式 A：输入预处理**
```
用户输入 "fix the bug"
  ↓
PromptSubmit Hook 拦截
  ↓
Hook 脚本执行：
  ├─ 注入项目上下文（"当前在 feature/auth 分支"）
  ├─ 注入最近的错误日志
  └─ 修改后的提示："fix the bug. Context: on feature/auth branch,
      recent error: TypeError in auth.ts:42"
  ↓
Agent 收到增强后的输入
```

**模式 B：操作拦截**
```
Agent 要执行 `rm -rf node_modules`
  ↓
PreToolUse Hook 拦截
  ↓
Hook 脚本评估：
  └─ 返回 { permissionDecision: 'deny',
            systemMessage: '请使用 npm ci 代替 rm + install' }
  ↓
Agent 收到拒绝 + 建议，调整策略
```

**模式 C：后处理触发**
```
Agent 编辑了 src/auth.ts
  ↓
PostToolUse Hook 触发
  ↓
Hook 脚本执行：
  ├─ 自动运行 prettier 格式化
  ├─ 自动运行 eslint --fix
  └─ 更新修改日志
  ↓
Agent 继续工作（不感知 Hook 的存在）
```

### 7.3 Hooks 配置方式

Hooks 通过 JSON 配置，无需修改源码：

```json
// .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "command": "python3 .claude/hooks/bash_guard.py"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit",
        "command": "npx prettier --write $CLAUDE_FILE_PATH"
      }
    ],
    "Stop": [
      {
        "command": "python3 .claude/hooks/auto_test.py"
      }
    ]
  }
}
```

**产品设计洞察**：
> Hooks 是"Agent 平台化"的关键设计。通过 Hooks，企业可以在不 fork 代码的前提下：强制代码规范（PreToolUse 拦截不符合规范的操作）、自动化流水线（PostToolUse 触发测试/部署）、注入企业上下文（PromptSubmit 注入内部文档）。这把 Claude Code 从"一个工具"变成了"一个可定制的平台"。

## 第八部分：与其他 Agent 产品的交互模式对比

### 8.1 五维度对比矩阵

| 维度 | Claude Code | Cursor | GitHub Copilot Workspace | OpenAI Agents SDK |
|------|-------------|--------|--------------------------|-------------------|

| **控制权粒度** | **连续谱系**（6 级信任阶梯） | 二元（Chat 或 Agent） | 固定（先 Plan 后 Execute） | 无内置（需自建） |
| **中断机制** | **三级**（Esc/Ctrl+C/×2） | 单级（Stop） | 单级（Cancel） | 无内置 |
| **进度反馈** | **四层**（Token/Spinner/Progress/Stream） | 二层（Spinner/Stream） | 一层（步骤列表） | 无内置 |

| **无头模式** | ✅ 完整支持 | ❌ | ❌ | ✅ 原生支持 |
| **多 Agent 协调** | ✅ Coordinator Mode | ❌ | ❌ | ✅ Handoff 模式 |
| **远程审批** | ✅ Channel Relay | ❌ | ❌ | ❌ |

### 8.2 核心差异分析

**Claude Code vs Cursor**：
- Cursor 的 Agent 模式是"全自动"，没有 Claude Code 的渐进式信任设计
- Cursor 没有 Plan Mode 的"先看方案再执行"
- Cursor 没有 Background Agent 的并行工作能力
- Cursor 的优势在于 IDE 原生集成（代码差异 UI 更直观）

**Claude Code vs GitHub Copilot Workspace**：
- Copilot Workspace 只有一种固定流程：Plan → Implement → Review
- Claude Code 的五种模式更灵活，用户可以根据场景切换
- Copilot Workspace 的 Plan 步骤是一次性的；Claude Code 的 Plan Mode 可以反复进入
- Copilot Workspace 的优势在于 GitHub 原生集成（PR 工作流）

**Claude Code vs OpenAI Agents SDK**：
- OpenAI SDK 是框架而非产品，交互模式由开发者实现
- OpenAI SDK 有 `needs_approval` 机制但远不如 Claude Code 精细
- OpenAI SDK 的 Handoff 模式类似 Coordinator Mode 的简化版
- Claude Code 的优势在于开箱即用的完整交互体系

## 第九部分：产品设计启示

### 9.1 Agent 产品的五条交互设计原则

基于 Claude Code 的源码分析，提炼出五条可迁移的设计原则：

**原则一：信任是谱系，不是开关**

```
错误做法：Agent 要么全自动，要么每步审批
正确做法：设计连续的信任级别，让用户按自己的节奏升级

Claude Code 的实现：
  Plan → Default → Always Allow → Accept Edits → Bypass → Background → Headless
  每一级都是用户主动选择的，而且随时可以回退
```

**原则二：用选项代替开放问题**

```
错误做法：Agent 问 "你想怎么处理这个 bug？"
正确做法：Agent 问 "这三种修复方案你选哪个？"（附带 preview）

Claude Code 的实现：
  AskUserQuestion 强制要求 2-4 个选项 + 描述 + 可选 preview
  用户的认知负荷从"生成答案"降为"评估选项"
```

**原则三：进度必须实时且分层**

```
错误做法：Agent 工作 30 秒后返回最终结果
正确做法：每秒都有进度反馈，不同粒度在不同位置

Claude Code 的实现：
  Layer 1: Token 流式渲染（最细粒度，实时显示思考过程）
  Layer 2: Spinner + activeForm（操作粒度，显示当前动作）
  Layer 3: Task Progress（任务粒度，显示整体进度）
  Layer 4: Token/Cost Budget（会话粒度，显示资源消耗）
```

**原则四：中断要有粒度**

```
错误做法：只有"全部停止"一个选项
正确做法：从轻到重多级中断，让用户精确控制

Claude Code 的实现：
  Esc（关个对话框）→ Ctrl+C（取消当前轮）→ Ctrl+C×2（强制停止）
  工具级别也区分：cancel（可以打断）vs block（等我做完）
```

**原则五：扩展点要暴露在生命周期关键节点**

```
错误做法：Agent 是黑盒，用户无法定制行为
正确做法：在每个关键节点暴露 Hook，让用户注入逻辑

Claude Code 的实现：
  18+ Hook 事件覆盖完整生命周期
  JSON 配置，无需修改源码
  企业可以通过 Hooks 强制安全策略、注入上下文、自动化流水线
```

### 9.2 如果要设计一个 Agent 产品的交互体系

```
优先级排序（按 ROI）：

P0 - 基础交互：
  ├─ Default Mode（权限审批 + 流式输出）—— 最小可用产品
  ├─ 结构化提问（选项式而非开放式）
  └─ Ctrl+C 中断

P1 - 信任升级：
  ├─ Plan Mode（规划先行）
  ├─ Always Allow 规则（减少审批疲劳）
  └─ Task Progress（进度可视化）

P2 - 高级场景：
  ├─ Background Agent（并行工作）
  ├─ Headless Mode（CI/CD 集成）
  └─ Hooks 扩展

P3 - 终局形态：
  ├─ Coordinator Mode（Agent 团队）
  ├─ Channel Relay（远程审批）
  └─ Worktree 隔离（并行开发）
```

## 总结

Claude Code 的人机协作体系不是"一种交互模式"，而是**一个协作模式的完整谱系**。它的核心设计洞察是：

1. **不同场景需要不同的控制权分配**——日常编码、大型重构、CI/CD 集成、团队协作，每种场景对"人的参与度"要求完全不同
2. **信任不是布尔值，而是连续变量**——从 Plan Mode 的"完全不信任"到 Headless Mode 的"完全信任"，中间有无数个中间态
3. **交互不是打断，而是协作**——AskUserQuestion 的选项式设计、Permission Dialog 的竞态审批、Task Progress 的实时反馈，都在最小化"打断感"的同时最大化"控制感"

这套体系的产品价值在于：它让 Agent 能同时服务"谨慎的新用户"和"大胆的专家用户"，不需要为不同用户群体开发不同的产品。一个产品，通过模式切换，覆盖从"新手辅导"到"全自动 CI/CD"的全部场景。

> **一句话总结**：Claude Code 的人机协作设计本质是一个"控制权的连续谱系 + 随时可回退的安全网 + 最小化干预的交互原语"，它重新定义了"人和 AI Agent 应该如何协作"这个问题的解空间。
