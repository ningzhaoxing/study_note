# Harness Engineering：驾驭 AI 的工程方法论

## 核心理念

Harness（马具）这个词精准地隐喻了这套方法论的本质：**AI 是一匹跑得很快的马，但需要马具来控制它的方向**。

在 AI Agent 时代，人的角色不再是编写每一行代码，而是：
1. **Context 工程师**：设计和维护 AI 可理解的知识体系
2. **方向把控者**：在关键节点提供审批与纠偏

这种转变的底层逻辑是：**让 AI 做执行，让人做决策**。

---

## 三大支柱

### 1. Context Engineering（上下文工程）

#### 核心原则：仓库即真相（Repository as Truth）

> "AI 不会去联网做任何 research，AI 所有的对业务的理解都要来自于 Context 仓库。不在 Context 里面的东西就等于没有。"

#### Context 仓库的内容结构

```
context/
├── requirements/          # PRD、需求文档
├── designs/              # 技术方案、架构设计
├── decisions/            # 架构决策记录（ADR）
├── meetings/             # 会议纪要
└── index.md             # 索引（每次更新后必须同步）
```

#### CLAUDE.md 的分层机制

CLAUDE.md 与 Context 仓库有本质区别：

| 维度       | CLAUDE.md           | Context 仓库         |
| -------- | ------------------- | ------------------ |
| **性质**   | 代码规范、技术规则           | 业务上下文、需求、设计        |
| **服务对象** | 代码研发 Agent          | 所有 Agent（PM/技术/研发） |
| **内容**   | 代码规范、测试规范、commit 风格 | PRD、技术方案、会议纪要、架构决策 |
| **价值定位** | 决定 Agent 怎么写代码      | 业务知识资产（真正值钱的部分）    |

**分层覆盖规则**：
```
/CLAUDE.md                    # 全局通用规则
/apps/tara/CLAUDE.md         # TARA 专属规则（覆盖全局）
/apps/hara/CLAUDE.md         # HARA 专属规则（覆盖全局）
```

子文件夹的 CLAUDE.md 会覆盖上级配置，实现 Context 隔离，避免 AI 在不同项目间"串台"。

#### 渐进式信息披露

不要一次性把所有信息塞给 AI，而是：
- 在索引中提供结构化导航
- 让 AI 根据任务需要主动检索
- 通过工具（Grep/Glob/Read）实现 Just-in-time 信息获取

---

### 2. Mechanical Constraints（机械约束）

通过**不可绕过的技术手段**强制执行规则，而非依赖 AI 的"自觉"。

#### 实施层级

1. **架构层约束**
   - Monorepo 结构强制模块边界
   - 依赖图限制跨模块引用
   - 接口契约强制类型安全

2. **CI/CD 层约束**
   - Pre-commit hooks 检查代码规范
   - 测试覆盖率门禁
   - 自动化 lint/format

3. **工具链约束**
   - 限制 AI 可用的命令集
   - 沙箱环境隔离
   - 权限分级管理

#### 设计原则

> "如果一个规则可以被机械地执行，就不要写在 Prompt 里——写在 CI/CD 里。"

---

### 3. Entropy Management（熵管理）

#### 问题本质

> "AI 很可能自动沿用之前犯的错、技术债，然后让技术债越来越大甩不掉。"

AI 会从现有代码中学习模式，如果代码库中存在技术债，AI 会：
- 识别为"项目风格"
- 在新代码中复制
- 导致熵值（混乱度）指数级增长

#### 解决方案

1. **定期清理技术债**
   - 设置固定的重构周期
   - 使用专门的 Skill（如 `repo-maintain`）
   - 在 CLAUDE.md 中规定清理规则

2. **自动化回收重构**
   - 识别重复代码模式
   - 自动提取公共逻辑
   - 清理未使用的依赖

3. **Context 仓库维护**
   - 及时更新索引
   - 归档过期文档
   - 保持信息的时效性

## 多 Agent 协作流程

### 理想工作流

```
PM Agent
  ↓ 输出 PRD
  ↓ 更新 Context 仓库索引

技术方案 Agent
  ↓ 读取 PRD
  ↓ 输出技术方案
  ↓ 更新 Context 仓库索引

代码研发 Agent
  ↓ 读取 PRD + 技术方案
  ↓ 开始开发
  ↓ 严格遵循 Context 仓库内容
```

### 关键约束

- **不在仓库里的不做**：避免 AI 自由发挥
- **不联网 research**：所有知识来自 Context
- **每次输出后更新索引**：保持知识库的可发现性

## 知识检索策略

### 从传统 RAG 到 Agentic Search

**传统 RAG 的问题**：
```
文档 → chunk 化 → 向量化 → 向量数据库
       ↓
用户 query → 语义匹配 → 结果 + query → 大模型生成
```
- 人替 AI 决定需要什么信息
- 语义匹配可能遗漏关键上下文
- 固定的检索策略缺乏灵活性

**Agentic Search 的优势**：
```
大模型自己决定"我还缺什么信息"
  ↓
使用工具（Grep、Glob、Read）自己搜索
  ↓
反思结果是否充分
  ↓
继续检索或开始生成
```

### Claude Code 的 5 个查询工具

1. **Glob**：文件模式匹配（`*.ts`, `src/**/*.tsx`）
2. **Grep**：内容搜索（正则表达式）
3. **Read**：文件读取
4. **Agent/Explore**：派发子 Agent 探索
5. **Agentic Search**：组合前几个工具 + 反思

### 工具复用原则

> "如果我们用 Sonnet 或 Opus 模型，要尽可能复用它的工具，因为这些工具会在后训练阶段加入模型的能力上面。外挂的能力支持度没有原生的好。"

## 实践建议

### 1. 从 Context 隔离开始

- 为不同项目创建独立的 CLAUDE.md
- 使用分层覆盖机制避免规则冲突
- 在全局 CLAUDE.md 中只保留通用规则

### 2. 建立 Context 仓库维护规范

在 CLAUDE.md 中明确规定：
```markdown
## Context 维护规则

- 每次发布技术方案后，必须更新 `context/index.md`
- PRD 放在 `context/requirements/`
- 会议纪要放在 `context/meetings/`
- 架构决策放在 `context/decisions/`
```

### 3. 设计分级检索结构

- 大的目录或树状结构通过文件系统管理
- 不需要完全依赖语义相似度匹配
- 让 AI 通过索引导航，而非盲目搜索

### 4. 定期执行熵管理

- 每个迭代周期预留重构时间
- 使用自动化工具识别技术债
- 在 Context 仓库中记录重构决策

---

## 总结

Harness Engineering 的本质是**通过工程化手段驾驭 AI 的能力边界**：

- **Context Engineering** 解决"AI 知道什么"
- **Mechanical Constraints** 解决"AI 能做什么"
- **Entropy Management** 解决"AI 会不会越做越乱"

当这三个支柱稳固后，人可以从编码者转变为架构师和决策者，让 AI 成为真正高效的执行伙伴。

> "以后人不要写任何代码，全让 Agent 写，人只负责把控方向以及提供约束。"

这不是让人失业，而是让人回归更有价值的工作：**设计系统、定义规则、把控方向**。
