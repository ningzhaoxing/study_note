# Claude Code 工具体系设计与查询类工具深度研究

> 研究对象：Claude Code CLI（Anthropic 官方 Agent 产品）
> 研究视角：Agent 产品设计 / 业务场景逆向推导
> 研究日期：2026-04-01

---

## 第一部分：工具体系全景设计

### 1.1 核心设计理念

Claude Code 的工具体系是一个**分层、可延迟加载、并发安全**的架构。其核心思想是：

> **工具是 Agent 与外部世界交互的唯一通道，系统通过精确控制"哪些工具可用、何时可用、如何组合使用"来约束和引导 Agent 的行为。**

这套设计反映了三个关键判断：

1. **工具即能力边界**：Agent 能做什么，完全取决于它拥有哪些工具。工具不是"附加功能"，而是 Agent 能力的根本定义。
2. **工具选择即策略**：面对同一任务，选择不同工具组合会产生截然不同的效率和效果。系统在 System Prompt 中嵌入了大量"何时用何工具"的指导。
3. **工具数量需要管控**：36+ 个工具如果全部塞进 prompt，会占用大量 token。因此需要延迟加载机制（Tool Search）来管理工具集合的大小。

### 1.3 工具分类全景

按业务职能分类，Claude Code 的 36+ 个工具可归为以下 8 大类：

| 分类 | 工具 | 职能定位 |
|------|------|----------|
| **查询类（信息获取）** | Glob, Grep, Read, ToolSearch | 从代码库和文件系统中获取信息 |
| **编辑类（文件操作）** | Edit, Write, NotebookEdit | 修改和创建文件 |
| **执行类（命令运行）** | Bash, PowerShell, REPL | 运行系统命令和脚本 |
| **协调类（子代理）** | Agent, SendMessage | 分发子任务、并行工作 |
| **任务管理类** | TaskCreate/Get/List/Update/Stop/Output | 追踪工作进度 |
| **规划类** | EnterPlanMode, ExitPlanMode, EnterWorktree, ExitWorktree | 工作流程管理 |
| **交互类** | AskUserQuestion, Brief | 与用户沟通 |
| **外部信息类** | WebFetch, WebSearch, MCP* | 获取互联网和外部服务信息 |
| **定时调度类** | CronCreate/Delete/List | 定时任务管理 |

### 1.4 工具加载策略：Always-Loaded vs Deferred

这是一个非常精妙的设计决策。Claude Code 没有把所有工具一股脑塞给模型，而是将工具分为两层：

**Always-Loaded（始终加载，约 10 个）**：
- Glob, Grep, Read, Edit, Write, Bash, Agent, AskUserQuestion, Skill, Brief

**Deferred（延迟加载，约 20+ 个）**：
- 所有 Task 工具、Cron 工具、Web 工具、Plan 工具、Worktree 工具、NotebookEdit、MCP 相关工具等

**设计逻辑**：
- Always-Loaded 工具是**每个对话都几乎一定会用到的核心能力**（搜索、读取、编辑、执行）
- Deferred 工具是**只有特定场景才需要的扩展能力**
- 这种分层直接降低了每次 API 调用的 prompt token 消耗
- 当模型需要延迟工具时，通过 `ToolSearch` 工具按关键词检索并激活

**产品设计启示**：在 Agent 产品中，工具不应该是"全量铺开"的。应该设计一个"核心工具集 + 按需加载"的分层机制，以平衡能力覆盖和 token 效率。

### 1.5 并发安全机制

Claude Code 引入了 `isConcurrencySafe()` 标记：

- **并发安全（可并行）**：Glob, Grep, Read, ToolSearch
- **非并发安全（需串行）**：Edit, Write, Bash

**设计逻辑**：所有只读查询工具都标记为并发安全，系统会自动将同一轮中的多个并发安全工具调用进行并行执行，极大提升搜索效率。

### 1.6 权限管控体系

每个工具都有 `checkPermissions()` 方法，形成三层权限防线：

1. **输入验证（validateInput）**：参数格式、路径合法性、安全路径检查（如阻止 UNC 路径防止 NTLM 凭据泄露）
2. **权限检查（checkPermissions）**：基于 allow/deny 规则判断操作是否被允许
3. **运行时防护**：阻止特殊设备文件（`/dev/zero`, `/dev/random` 等）防止进程挂起

## 第二部分：查询类工具深度研究

### 2.1 查询类工具定位与设计逻辑

在一个软件工程 Agent 中，**"理解代码"是一切操作的前提**。Agent 不可能修改它没有读过的代码，也不可能修复它没有定位到的 bug。因此，查询类工具在整个工具体系中扮演着**基础设施级别**的角色。

Claude Code 的查询工具体系解决的核心问题是：

> **如何让一个 AI Agent 在几万甚至十万行的大型代码库中，高效且精准地找到它需要的信息？**

这个问题的难度在于：
- 代码库太大，不可能全部塞进 context window
- 搜索目标多样化：可能是文件名、代码内容、特定模式
- 需要在速度、精度和 token 消耗之间取得平衡

### 2.2 查询类工具清单

Claude Code 共有 **5 个核心查询类工具**，构成一个完整的信息获取链路：

| 工具 | 一句话定位 | 底层引擎 | 输入 | 输出 |
|------|-----------|---------|------|------|
| **Glob** | 按文件名/路径模式查找文件 | ripgrep --files | glob 通配符 | 文件路径列表 |
| **Grep** | 按内容搜索文件 | ripgrep | 正则表达式 | 匹配行/文件列表/匹配计数 |
| **Read** | 读取文件完整内容 | Node.js fs | 文件路径 | 文件内容（文本/图片/PDF/Notebook） |
| **Agent** | 复杂多步骤搜索代理 | 子进程 Claude 实例 | 自然语言任务描述 | 搜索结果摘要 |
| **ToolSearch** | 在延迟工具中搜索 | 内存关键词匹配 | 关键词/工具名 | 匹配的工具定义 |

### 2.3 Glob 工具：文件定位器

#### 定位
Glob 是整个搜索链路的**起点工具**——当你知道大概在找什么类型的文件，但不知道具体路径时使用。

#### 核心参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `pattern` | string, 必填 | glob 通配符模式，如 `**/*.ts`, `src/**/*.test.js` |
| `path` | string, 可选 | 搜索的起始目录，默认为当前工作目录 |

#### 实现细节

**底层引擎**：使用 ripgrep 的 `--files` 模式（不是 Node.js 的 glob 库），这意味着：
- 性能极高，即使面对超大代码库也能快速返回
- 自动尊重 `.gitignore` 规则
- 包含隐藏文件

**结果限制**：硬限制 100 个文件，超出会设置 `truncated: true` 标记

**结果排序**：按文件修改时间排序（最旧的在前），这个排序策略暗示了一个假设——最近修改的文件更可能与当前工作相关

**Token 优化**：所有路径都被转换为相对路径（相对于当前工作目录），以减少 token 消耗

#### 典型使用场景
```
# 找到所有 TypeScript 测试文件
Glob(pattern: "**/*.test.ts")

# 找到 src/tools 目录下的所有入口文件
Glob(pattern: "src/tools/**/index.ts")

# 找到所有配置文件
Glob(pattern: "**/*.config.{js,ts,json}")
```

#### 产品设计洞察

Glob 的设计揭示了一个关键原则：**先定位文件，再深入内容**。这是一个"漏斗式"搜索策略的第一步——先缩小范围到文件级别，再通过 Grep 或 Read 进入内容级别。100 个文件的硬限制也说明了设计者对 token 效率的极端关注。

### 2.4 Grep 工具：内容搜索器

#### 定位
Grep 是**代码库内容级搜索的核心工具**——当你需要找到包含特定代码模式、函数名、变量名的文件或代码行时使用。

#### 核心参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `pattern` | string, 必填 | 正则表达式搜索模式 |
| `path` | string, 可选 | 搜索路径（文件或目录） |
| `glob` | string, 可选 | 文件名过滤（如 `*.ts`） |
| `type` | string, 可选 | 按语言类型过滤（如 `js`, `py`, `rust`） |
| `output_mode` | enum, 可选 | `files_with_matches`(默认) / `content` / `count` |
| `-B`, `-A`, `-C` | number, 可选 | 上下文行数（前/后/前后） |
| `-i` | boolean, 可选 | 大小写不敏感 |
| `-n` | boolean, 可选 | 显示行号（默认 true） |
| `head_limit` | number, 可选 | 输出限制（默认 250，传 0 为无限） |
| `offset` | number, 可选 | 分页偏移量 |
| `multiline` | boolean, 可选 | 跨行匹配模式 |

#### 三种输出模式的设计考量

这是一个极其精妙的设计——Grep 不是简单地返回搜索结果，而是提供了三种输出策略：

**1. `files_with_matches`（默认模式）**
- 只返回匹配的文件路径列表
- 按修改时间排序（最新的在前）
- Token 消耗最小
- **使用场景**：初步探索——"哪些文件涉及了这个概念？"

**2. `content` 模式**
- 返回匹配的实际代码行
- 支持上下文行数（-B/-A/-C）
- 支持行号显示
- **使用场景**：精确定位——"这个函数在哪里被调用？调用上下文是什么？"

**3. `count` 模式**
- 只返回每个文件的匹配数量
- **使用场景**：评估影响面——"这个 API 在整个代码库中被使用了多少次？"

**设计逻辑**：这三种模式实际上对应了搜索过程中的三个递进阶段：
1. 先用 `files_with_matches` 确定范围（哪些文件相关）
2. 再用 `count` 评估规模（影响面多大）
3. 最后用 `content` 获取细节（具体代码是什么）

#### 关键实现细节

**VCS 目录排除**：自动排除 `.git`, `.svn`, `.hg`, `.bzr`, `.jj`, `.sl` 等版本控制目录

**行长度限制**：最大 500 字符/行，这个限制直接过滤掉了 base64 编码内容和压缩后的 minified 代码——这些内容对搜索毫无意义但会严重浪费 token

**默认 head_limit = 250**：这个数字经过仔细权衡：
- 源码注释明确提到："Unbounded content-mode greps can fill up to the 20KB persist threshold"
- 250 条结果足够覆盖绝大多数搜索需求
- 同时防止一次搜索就耗尽大量 context token

**分页机制**：通过 `offset` + `head_limit` 实现分页，允许 Agent 在需要时逐步获取更多结果

**超时控制**：
- 普通平台：20 秒超时
- WSL 环境：60 秒超时（因为 WSL 文件 I/O 性能显著更差）
- 超时时抛出专门的 `RipgrepTimeoutError`，而不是返回空结果，确保 Agent 知道搜索未完成

#### 产品设计洞察

Grep 的三种输出模式设计体现了**渐进式信息获取**的理念：先给 Agent 最轻量的信息（文件列表），只有在需要时才提供更详细的信息（内容）。这在有限 context window 下是必须的策略。head_limit 的默认值设计则体现了**防御性设计**——保护 Agent 不被自己的搜索结果淹没。

### 2.5 Read 工具：文件内容读取器

#### 定位
Read 是**信息获取链路的"最后一公里"**——当你已经精确定位到某个文件，需要阅读其完整内容时使用。

#### 核心参数

| 参数 | 类型 | 说明 |
|------|------|------|
| `file_path` | string, 必填 | 文件的绝对路径 |
| `offset` | number, 可选 | 起始行号 |
| `limit` | number, 可选 | 读取行数 |
| `pages` | string, 可选 | PDF 页面范围（如 `"1-5"`） |

#### 多类型文件支持

Read 不仅仅是一个文本读取工具，它是一个**多模态文件理解器**：

| 文件类型 | 处理方式 | 输出格式 |
|----------|---------|---------|
| **文本文件**（.ts, .py, .md 等） | 逐行读取，附加行号 | `cat -n` 格式的带行号文本 |
| **图片**（.png, .jpg, .gif, .webp） | 检测格式 → 自适应压缩 → base64 | 图片 content block |
| **PDF** | 小文件整体读取 / 大文件按页提取 | PDF content block 或页面图片 |
| **Jupyter Notebook**（.ipynb） | 解析所有 cell（代码+输出+可视化） | 结构化 cell 数组 |

#### 关键优化：文件去重（Deduplication）

这是 Read 工具中最精妙的设计：

```
当 Agent 在同一个对话中第二次读取同一个文件时：
1. 检查文件的修改时间（mtime）是否改变
2. 如果未改变 → 返回 "file_unchanged" 桩（stub），而不是完整内容
3. 模型被告知："文件未变，请参考之前读取的内容"
```

**为什么这很重要？**
- 源码注释明确提到：**约 18% 的 Read 调用是重复读取同一文件**
- 这些重复读取会浪费 **约 2.64% 的整个系统的 cache_creation token**
- 通过返回一个轻量桩消息，大幅减少 token 消耗

**产品设计启示**：这个去重机制揭示了一个 Agent 产品的核心挑战——Agent 在多轮对话中会反复访问同一资源。设计一个"资源级缓存 + 变更检测"机制可以显著降低成本。

#### 安全防护

Read 工具的安全防护极其细致：

1. **设备文件阻止**：阻止读取 `/dev/zero`（无限输出）、`/dev/random`（阻塞）、`/dev/stdin`（阻塞输入）等特殊文件
2. **二进制文件检测**：拒绝读取二进制文件（PDF、图片、SVG 除外）
3. **UNC 路径防护**：阻止 `\\server\share` 形式的路径，防止 NTLM 凭据泄露
4. **Token 上限校验**：读取后验证 token 数量（默认上限 25,000 token），超出则抛出错误

#### 大小限制策略

| 限制项 | 默认值 | 说明 |
|--------|--------|------|
| 最大行数 | 2000 行 | 单次读取的默认行数上限 |
| 最大文件体积 | 256 KB | 超出需使用 offset/limit 分段读取 |
| 最大输出 token | 25,000 | 防止单个文件占满 context |
| PDF 最大页数 | 20 页/次 | 大 PDF 必须分页读取 |

#### 产品设计洞察

Read 工具的设计体现了一个核心取舍：**完整性 vs. 效率**。默认 2000 行的限制意味着大多数源文件可以一次性完整读取，但对于超大文件（如生成的代码、日志文件）则强制分段。这是一个"覆盖 80% 常见场景，兜底 20% 极端场景"的策略。

### 2.6 Agent 工具（搜索能力维度）：智能搜索代理

#### 定位
Agent 工具在查询场景中的定位是**复杂搜索任务的编排器**——当搜索需要多轮交互、跨多个文件/目录、需要理解上下文才能确定下一步搜索方向时使用。

#### 查询场景下的特殊子代理类型

Agent 工具内置了专门的探索型子代理：

**`Explore` 子代理**：
- 专门用于代码库探索
- 拥有除 Agent, Edit, Write, NotebookEdit 外的所有工具（即只有查询能力，无修改能力）
- 适用于"需要 3 次以上查询"的深度探索任务

#### Agent 作为查询工具的两个核心价值

**1. Context 隔离**
- 子代理在独立的 context 中运行
- 搜索过程中产生的大量中间结果不会污染主对话的 context
- 只有最终摘要返回给主 Agent

**2. 并行搜索**
- 可以同时启动多个子代理搜索不同方向
- 例如：一个子代理搜索函数定义，另一个搜索调用处，第三个搜索测试文件

#### 产品设计洞察

Agent-as-Search-Tool 的设计揭示了一个重要洞察：**在大型代码库中，搜索本身就是一个需要"智能"的任务**。简单的关键词搜索（Grep）能解决 80% 的问题，但剩下的 20%（如"理解这个模块的整体架构"）需要一个能"思考"的搜索代理。

### 2.7 ToolSearch 工具：工具发现器

#### 定位
ToolSearch 是一个**元查询工具**——它搜索的不是代码，而是其他工具。当延迟加载的工具需要被激活时使用。

#### 三种查询模式

**1. 精确选择模式**
```
query: "select:NotebookEdit,WebFetch"
```
直接按工具名选择，支持逗号分隔多选。

**2. 关键词搜索模式**
```
query: "notebook jupyter"
```
基于关键词评分匹配：
- 工具名匹配：+10 分
- 工具名部分匹配：+5 分
- searchHint 匹配：+4 分（策划的能力描述短语）
- 描述匹配：+2 分

**3. 必选+可选模式**
```
query: "+slack send"
```
`+` 前缀标记必须匹配的术语，其余为可选评分术语。

#### 产品设计洞察

ToolSearch 的存在证明了一个设计原则：**当工具数量增长到一定规模，工具本身也需要被"搜索"**。这实际上是一个"工具的工具"（meta-tool）设计模式。在 MCP（Model Context Protocol）生态下，随着第三方工具的不断接入，这种工具发现机制将变得越来越重要。

## 第三部分：查询工具的协作体系

### 3.1 搜索策略的整体编排

Claude Code 在 System Prompt 中为模型规定了明确的**工具选择决策树**：

```
                    用户的查询需求
                         │
              ┌──────────┴──────────┐
              │ 是否知道具体文件路径？│
              └──────────┬──────────┘
                    │         │
                   是         否
                    │         │
              ┌─────┴───┐   ┌┴────────────────────┐
              │  Read    │   │ 是否知道文件名模式？  │
              │ 直接读取  │   └──────────┬──────────┘
              └─────────┘         │           │
                                 是           否
                                  │           │
                          ┌───────┴──┐  ┌─────┴──────────────┐
                          │  Glob    │  │ 是否知道内容关键词？ │
                          │ 模式匹配  │  └──────────┬─────────┘
                          └─────────┘        │          │
                                            是          否
                                             │          │
                                     ┌───────┴──┐  ┌───┴──────────┐
                                     │  Grep    │  │    Agent     │
                                     │ 内容搜索  │  │  智能探索代理  │
                                     └─────────┘  └──────────────┘
```

System Prompt 中的原文指引：

> - **简单、定向的搜索**（特定文件/类/函数）→ 直接使用 Glob 或 Grep
> - **更广泛的代码库探索和深度研究** → 使用 Agent 工具的 Explore 子代理
> - 后者比直接使用 Glob/Grep **更慢**，所以只在简单搜索不够用或任务明显需要 3 次以上查询时使用

### 3.2 典型协作模式

#### 模式一：漏斗式搜索（最常见）

```
Glob("**/*.ts") → 找到所有 TS 文件
    │
    ▼
Grep(pattern: "export class.*Tool", glob: "*.ts") → 找到包含 Tool 类定义的文件
    │
    ▼
Read("/src/tools/GrepTool/GrepTool.ts") → 读取具体实现
```

**适用场景**：知道大概方向但不确定具体位置。每一步都缩小范围。

#### 模式二：并行搜索（高效模式）

```
┌──────────────────────────────────────────────────┐
│  同一轮内并行调用（isConcurrencySafe = true）     │
│                                                  │
│  Grep(pattern: "function authenticate")          │
│  Grep(pattern: "class AuthService")              │
│  Glob(pattern: "**/auth/**")                     │
│                                                  │
│  → 三个结果同时返回，综合分析                      │
└──────────────────────────────────────────────────┘
```

**适用场景**：已经明确要找什么，需要从多个维度快速定位。

#### 模式三：Agent 委托搜索（复杂探索）

```
Agent(subagent_type: "Explore", prompt: "理解认证模块的架构")
    │
    │  子代理内部自主执行：
    │  ├─ Glob("**/auth/**") → 找到认证相关文件
    │  ├─ Grep("import.*auth") → 找到依赖关系
    │  ├─ Read(auth/index.ts) → 读取入口文件
    │  ├─ Read(auth/middleware.ts) → 读取中间件
    │  └─ Grep("AuthService") → 追踪类的使用
    │
    ▼
返回结构化分析结果给主 Agent
（所有中间搜索结果不进入主 context）
```

**适用场景**：开放性探索任务，需要多轮搜索+理解，中间结果对主任务无直接价值。

#### 模式四：Grep 三阶段搜索（影响面分析）

```
Grep(pattern: "oldAPI", output_mode: "files_with_matches")
→ "哪些文件用了 oldAPI？"（10 个文件）

Grep(pattern: "oldAPI", output_mode: "count")
→ "总共用了多少次？"（47 次，分布在 10 个文件中）

Grep(pattern: "oldAPI", output_mode: "content", path: "src/core/handler.ts", -C: 3)
→ "在关键文件中的具体使用上下文是什么？"
```

**适用场景**：评估重构影响面、API 迁移规划。

### 3.3 Token 效率优化的全链路设计

Claude Code 在查询链路的每个环节都进行了 token 优化：

| 环节 | 优化策略 | 预期效果 |
|------|---------|---------|
| Glob 输出 | 路径相对化（去掉绝对路径前缀） | 减少约 30-50% 路径 token |
| Glob 结果 | 硬限制 100 个文件 | 防止无限膨胀 |
| Grep 输出 | 默认 `files_with_matches` 模式（只返回文件名） | 比 `content` 模式节省 10x+ token |
| Grep 内容 | 默认 head_limit=250 行 | 防止大范围搜索的结果爆炸 |
| Grep 行长 | 最大 500 字符/行 | 过滤 minified/base64 噪音 |
| Read 去重 | 同文件重复读取返回桩 | 节省约 2.64% 的系统级 token |
| Read 大小 | 默认 2000 行 / 256KB / 25000 token 上限 | 防止单文件占满 context |
| Agent 搜索 | 子代理 context 隔离 | 中间结果不污染主 context |

### 3.4 排序策略的设计哲学

不同工具使用了不同的排序策略，背后都有明确的产品逻辑：

| 工具 | 排序方式 | 设计逻辑 |
|------|---------|---------|
| **Glob** | 按修改时间排序（最旧在前） | 旧文件通常是稳定的核心模块，优先呈现 |
| **Grep (files_with_matches)** | 按修改时间排序（**最新在前**） | 最近修改的文件更可能与当前工作相关 |
| **Read** | 按行号顺序 | 保持代码的自然阅读顺序 |

Glob 和 Grep 的排序方向**相反**，这不是一个 bug，而是反映了不同搜索场景的需求差异：
- Glob（找文件）：你在探索代码库结构，核心文件通常最稳定
- Grep（搜内容）：你在定位最近的变更，新修改的文件最值得关注

## 第四部分：设计总结与产品启示

### 4.1 核心设计原则

从 Claude Code 查询工具体系中，可以提炼出以下 Agent 产品设计原则：

**原则 1：渐进式信息获取**
不要一次性给 Agent 所有信息。先给概览（文件列表），再给细节（内容），最后给上下文（周围代码）。这是在有限 context 下的必然策略。

**原则 2：防御性 Token 管理**
每个工具都有明确的输出上限（100 文件、250 行、25000 token）。这些不是技术限制，而是产品设计决策——保护 Agent 不被自己的输出淹没。

**原则 3：多模式输出**
同一个工具（如 Grep）提供多种输出模式，让 Agent 根据当前需求选择最合适的信息密度。

**原则 4：查询工具并发化**
所有只读查询工具都标记为并发安全，系统自动并行执行。这是提升搜索效率的关键手段。

**原则 5：Agent-as-Tool 的递归设计**
Agent 工具本身就是一个查询工具——它将"复杂搜索"这个能力封装为一个工具，实现了搜索能力的递归嵌套。

**原则 6：缓存与去重**
对于反复读取的文件，通过 mtime 检测+桩返回实现去重。这是一个成本优化措施。

### 4.3 对 Agent 产品设计的启发

1. **工具体系设计是 Agent 产品的核心架构决策**：工具不是"功能列表"，而是 Agent 能力的根本定义。工具的种类、参数设计、输出格式、使用指引，共同构成了 Agent 的"行为框架"。

2. **查询能力决定了 Agent 的"智商上限"**：一个不能高效搜索的 Agent 就像一个不能翻阅资料的人——即使推理能力再强，也无法解决实际问题。

3. **Token 经济学是贯穿始终的设计约束**：从工具的延迟加载、搜索结果的分页限制、文件读取的去重机制，到路径的相对化处理——每一个设计决策都在优化 token 效率。

4. **"何时用何工具"的指引与工具本身同等重要**：Claude Code 在 System Prompt 中花了大量篇幅指导工具选择，这意味着**工具的 prompt 设计是产品设计的一部分**，而不是技术实现细节。

5. **递归代理模式是处理复杂搜索的关键**：通过 Agent-as-Tool 的递归设计，将复杂的多步骤搜索封装为一个"高级工具"，同时通过 context 隔离保护主对话的 token 空间。
