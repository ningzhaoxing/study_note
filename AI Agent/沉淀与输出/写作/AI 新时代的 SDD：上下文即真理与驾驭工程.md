# AI 新时代的 SDD：上下文即真理与驾驭工程

> 当 AI 写了 90% 的代码，工程师的核心产出是什么？

---

## 一个反直觉的事实

用 AI Coding 用得越深，你会越频繁地打开 Markdown 编辑器，而不是代码编辑器。

这不是玩笑。看看任何一个深度使用 Claude Code 的项目仓库，`docs/` 目录的体量开始逼近 `src/`。PRD、技术方案、TestPlan、CLAUDE.md、agents.md——这些 Markdown 文件不再是"写完就不看的附件"，而是整个开发流程的驱动核心。代码反而变成了副产品。

这背后是一个正在发生的范式转移：从 Code-Driven Development 到 Spec-Driven Development（SDD）。支撑这个转移的，是一条贯穿始终的逻辑链——从 Spec 文档体系的建立，到上下文的注入，到文档产出的 Agent 化，再到人对整个流程的驾驭。这四件事不是独立的步骤，而是一条链，断了任何一环都不转。

下面沿着这条链，一环一环地讲。

---

## CLAUDE.md 即 Spec

在传统开发中，项目规格散落在 Confluence、飞书文档、Slack 消息甚至"老员工的脑子里"。人类可以忍受这种混乱——我们有记忆、有默契、有"你去问一下张哥"的兜底策略。

AI 不行。AI 不会联网查资料，不会去 Slack 翻历史消息，不会"问一下张哥"。它所有对项目的理解，100% 来自你喂给它的上下文。你不写的东西，对 AI 来说就不存在。

所以需要一个地方，把项目的所有关键信息汇聚起来。这就是 `CLAUDE.md` 的定位——整个项目的 Spec 总纲。它回答三件事：

1. **这个项目是什么** —— 背景、目标、核心模块、技术架构
2. **怎么做** —— 代码规范、测试规范、commit 风格、分支策略
3. **去哪找更多信息** —— 索引指向 PRD、技术方案、TestPlan 等详细文档

```
CLAUDE.md（总纲）
  ├── → context/requirements/PRD.md
  ├── → context/designs/技术方案.md
  ├── → context/test-plans/TestPlan.md
  └── → context/decisions/ADR-001.md
```

CLAUDE.md 本身不承载所有信息，它是一张地图。AI 从这张地图出发，根据当前任务按需检索深层文档——Claude Code 的设计哲学管这个叫"渐进式信息披露"：不要把所有东西一股脑塞给 AI，让它像翻文档一样，先看目录，再看细节。

写 CLAUDE.md 这件事看起来不起眼，但它实际上在做一件很重的事：把散落在团队各处的隐性知识，固化成机器可读的显性规格。

## 上下文即真理

有了 Spec 文档体系之后，接下来要建立一个认知：这些文档不是"参考资料"，它们是 AI 的唯一真相来源。

这个概念借鉴自 Claude Code 的设计哲学——"Repository as Truth"（仓库即真相）。说直白点：不在上下文里的东西，就等于不存在。

- PRD 里没写的需求，AI 不会去猜
- 技术方案里没有的约束，AI 不会去遵守
- TestPlan 里没定义的验收标准，AI 不会去满足

很多人用 AI Coding 翻车，不是 AI 能力不够，是上下文给得不够。你给一句"帮我写一个用户登录功能"，它当然能写出来——但它不知道你们用 JWT 还是 Session，不知道密码策略，不知道失败三次要锁定。于是你得到一个"能跑但全错"的实现。

所以，在让 AI 动手之前，先把真相写清楚。

AI Coding 的上下文体系分两层：

**主上下文（驱动层）：**
- **PRD** —— 做什么、为什么做
- **技术实现文档** —— 怎么做、用什么做
- **TestPlan** —— 什么叫做完了、什么绝对不能做

**辅助上下文（约束层）：**
- **代码本身** —— 现有实现、类型定义、接口契约
- **CLAUDE.md** —— 编码规范、技术规则、工具约束

注意优先级：文档是主，代码是辅。这跟传统开发完全相反。传统开发中代码是唯一真相，文档是附属品。AI Coding 中反过来了——AI 从文档获得明确指令，比从代码中"猜意图"高效得多。

其中 TestPlan 值得单独说一下。这里说的不是单元测试用例（那是 AI 自己的事），而是高层验收标准和禁止操作清单：

```markdown
## Must Have（不通过就不算完成）
- [ ] 输入有效凭证，能成功登录并返回 token
- [ ] 错误密码三次后锁定账户 15 分钟
- [ ] Token 过期后自动拒绝请求，返回 401

## Redline（绝对不能做）
- 不允许明文存储密码
- 不允许在日志中输出 token
- 不允许跳过 rate limit 检查
```

TestPlan 有个隐藏好处：写它的过程会逼你想清楚需求的边界。它既约束 AI，也约束你自己，让你没法"先做做看再说"。

## 文档产出的 Agent 化

上面解决了"AI 执行时依据什么"的问题。但还有一个更前置的问题：**这些文档本身，谁来写？**

传统答案是人。产品经理写 PRD，架构师写技术方案，QA 写 TestPlan。问题是，SDD 对文档的质量和完备度要求很高，而人最不愿意做的事情恰恰就是写文档。

解法是：为每一类文档沉淀出一个专门的 Agent。PRD Agent 负责产出需求文档，技术方案 Agent 负责产出设计文档，TestPlan Agent 负责产出测试计划。每个 Agent 对应一份 `agents.md`，定义了它的职责边界、输入输出、工作流程和质量标准。

```
agents/
├── prd-agent.md           # PRD Agent
│   ├── 输入：用户故事、竞品分析、业务目标
│   ├── 输出：结构化 PRD（背景/目标/功能清单/优先级/约束）
│   └── 质量标准：每个功能点必须有验收条件
│
├── design-agent.md        # 技术方案 Agent
│   ├── 输入：PRD + 现有架构 + 技术约束
│   ├── 输出：技术方案（架构设计/接口定义/数据模型/关键决策）
│   └── 质量标准：必须包含替代方案对比和决策理由
│
└── testplan-agent.md      # TestPlan Agent
    ├── 输入：PRD + 技术方案
    ├── 输出：TestPlan（Must Have/Need Have/Should Have/Redline）
    └── 质量标准：Must Have 必须可机械验证，Redline 必须可自动检查
```

这不是三个 prompt 模板。每个 Agent 知道自己该读什么上下文、产出什么格式、遵守什么标准，而且它们之间有明确的上下游关系：PRD Agent 的输出喂给技术方案 Agent，两者的输出再喂给 TestPlan Agent。

这样一来，人的角色从"写文档"变成了"审文档"。跟 PRD Agent 对话，描述业务背景，回答几个关键问题，它产出一份结构完整的初稿，人来审阅修正。这个心理门槛跟从零写一份 PRD 完全不同。

而且 agents.md 是可迭代的。团队用得越多，规则和模板越精准，就像打磨模具——用得越久，产出越稳定。最终 agents.md 本身成了团队知识资产：编码了"我们写 PRD 的最佳实践"、"我们做技术决策的框架"、"我们定义验收标准的模板"。

至此形成了一个自举闭环：

```
agents.md 定义各 Agent 如何产出文档
    ↓
PRD Agent / 技术方案 Agent / TestPlan Agent 产出文档
    ↓
文档作为上下文真相，由 CLAUDE.md 索引
    ↓
AI Coding Agent 基于上下文生成代码
```

文档驱动代码，Agent 驱动文档。那人在这个循环里做什么？

## Harness Engineering——人的新角色

"Harness"在英文里是马具。AI 是一匹跑得极快的马，但需要缰绳和方向。人不需要亲自跑，但三件事必须人来做：

**造马具：设计和迭代文档体系。**

建立 CLAUDE.md → PRD → 技术方案 → TestPlan → agents.md 这套体系，持续迭代每个 Agent 的规则，让文档产出的质量越来越稳定。文档质量决定了 AI 产出质量的天花板。

**拉缰绳：审阅关键节点。**

不是每一行代码都需要人审。但几个关键节点必须人来把关：
- Agent 产出的 PRD、技术方案、TestPlan —— 文档审阅
- 架构变更和安全相关的代码 —— Code Review
- TestPlan 的 Must Have 验收 —— 最终交付确认

**清马道：管理熵增。**

AI 生成的代码天然趋向熵增。它会从现有代码中学习模式，如果代码库有技术债，AI 会把技术债当成"项目风格"然后大量复制。人需要定期清理技术债、更新过时文档、维护索引的可发现性。不做这件事，上下文会逐渐腐烂，AI 的产出质量跟着一起退化。

控制权重新分配之后，表格长这样：

| | 传统开发 | AI 时代 |
|---|---|---|
| **人做的事** | 写代码、调试、测试 | 审文档、审方案、管质量、迭代 agents.md |
| **AI 做的事** | 代码补全、简单建议 | 写文档、实现功能、修 Bug、写测试、重构 |
| **核心产出** | 代码 | 文档 |
| **质量瓶颈** | 人的编码能力 | 文档的完备度 |

## 一条链

回头看，四个环节串成一条链：

```
CLAUDE.md 即 Spec
    → 上下文即真理（PRD + 技术方案 + TestPlan）
        → agents.md（PRD/技术方案/TestPlan 各自的 Agent 工作流）
            → Harness Engineering（人审文档、审节点、管熵增）
```

断了任何一环：

- 没有 CLAUDE.md → AI 不知道项目是什么
- 没有上下文文档 → AI 乱猜需求
- 没有 agents.md → 文档产出全靠人肉，SDD 无法规模化
- 没有 Harness Engineering → 没人审阅维护，文档和代码一起腐烂

项目仓库中最有价值的不是代码——代码随时可以让 AI 重新生成。PRD、技术方案、TestPlan、CLAUDE.md、agents.md，这些凝结了人的判断和决策的文档，才是真正的核心资产。

---

## 写在最后

如果你今天开始一个新项目，试试这个顺序：

1. 先写 CLAUDE.md，不写代码
2. 用 PRD Agent 产出 PRD，人审阅修正
3. 用技术方案 Agent 产出设计文档，人审阅修正
4. 用 TestPlan Agent 产出验收标准和禁区，人审阅修正
5. 然后让 AI Coding Agent 基于以上上下文去写代码

前四步做扎实了，第五步几乎不需要你操心。

这不是让人失业。这是让人回归更有价值的工作：**设计系统、定义规则、把控方向**。
