最近在做资产识别 Agent，有个问题想了挺久：咨询和识别要不要用同一个 Agent？

最直觉的方案是单 Agent 搞定一切，让 LLM 自己判断该干什么。但最后我没这么做——用了显式路由，Go 代码分类意图，再路由到专用 Agent。

做完之后去看了同事负责的损害场景 Agent，发现他走的是完全相反的路子：单 Agent，20 个工具全塞进去，让 LLM 自己决定干什么。两个模块面临的问题差不多，两份代码却截然不同。

这让我开始想，我们俩的分歧到底从哪里来。

---

## 我为什么没用单 Agent

说到底，还是不够信任 LLM。

不是觉得模型会犯低级错误，而是全部交给 AI 来做，上下文会越来越大，注意力会分散，在边界情况下更容易出问题。工具越多、历史越长，模型做出错误判断的概率就越高——这不是模型的问题，是概率问题。

另外还有一个很现实的考虑：工具越多、提示词越长，每次调用的 token 消耗越大，响应也越慢。把咨询和识别塞进同一个 Agent，意味着每次咨询都要带着一堆识别工具的描述跑一遍，这些开销在咨询场景下完全是浪费。

所以我选了更保守的方案：把工具集按场景拆开，每个 Agent 只做一件事。

---

## 但同事的选择也跑通了

同事的损害场景 Agent 持有约 20 个工具，让 LLM 通过 ReAct 自主决定执行路径。

损害场景和资产识别面临的问题其实差不多——都有咨询和执行两种模式，都涉及数据写入，都需要处理用户的自然语言输入。两个模块走向了不同的架构，不是因为场景有本质差异，而是各自对一个问题的回答不同：**该把信任放在哪里？**

我把信任放在代码上，他把信任放在模型上。两种选择都能跑通，只是取舍不同。

这让我意识到，我之前对 LLM 的保留，可能比实际需要的更多。

---

## 显式路由的边界

不过在做这个系统的过程中，我发现显式路由有一个地方绕不过去。

用户可能先咨询"这批资产里哪些是高风险的"，然后说"好，那就识别这些"。第二句话里的"这些"依赖第一轮的上下文，意图识别层如果只看当前消息，根本不知道该识别什么。

一个解法是：让意图识别结合历史记录做一次总结，先搞清楚"用户这次到底想干什么"，再路由。技术上可行，但这样一来意图识别就不再是一个简单的分类器了，它开始需要理解上下文、做推理——原本最干净的一层，越来越像一个小 Agent。

这说明显式路由有一个根本性前提：每条消息可以被独立分类。一旦任务复杂到需要跨轮次理解，这个前提就开始松动，你要么在路由层加复杂度，要么承认这个场景不适合显式路由。

这样来说，Agent 的整体设计架构就会变得极为复杂，而“简单”才是最好的设计哲学。

单 Agent 在这里反而没有这个问题——所有历史都在同一个 context 里，LLM 自己处理上下文，不需要额外设计。

---

## 重新理解"信任 LLM"

这两种架构放在一起看，让我对"信任 LLM"这件事有了不一样的理解。

信任 LLM 不是"把所有决策都交给它"，而是**清楚地知道在哪里信任它**。语义理解、动态推理、处理模糊的自然语言——这些是 LLM 真正擅长的，交给它没问题。但确定性的路由规则、工具集的边界划分——这些用代码来保证更直接，也更省 token。

从这个角度看，显式路由不是"不够 AI Native"，而是在正确的地方用了代码，在正确的地方用了 LLM。AI Native 的核心不是"用了多少 AI"，而是"在合适的地方用了 AI"。

---

## 未来一定会往单 Agent 走

但有一点是确定的：随着模型能力持续提升，单 Agent 能处理的场景边界会越来越宽。OpenAI 的 Swarm、Anthropic 的 Agent SDK、Google 的 ADK，都在往"更少的显式路由、更多的 Agent 自主决策"方向走。

现在选显式路由，是因为当前阶段这样更稳、更省。但这不意味着这套架构要一直维持下去。

所以我在设计时刻意把意图识别层解耦出来——它只负责分类，不知道分类结果会被怎么用。路由层用注册表模式，新增意图只需要 `router.Register("new_intent", handler)`，不动主流程。如果有一天要切换到单 Agent，实现一个 OmniAgent，把所有路由指向它，Handler 层和意图识别层几乎不用动。

架构演进的触发条件不只是"模型变强了"，还需要"业务边界本身变得模糊了"。现在的解耦，是为了到时候能低成本地做这个决定。
