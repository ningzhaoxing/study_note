# Tool 转 Skill 调研结论

**调研时间**：2026-03-03
**决策结论**：不进行重构

---

## 结论

**不进行 Tool 到 Skill 的重构**

---

## 核心论述

### 1. 流程封装问题

**当前架构特点**：
- 工具设计原子化、灵活
- LLM 可以根据用户意图动态组合工具
- 支持多种意图类型（咨询、查询、生成、分析）

**Skill 封装的问题**：
- 封装后流程固定，失去灵活性
- 无法处理咨询类问题（占 30% 用户请求）
- 无法灵活查询数据（占 20% 用户请求）
- 80% 的工具调用是动态的，不存在固定流程

**结论**：
> 当前业务需要高度灵活的工具组合，不适合固定流程的 Skill 封装

### 2. Token 节省问题

**Skill 的核心价值**：
- 减少 LLM 调用次数
- 降低 Token 消耗

**实际收益分析**：

| 方案 | Token 节省 | 代价 |
|-----|-----------|------|
| 全面封装 | 84% | 失去灵活性，无法处理咨询和查询 |
| 局部封装 | 13-15% | 保持灵活性 |

**关键数据**：
- 当前工具量：38 个（合理范围内）
- 代码重复率：2%（远低于 20% 阈值）
- 按意图分类后，每种场景实际使用 5-10 个工具

**结论**：
> Token 节省收益有限（13-15%），不值得牺牲灵活性

## 决策依据

### 工具量统计

| Agent | 工具数量 | 评估 |
|-------|---------|------|
| AssetIdentificationAgent | 6 | ✅ 合理 |
| DamageScenarioAgent | 22 | ✅ 合理（按意图分类后 5-10 个/场景） |
| ThreatScenarioAgent | 10 | ✅ 合理 |
| **总计** | **38** | **✅ 在 LLM 处理能力范围内** |

### 代码重复率

- 实际重复率：**2%**（200 行 / 9755 行）
- 重复工具：2 个
- 重构阈值：20%
- **结论**：远低于阈值

### 串行调用分析

| Agent | 必定串行 | 动态选择 | 适合封装 |
|-------|---------|---------|---------|
| AssetIdentificationAgent | 100% | 0% | ✅ |
| DamageScenarioAgent | 5% | 80% | ❌ |
| ThreatScenarioAgent | 30% | 40% | ❌ |

**结论**：只有 1 个 Agent 适合封装，但单独封装收益有限

---

## 未来重新评估条件

当出现以下情况时，重新评估是否需要 Skill 封装：

1. **工具量显著增长**
   - 单个 Agent 工具数量 > 30 个
   - 总工具数量 > 60 个

2. **代码重复率上升**
   - 代码重复率 > 20%
   - 重复工具 > 5 个

3. **Token 成本成为瓶颈**
   - Token 消耗 > 5000 tokens/请求
   - Token 成本显著影响运营

4. **业务流程固化**
   - 出现明确的固定业务流程（固定流程占比 > 80%）
   - 咨询类和查询类请求占比 < 10%
