# 字节跳动飞书集成平台（AnyCross）

> 实习单位：字节跳动 - 飞书（2025.06 - 2025.10）
> 岗位：后端开发实习生

---

## 一、项目背景

**飞书集成平台（AnyCross）** 是字节跳动旗下企业级 **iPaaS（Integration Platform as a Service）** 产品，通过**低代码/无代码**方式连接企业内外部系统，构建统一的自动化工作流。

**业务价值**：解决企业多系统数据孤岛和跨系统协作成本高的问题。例如：当 CRM 系统新建了一条线索，自动触发飞书消息通知销售，同时更新 Jira 任务——这整个流程无需写代码，在 AnyCross 配置即可。

**我负责的模块**：
- **Trigger 模块**：监听外部系统事件（如 Gmail 收到邮件），触发工作流
- **Action 模块**：执行工作流中的操作（如发飞书消息），负责可观测性设计

## 二、核心工作内容

### 2.1 TriggerDraft 接口重构

**背景**：TriggerDraft（触发器草稿）原本用一个 `extra map[string]interface{}` 字段存储不同类型触发器的差异化配置，导致类型不安全、业务逻辑散乱。

**问题根源**：
```go
// 重构前：所有触发器配置混在 extra 里
type TriggerDraft struct {
    TriggerID string
    Extra     map[string]interface{} // 类型不安全，不同触发器往里塞不同字段
}
// 业务代码里充满了 type assertion 和 nil 判断
pollingInterval := draft.Extra["polling_interval"].(int)  // 运行时才报错
```

**重构方案**：接口抽象 + 类型细分

```go
// 重构后：触发器草稿接口
type ITriggerDraft interface {
    TriggerType() TriggerType
    Validate() error
    ToConfig() TriggerConfig
}

// 轮询型触发器（定时拉取数据）
type PollingTriggerDraft struct {
    Interval    time.Duration
    DataFetcher DataFetcherConfig
}

// 即时触发器（Webhook 主动推送）
type InstantTriggerDraft struct {
    WebhookURL string
    SecretKey  string
}
```

**收益**：
- 消除 `map[string]interface{}` 的类型不安全问题
- 支持 `subscribe` 全生命周期的配置化开发（不同类型触发器有完全不同的生命周期管理）
- 编译期类型检查，减少运行时错误

### 2.2 DataFetcher 模型演进（G1 → G2）

**背景**：Gmail 的事件拉取是一个**两步异步流程**：
1. 首先通过 Gmail History API 获取变更记录（只有 historyID + messageID）
2. 再通过 messageID 调用 Gmail Message API 获取邮件详情

**原有 G1 模型的局限**：DataFetcher 假设"一次 API 调用能获取完整数据"，不支持这种需要多步 API 串联的数据拉取逻辑。

**G2 模型设计**：

```
G2 DataFetcher 状态机：

   ┌─────────────┐      historyID 变更      ┌──────────────────┐
   │  IDLE 空闲   │ ──────────────────────→ │ FETCHING_IDS     │
   └─────────────┘                          │ （拉取消息ID列表） │
                                            └────────┬─────────┘
                                                     │ IDs 获取成功
                                                     ↓
                                            ┌──────────────────┐
                                            │ FETCHING_DETAILS │
                                            │ （并发拉取消息详情）│
                                            └────────┬─────────┘
                                                     │ 详情获取完成
                                                     ↓
                                            ┌──────────────────┐
                                            │   READY 就绪      │
                                            │   触发工作流       │
                                            └──────────────────┘
```

**落地成果**：完成 **Gmail、Slack、Outlook、Teams** 四款物料从 G1 → G2 的迁移。

### 2.3 Action Trace 存储方案（面试重点）

**业务需求**：
- 用户在调试 Action（工作流节点）时，需要**即时查询**每次执行的入参、出参、耗时、错误信息
- Action 调用量大（QPS 300~500），执行记录需要**持久化**，不能只存 Redis

**方案设计**：Redis + RocketMQ + MySQL 三层架构

```
用户发起 Action 调用
         ↓
┌────────────────────────────────────────────────────┐
│                 Action 执行完成                      │
│                        ↓                            │
│    ┌───────────────────────────────────────────┐   │
│    │           并发写（双写）                    │   │
│    │                                           │   │
│    ↓                           ↓               │   │
│  写入 Redis                 发送 RocketMQ       │   │
│  (TTL: 7天)                （异步）             │   │
│  ↑                               ↓             │   │
│  用户调试查询                  Consumer          │   │
│  立即可见 ✅                  异步持久化          │   │
│                               写入 MySQL         │   │
│                              （历史查询）         │   │
└────────────────────────────────────────────────────┘
```

**关键设计决策**：

| 问题 | 方案 | 原因 |
|------|------|------|
| 用户调试需要立即看到结果 | 先写 Redis | Redis 写入 < 1ms，用户体验好 |
| 数据需要持久化（7天+） | MQ → MySQL | 异步持久化不影响主链路 |
| QPS 300-500 的写入压力 | RocketMQ 削峰 | MQ 消费速率可控，保护 MySQL |
| Redis 数据丢失兜底 | MySQL 长期存储 | 超过 TTL 的历史数据从 MySQL 查 |

**面试解释**：

> 直接同步写 MySQL 有两个问题：1）QPS 300-500 会对 MySQL 造成较大写压力；2）MySQL 写入 < Redis 慢，影响 Action 主链路响应时间。我们的方案是：Action 执行后先写 Redis（供即时调试查询，TTL 7 天），同时发一条消息到 RocketMQ，Consumer 异步把数据持久化到 MySQL（供历史查询）。MQ 起到了流量削峰的作用，MySQL 的消费速率由 Consumer 控制，不会被高峰期的 Action 调用直接冲垮。

## 三、技术亮点总结（面试口述版本）

**开场（30秒）**：
> 我在字节跳动飞书做后端实习，主要负责飞书集成平台（AnyCross）的 Trigger 模块能力建设和 Action 运行时可观测性。AnyCross 是企业级 iPaaS 产品，通过低代码方式连接企业系统，自动化工作流。

**亮点 1 - 接口重构**：
> 我主导了 TriggerDraft 的接口重构，把原来用 extra map 存储差异化配置的方式，重构为 ITriggerDraft 接口 + 不同类型的具体实现（PollingTriggerDraft/InstantTriggerDraft），消除了类型不安全问题，支持了 subscribe 全生命周期的配置化开发。

**亮点 2 - DataFetcher 演进**：
> 分析了 Gmail 两步拉取机制（historyID → messageID → message 详情），识别出原有 G1 DataFetcher 模型不支持多步 API 串联的局限，设计了 G2 模型并完成 Gmail、Slack、Outlook、Teams 四款物料的迁移。

**亮点 3 - Action Trace 方案**：
> 设计了 Redis + RocketMQ + MySQL 的三层 Trace 存储方案：写入 Redis 保障用户调试的即时可见性，通过 RocketMQ 异步持久化到 MySQL，MQ 对 QPS 300-500 的 Action 调用进行流量削峰，保护核心链路稳定。
