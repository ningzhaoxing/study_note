# AIcanTARA - 汽车网络安全 TARA 智能化平台

> 实习单位：北京思格涡野智能科技有限公司（2025.11 - 至今）
> 岗位：Agent/后端开发实习生

---

## 一、项目背景

### 业务背景

**TARA（Threat Analysis and Risk Assessment，威胁分析与风险评估）** 是汽车网络安全领域的核心合规流程，对齐：
- **ISO/SAE 21434**：汽车网络安全工程国际标准
- **UNECE R155**：联合国汽车网络安全法规（进入中国市场必须满足）

**传统痛点**：人工 TARA 需要安全专家逐一分析每个 ECU（电子控制单元）的资产、威胁、攻击路径，耗时长（一辆车可能有 100+ ECU），依赖专家经验，难以标准化。

**AIcanTARA 目标**：通过 AI 自动化覆盖"资产识别 → 攻击树生成 → 风险评估 → 报告导出"的完整流程，将传统人工 TARA **提效约 80%**。

## 三、核心工作内容

### 3.1 Agent 工具体系与复用架构（Skill Middleware）

**问题**：多个 Agent（资产识别、攻击树、风险管理）都需要"获取项目上下文"、"查询资产信息"等基础能力，如果每个 Agent 单独实现工具，重复代码多、维护成本高。

**解决方案**：设计 **Skill Middleware 架构**

```
┌──────────────────────────────────────────────────┐
│              Agent 工具注册层                      │
│                                                  │
│  CoreSkill（核心技能，所有 Agent 共享）            │
│  ├─ ProjectContextTool（获取项目信息）             │
│  └─ AssetQueryTool（查询资产列表）                 │
│                                                  │
│  AssetSkill（资产识别专用）                       │
│  ├─ AssetClassifyTool（资产分类）                  │
│  └─ DamageScenarioTool（损害场景分析）            │
│                                                  │
│  RiskSkill（风险管理专用）                        │
│  ├─ ThreatRatingTool（威胁评级）                  │
│  └─ RiskAcceptanceTool（风险接受判断）            │
└──────────────────────────────────────────────────┘
```

**效果**：
- CoreSkill 中的工具**跨 Agent 复用**，只实现一次
- 新增 Agent 只需声明依赖的 Skill 列表，不需要重新实现工具
- 工具更新只修改对应 Skill，不影响其他 Agent

### 3.2 Agent 模式选型与并发调度

根据不同模块的任务特征，选用不同的 Agent 范式：

| 模块 | 选用范式 | 原因 |
|------|---------|------|
| **资产识别** | ReAct | 需要多轮查询，根据资产类型动态选择分析策略 |
| **风险管理** | ReAct | 需要根据中间分析结果调整风险评估策略 |
| **攻击树生成** | Pipeline | 流程固定：检索Neo4j模板 → LLM生成节点 → 写回Neo4j |
| **并发调度** | Worker Pool | 多用户同时发起分析任务，需要并发控制 |

**Worker Pool 设计**（解决多用户并发分析）：

```go
// 控制并发数，避免 LLM API 调用超额
type AnalysisPool struct {
    semaphore chan struct{} // 信号量
}

func (p *AnalysisPool) RunAnalysis(ctx context.Context, task *AnalysisTask) <-chan Result {
    resultCh := make(chan Result, 1)
    go func() {
        p.semaphore <- struct{}{}  // 获取执行槽
        defer func() { <-p.semaphore }()

        // 每个任务独立的 Agent 实例（状态隔离）
        agent := buildAgentInstance(task)
        result, err := agent.Run(ctx, task.Input)
        resultCh <- Result{Data: result, Err: err}
    }()
    return resultCh
}
```

### 3.3 Agent 指令体系设计（15+ 意图分类 + 状态机）

**问题**：用户与 AIcanTARA 交互时，同一个问题可能对应完全不同的业务操作（查询资产 vs 发起分析 vs 导出报告），需要准确识别意图并编排多步骤流程。

**解决方案**：设计结构化指令体系

```
用户输入
  ↓
意图分类（15+ 意图）
  ├─ 查询类：查资产、查威胁、查报告状态
  ├─ 分析类：触发资产识别、启动风险分析、生成攻击树
  ├─ 编辑类：修改威胁评级、更新资产信息
  └─ 导出类：生成TARA报告、导出Excel
  ↓
状态机流转
  ├─ 分析任务：等待确认 → 分析中 → 待审核 → 已完成
  └─ 每个状态转移有明确的前置条件和后置操作
```

**工程约束**（保障输出质量）：
- **Prompt 注入防护**：用户输入经过过滤，防止通过输入修改系统指令
- **零缓存原则**：LLM 推理输出不缓存，每次重新生成（避免过时的分析结果）
- **分批策略**：资产数量多时按批次分析，避免单次 Token 超限

### 3.4 试用版功能管控（JWT + 版本路由）

**需求**：平台需要区分试用版和付费版用户，不同版本对应不同的功能集合（如试用版限制分析资产数量、不支持报告导出等）。

**方案**：扩展 JWT Token + 抽象层设计

```go
// JWT 中增加 plan 字段
type Claims struct {
    UserID string `json:"user_id"`
    Plan   string `json:"plan"` // "trial" | "standard" | "enterprise"
    jwt.RegisteredClaims
}

// LicenseConfig 抽象层（封装版本差异）
type LicenseConfig interface {
    MaxAssetsPerAnalysis() int
    CanExportReport() bool
    CanUseAdvancedTemplates() bool
}

// Service 层只依赖 License 对象，不感知具体版本
func (s *AnalysisService) StartAnalysis(ctx context.Context, req *AnalysisRequest) error {
    license := ctx.Value(licenseKey).(LicenseConfig)
    if len(req.Assets) > license.MaxAssetsPerAnalysis() {
        return ErrExceedsLimit
    }
    // ...
}
```

**设计价值**：Service 层不感知版本标识（"trial"/"standard"），只依赖 License 对象的能力接口，预留多维度扩展能力（未来可以按模块单独限制）。

## 四、技术亮点总结（面试口述版本）

**开场（30秒）**：
> AIcanTARA 是面向汽车网络安全领域的 TARA 智能化平台，核心是用 AI Agent 自动化完成从资产识别到报告生成的完整分析流程，对齐 ISO/SAE 21434 标准。我在项目中主要负责 Agent 模块的架构设计和工程落地。

**亮点 1 - Skill Middleware**：
> 多个 Agent 需要共享基础工具（项目上下文、资产查询），我设计了 Skill Middleware 架构，将工具按领域模块化，核心能力统一注册，新增 Agent 只需声明依赖的 Skill，降低了接入成本。

**亮点 2 - 范式选型**：
> 我根据任务特征做了差异化的 Agent 范式选型：资产识别需要动态决策用 ReAct，攻击树生成流程固定用 Pipeline，多用户并发用 Worker Pool。这种组合避免了"一刀切"的过度设计。

**亮点 3 - 版本管控**：
> 试用版功能管控中，我设计了 LicenseConfig 抽象层，Service 层只依赖 License 接口而非硬编码版本判断，保留了多维度扩展能力。
