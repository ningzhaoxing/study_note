# goroutine 使用与最佳实践

> goroutine 是 Go 并发的基石，但"能用 goroutine"不等于"应该随意创建 goroutine"。本文聚焦 goroutine 的使用场景、生命周期管理、泄漏排查和性能优化。

---

## 一、goroutine 基础

### 1.1 创建方式

```go
// 最简单的创建
go func() {
    fmt.Println("hello from goroutine")
}()

// 传参（推荐闭包传参而不是捕获外部变量）
for i := 0; i < 5; i++ {
    i := i  // 关键：重新声明，避免闭包捕获同一变量
    go func() {
        fmt.Println(i)
    }()
}
```

### 1.2 goroutine vs 线程资源对比

| 维度 | goroutine | OS Thread |
|-----|----------|-----------|
| 初始栈 | **2~8 KB**，动态伸缩 | 固定 1~8 MB |
| 创建耗时 | ~1 μs | ~100 μs |
| 上下文切换 | ~100 ns（用户态） | ~1 μs（内核态） |
| 可创建数量 | 数十万级别 | 受 OS 限制（通常 < 1万） |

## 二、goroutine 生命周期管理

### 2.1 使用 context 控制退出

```go
func worker(ctx context.Context, id int) {
    for {
        select {
        case <-ctx.Done():
            fmt.Printf("worker %d 退出: %v\n", id, ctx.Err())
            return
        default:
            // 正常工作
            doWork()
        }
    }
}

func main() {
    ctx, cancel := context.WithCancel(context.Background())
    defer cancel()

    for i := 0; i < 3; i++ {
        go worker(ctx, i)
    }

    time.Sleep(5 * time.Second)
    cancel() // 通知所有 worker 退出
}
```

### 2.2 使用 WaitGroup 等待完成

```go
var wg sync.WaitGroup

for i := 0; i < 10; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()  // 确保一定调用
        doWork(id)
    }(i)
}

wg.Wait()  // 阻塞直到所有 goroutine 完成
```

### 2.3 goroutine 中的 panic 处理

```go
// goroutine 内的 panic 不会被外部 recover 捕获！
// 必须在每个 goroutine 内部 recover
func safeGo(fn func()) {
    go func() {
        defer func() {
            if r := recover(); r != nil {
                log.Printf("goroutine panic: %v\n%s", r, debug.Stack())
            }
        }()
        fn()
    }()
}
```

## 三、goroutine 泄漏

### 3.1 什么是 goroutine 泄漏

goroutine 启动后由于某种原因**无法正常退出**，长期占用内存和调度资源，随时间积累导致内存增长、性能下降。

### 3.2 常见泄漏场景

**场景 1：channel 发送方退出，接收方阻塞**
```go
// 错误示例：ch 没有关闭，goroutine 永久阻塞
func leak() {
    ch := make(chan int)
    go func() {
        val := <-ch  // 永久阻塞！
        fmt.Println(val)
    }()
    // ch 从未发送数据，goroutine 泄漏
}

// 修复：通过 context 控制退出
func fixed(ctx context.Context) {
    ch := make(chan int)
    go func() {
        select {
        case val := <-ch:
            fmt.Println(val)
        case <-ctx.Done():
            return  // context 取消时退出
        }
    }()
}
```

**场景 2：HTTP 请求 goroutine 无超时**
```go
// 危险：若下游服务不响应，goroutine 永久挂起
go func() {
    resp, _ := http.Get("http://slow-service.example.com")
    process(resp)
}()

// 安全：设置超时
go func() {
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    req, _ := http.NewRequestWithContext(ctx, "GET", "http://slow-service.example.com", nil)
    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        return  // 超时或取消时退出
    }
    process(resp)
}()
```

**场景 3：sync.WaitGroup 计数错误**
```go
// 错误：wg.Add 在 goroutine 内部，可能在 Wait 之后才 Add
go func() {
    wg.Add(1)  // 错误位置！
    defer wg.Done()
}()
wg.Wait()  // 可能提前返回

// 正确：在 goroutine 外部 Add
wg.Add(1)
go func() {
    defer wg.Done()
}()
wg.Wait()
```

### 3.3 泄漏检测方法

```go
// 1. 监控 goroutine 数量
fmt.Println("goroutine count:", runtime.NumGoroutine())

// 2. pprof goroutine 分析（推荐）
import _ "net/http/pprof"
// 访问 http://localhost:6060/debug/pprof/goroutine?debug=2
// 可以看到所有 goroutine 的调用栈

// 3. 使用 goleak 库（单元测试中检测泄漏）
import "go.uber.org/goleak"
func TestMyFunc(t *testing.T) {
    defer goleak.VerifyNone(t)
    // 测试代码...
}
```

## 四、goroutine 池

### 4.1 为什么需要 goroutine 池

虽然 goroutine 创建开销低，但在以下场景仍需控制数量：
- **防止 goroutine 数量无限增长**：高并发请求涌入时，不加限制的 `go func()` 会创建大量 goroutine
- **复用已有 goroutine**：避免频繁创建销毁的开销
- **背压控制**：通过池大小限制并发，实现流量控制

### 4.2 简单实现

```go
type WorkerPool struct {
    taskCh chan func()
    wg     sync.WaitGroup
}

func NewWorkerPool(size int) *WorkerPool {
    pool := &WorkerPool{
        taskCh: make(chan func(), size*2), // 带缓冲的任务队列
    }
    for i := 0; i < size; i++ {
        pool.wg.Add(1)
        go func() {
            defer pool.wg.Done()
            for task := range pool.taskCh {
                task()
            }
        }()
    }
    return pool
}

func (p *WorkerPool) Submit(task func()) {
    p.taskCh <- task
}

func (p *WorkerPool) Shutdown() {
    close(p.taskCh)  // 关闭后 worker goroutine 自然退出
    p.wg.Wait()
}
```

### 4.3 生产级方案

推荐使用 `ants` 库（业界标准 goroutine 池）：
```go
import "github.com/panjf2000/ants/v2"

pool, _ := ants.NewPool(10000)
defer pool.Release()

for i := 0; i < 1000000; i++ {
    pool.Submit(func() {
        doWork()
    })
}
```

## 五、面试高频问题

### Q1：goroutine 一定比线程高效吗？

**答**：在大量并发的场景下是的，但不是绝对的：
- goroutine 调度仍有开销（Go runtime 维护调度循环）
- 若任务极少（< 10 个），直接用线程或单线程可能更简单
- I/O 密集型场景 goroutine 优势最大；CPU 密集型任务，goroutine 并不比线程快（最终还是跑在 M 上）

### Q2：如何保证 goroutine 安全退出？

**答**：三种方式：
1. **Context 取消**：通过 `ctx.Done()` channel 通知退出（最推荐，可传播取消信号）
2. **关闭 channel**：通过关闭 channel 广播退出信号（适合一对多通知）
3. **超时控制**：`context.WithTimeout` 或 `context.WithDeadline`

### Q3：开多少个 goroutine 合适？

**答**：取决于任务类型：

**CPU 密集型 → goroutine 数量 ≈ `GOMAXPROCS`（核心数）**

CPU 密集型任务（纯计算、加密、压缩等）特点是 goroutine **一直在跑，不会主动让出 P**：

```
4 核机器，GOMAXPROCS=4

goroutine 数 = 4：4 个 P 满载运行 ✅
goroutine 数 = 400：仍然只有 4 个 P 同时执行
  → 剩余 396 个在队列排队
  → P 不断在 goroutine 间切换（上下文切换开销）
  → 实际吞吐量反而下降 ❌
```

瓶颈是**核心数**，多余的 goroutine 只增加调度开销。

**I/O 密集型 → goroutine 可以远多于核心数**

I/O 密集型任务（网络、数据库、文件）特点是 goroutine **大量时间在等待**，等待时进入 `Gwaiting` 状态，**主动释放 P**：

```
goroutine A 发起网络请求 → Gwaiting，P 释放
                              ↓
                         P 立刻调度 goroutine B 运行
                         goroutine B 等数据库 → P 释放
                              ↓
                         P 再调度 goroutine C ...
```

1 个 P 可以**交替服务**大量处于等待中的 goroutine，CPU 始终有活干。瓶颈是**等待时间**，多开 goroutine 让 CPU 在等待间隙持续工作，提升吞吐。

**实践原则**：通过 goroutine 池控制上限，避免无限创建
