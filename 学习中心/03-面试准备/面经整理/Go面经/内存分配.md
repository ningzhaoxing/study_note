# Go 内存分配器

> Go 使用**分层内存分配器**，借鉴 tcmalloc 设计，核心思路是**线程本地缓存 + 分级管理**，大幅减少全局锁竞争。面试重点：三级缓存架构（mcache/mcentral/mheap）→ 分配流程 → 大/小/微对象的不同路径。

---

## 一、整体架构

```
Go 内存分配器层级（由小到大）：

用户 goroutine 申请内存
        ↓
  mcache（P 的线程本地缓存，无锁）
  ├─ tiny allocator（< 16B 微对象）
  └─ 67 种 size class 的 mspan 缓存
        ↓（mcache 缺货时）
  mcentral（每种 size class 一个，有锁）
  └─ 管理同一 size class 的所有 mspan
        ↓（mcentral 缺货时）
  mheap（全局堆管理器，有锁）
  └─ 管理从 OS 申请的大块内存（arena）
        ↓
  OS（mmap 系统调用）
```

---

## 二、核心概念

### 2.1 内存布局：arena / span / object

```
arena（堆内存区域）
  │  Go 向 OS 申请的大块连续内存
  │  64 位系统：最大 512 GB（被分割成 64MB 的 arena chunk）
  │
  ├─ mspan（内存管理单元）
  │    │  一个或多个连续的 8KB page 组成一个 mspan
  │    │  同一个 mspan 内只存放同一 size class 的对象
  │    │
  │    └─ object slot（对象槽位）
  │         同一 mspan 被均匀切割成多个同大小的 slot
  │         每个 slot 存放一个用户对象
```

### 2.2 Size Class（67 种规格）

Go 预定义了 67 种 size class，覆盖 8B ~ 32KB 的对象：

| Size Class | 对象大小上限 | 每个 mspan 的 page 数 |
|-----------|------------|---------------------|
| 1 | 8 B | 1 |
| 2 | 16 B | 1 |
| ... | ... | ... |
| 32 | 512 B | 4 |
| ... | ... | ... |
| 67 | 32 KB | 4 |

> 按 size class 分配的好处：相同大小的对象放在同一个 mspan，减少内存碎片，GC 清除时也更高效。

### 2.3 mspan 结构

```go
type mspan struct {
    next       *mspan        // 链表的下一个 span
    prev       *mspan        // 链表的上一个 span
    startAddr  uintptr       // span 的起始地址
    npages     uintptr       // span 包含的 page 数量

    nelems     uintptr       // 可存放的对象数量（slot 总数）
    freeindex  uintptr       // 下一个空闲 slot 的索引

    allocBits  *gcBits       // 位图：每个 bit 对应一个 slot，1=已分配，0=空闲
    gcmarkBits *gcBits       // 位图：GC 标记用，1=可达（存活），0=垃圾

    spanclass  spanClass     // size class 编号（包含是否含指针信息）
}
```

## 三、三级缓存详解

### 3.1 mcache（一级缓存，P 持有，无锁）

每个 **P**（逻辑处理器）持有一个 `mcache`，goroutine 在 P 上运行时直接从 mcache 分配，**无需加锁**：

```go
type mcache struct {
    // 微对象分配器（用于 < 16B 且不含指针的对象）
    tiny       uintptr  // 当前微对象分配块的起始地址
    tinyoffset uintptr  // 当前已用偏移
    tinyAllocs uintptr  // 微对象分配计数

    // 67 种 size class × 2（含指针/不含指针）= 134 个 mspan 缓存
    alloc [numSpanClasses]*mspan
}
```

分配流程：
```
1. 找到对应 size class 的 mspan（alloc[spc]）
2. 从 mspan.freeindex 取一个空闲 slot
3. 更新 freeindex，返回地址
（全程无锁，极快）
```

若对应 size class 的 mspan 已用完，向 mcentral 申请新的 mspan。

### 3.2 mcentral（二级缓存，按 size class，有锁）

每种 size class 对应一个 `mcentral`，管理该规格的所有 mspan：

```go
type mcentral struct {
    spanclass spanClass
    partial   [2]spanSet  // 有空闲 slot 的 mspan（两个集合，GC 轮转用）
    full      [2]spanSet  // 没有空闲 slot 的 mspan
}
```

当 mcache 需要新 mspan 时：
```
1. 从 mcentral.partial 取一个有空闲的 mspan 给 mcache
2. 若 partial 为空，从 mheap 申请新的 pages 并切割成新 mspan
```

### 3.3 mheap（三级，全局，有锁）

```go
type mheap struct {
    // 空闲 page 管理（基数树 radix tree）
    free mTreap

    // 每种 size class 的 mcentral
    central [numSpanClasses]struct {
        mcentral mcentral
        // ...
    }

    // arena 管理
    arenas [1 << arenaL1Bits]*[1 << arenaL2Bits]*heapArena
}
```

mheap 负责：
1. 管理从 OS 申请的所有内存（arena）
2. 为 mcentral 提供新的 page 资源
3. 管理 > 32KB 的大对象（直接从 mheap 分配，不经过 mcache/mcentral）

## 四、对象分配流程

### 4.1 按对象大小分三类

| 对象大小 | 分类 | 分配路径 |
|---------|------|---------|
| **< 16 B** 且不含指针 | 微对象（tiny） | mcache 的 tiny allocator（多个微对象拼入同一个 16B 块） |
| **16 B ~ 32 KB** | 小对象 | mcache → mcentral → mheap |
| **> 32 KB** | 大对象 | 直接从 mheap 分配（不经过 mcache/mcentral） |

### 4.2 小对象分配步骤

```
1. 计算对象属于哪个 size class
2. 从当前 P 的 mcache 中取对应 mspan 的空闲 slot
   → 命中：直接返回（无锁，最快）
3. mcache 未命中：向 mcentral 申请该 size class 的新 mspan
   → mcentral 有空闲 mspan：返回给 mcache
4. mcentral 也没有：向 mheap 申请新的连续 page
   → mheap 将 page 切割成新 mspan，返回给 mcentral，再给 mcache
5. mheap 也不够：向 OS 申请新的 arena 内存（mmap）
```

### 4.3 微对象分配（tiny allocator）

对于 `< 16B` 且**不含指针**的小对象（如 `int8`, `bool`, 小 string header 等），使用 tiny allocator：

```
mcache.tiny：指向当前 16B 的分配块
mcache.tinyoffset：当前已用的偏移量

分配微对象时：
  若 tinyoffset + size <= 16：直接在当前 16B 块中分配，tinyoffset += size
  否则：申请新的 16B 块（从 size class 2 的 mspan 中取）
```

**好处**：多个微对象共享一个 slot，大幅减少小对象的内存开销和 GC 压力。

## 五、面试高频问题

### Q1：Go 内存分配为什么快？

**答**：
1. **线程本地缓存（mcache）**：goroutine 分配内存时直接从 P 的 mcache 取，完全无锁
2. **分 size class**：相同大小的对象在同一 mspan 中，分配只需取 slot，O(1)
3. **对象复用**：GC 后释放的 slot 会被下一次分配复用，不频繁向 OS 申请内存

### Q2：mcache 和 P 的关系？

**答**：mcache 是 P 的私有缓存，一个 P 持有一个 mcache。goroutine 在某个 P 上运行时，其内存分配操作直接使用该 P 的 mcache，不需要任何锁。P 被 M 使用时才有意义，P 空闲时 mcache 也闲置。

### Q3：大对象（> 32KB）为什么不走 mcache？

**答**：大对象占用大量内存，若放入 mcache 会频繁换入换出，缓存效果差。大对象直接从 mheap 分配（需要加锁），本身大对象的创建频率也低，加锁开销在整体比例中较小。

### Q4：Go 内存分配和 GC 如何配合？

**答**：
- mspan 中的 `gcmarkBits` 和 `allocBits` 是 GC 与分配器的接口：GC 标记阶段设置 `gcmarkBits`，清除阶段将 `gcmarkBits` 赋值给 `allocBits`（存活的 bit 标为 1=已分配，死掉的 bit 标为 0=空闲）
- 下次分配时，直接查 `allocBits` 找空闲 slot，已被 GC 释放的 slot 可以立即复用
