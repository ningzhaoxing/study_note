# Go 垃圾回收（GC）机制

> Go GC 经过多个版本演进，核心是**并发三色标记 + 混合写屏障**。面试重点：三色标记流程 → 为什么需要写屏障 → 混合写屏障解决什么问题 → STW 在哪里发生。

---

## 一、GC 版本演进

| 版本         | 机制                               | 核心问题                      |
| ---------- | -------------------------------- | ------------------------- |
| Go 1.3 及之前 | **标记-清除 + 全程 STW**               | STW 时间过长（整个标记+清除期间暂停程序）   |
| Go 1.5     | **三色标记 + 插入写屏障**（堆对象），仅标记结束时 STW | 栈对象无写屏障，栈扫描需要 STW re-scan |
| Go 1.8     | **混合写屏障**（堆+栈对象均覆盖），STW 极短       | 当前版本，STW < 1ms            |

---

## 二、标记-清除算法（v1.3 前，了解历史）

### 2.1 算法流程

```
1. STW（Stop The World）：暂停所有 goroutine
2. 标记（Mark）：从根对象（全局变量、栈上变量）出发，
                 递归标记所有可达对象
3. 清除（Sweep）：释放未被标记的对象
4. 恢复程序（Start The World）
```

### 2.2 缺点

- **程序暂停时间长**：整个标记+清除过程期间，所有 goroutine 都不能运行
- **全堆扫描**：每次 GC 都需要遍历整个堆，耗时与堆大小成正比
- **堆碎片**：清除后内存不连续

## 三、三色标记算法（v1.5+）

### 3.1 三种颜色的含义

| 颜色     | 含义                   | 存放位置       |
| ------ | -------------------- | ---------- |
| **白色** | 未被访问（潜在垃圾）           | 初始所有对象都是白色 |
| **灰色** | 已被发现但子对象未扫描完         | 标记队列（待处理）  |
| **黑色** | 已扫描完毕，其引用的所有子对象也都处理过 | 已标记集合      |

### 3.2 标记过程

```
初始状态：所有对象 → 白色

Step 1：将根对象（全局变量、栈变量等）加入灰色队列
         白色 → 灰色

Step 2：循环处理灰色队列：
  取出一个灰色对象 A
    → 将 A 引用的所有白色对象变为灰色（加入队列）
    → 将 A 自身变为黑色（处理完毕）

Step 3：灰色队列为空时，剩余白色对象 = 不可达 = 垃圾
        释放所有白色对象
```

```
示例：
初始：
  root → A(白) → B(白) → D(白)
             ↘ C(白)

Step1：root 扫描，A 变灰
  灰色: [A]  黑色: []  白色: [B,C,D]

Step2：处理 A，B/C 变灰，A 变黑
  灰色: [B,C]  黑色: [A]  白色: [D]

Step3：处理 B，D 变灰，B 变黑
  灰色: [C,D]  黑色: [A,B]  白色: []

Step4：处理 C（无子对象），C 变黑；处理 D，D 变黑
  灰色: []  黑色: [A,B,C,D]  白色: []

所有可达对象都是黑色，没有白色垃圾。
```

### 3.3 并发标记的问题：对象消失

三色标记可以与程序并发进行，但程序在标记期间可能修改引用关系，出现**对象消失**问题：

```
条件1 + 条件2 同时满足时，白色对象被误删：
 
条件1：黑色对象 B 新增了对白色对象 D 的引用
       （B 已经扫描完，不会再被扫描，D 不会被标记为灰色）

条件2：灰色对象 A（原本引用 D）删除了对 D 的引用
       （如果通过 A 到 D 的路径被删除）

结果：D 是可达的（B → D），但 GC 不知道，最终 D 被当成白色垃圾释放 → 悬挂指针！
```

**解决思路**：破坏上述两个条件之一：
- **强三色不变式**：黑色对象不能直接引用白色对象（使用**插入写屏障**保证）
- **弱三色不变式**：所有被黑色对象引用的白色对象，必须有从灰色对象可达的路径（使用**删除写屏障**保证）

## 四、写屏障（Write Barrier）

写屏障是在**对象引用关系变化时**自动触发的一小段代码钩子，用于维护三色不变式。

### 4.1 插入写屏障（Go 1.5）

**触发时机**：黑色对象新增对白色对象的引用时

**操作**：将被引用的白色对象**变为灰色**（防止被漏标）

```
当执行 B.field = D（B 是黑色，D 是白色）时：
  触发写屏障：将 D 变为灰色
  → 保证黑色对象不会持有未标记的白色对象引用
```

**缺陷**：**栈上对象不启用插入写屏障**（性能考虑，栈操作非常频繁）。因此 goroutine 栈上的新引用不被跟踪，标记结束后需要 **STW 重新扫描所有栈**（re-scan），这段时间仍然较长。

### 4.2 删除写屏障（Dijkstra，了解）

**触发时机**：对象删除对另一对象的引用时

**操作**：被删除引用的对象如果是白色，先变为灰色（防止未来无法找到它）

```
当执行 A.field = nil（删除 A → D 的引用，A 是灰色，D 是白色）时：
  触发写屏障：将 D 变为灰色
  → 保证灰色对象删除引用后，被删除的白色对象不会立即消失
```

**缺陷**：回收精度低，当前轮 GC 结束后某些对象本可以回收，却因为被保护变成灰色，要等下一轮 GC 才能回收。

### 4.3 混合写屏障（Go 1.8，当前版本）

**混合写屏障 = 插入写屏障 + 删除写屏障的结合**，并且**栈上操作也生效**：

**规则**：
1. GC 开始时，将**所有栈上的可达对象**直接标记为**黑色**（stack scanning 阶段）
2. GC 期间，堆上**新创建的对象**直接标记为**黑色**（不需要 re-scan）
3. 堆上**被引用的对象**（插入）变为灰色
4. 堆上**被删除引用的对象**变为灰色

```
混合写屏障伪代码（堆对象的写操作）：

writePointer(slot, ptr):
    shade(*slot)  // 删除写屏障：旧值变灰
    if current goroutine's stack is grey:
        shade(ptr)  // 插入写屏障：新值变灰
    *slot = ptr
```

**效果**：不再需要 STW re-scan 栈，STW 仅发生在 GC 启动和结束的极短时刻（< 1ms）。

## 五、完整 GC 流程（Go 1.8+）

```
1. GC 触发（堆内存达到阈值或手动 runtime.GC()）
         ↓
2. Mark Setup（STW，极短）
   - 开启写屏障
   - 将 goroutine 栈上对象标记为黑色
   - 将全局变量等根对象加入灰色队列
         ↓
3. Marking（并发，与程序同时运行）
   - 后台 GC goroutine 并发处理灰色队列
   - 写屏障跟踪期间发生的引用变化
   - Mutator Assist：若用户 goroutine 分配内存过快，辅助参与 GC 标记
         ↓
4. Mark Termination（STW，极短）
   - 处理最后的灰色对象
   - 关闭写屏障
         ↓
5. Sweeping（并发）
   - 后台 goroutine 并发清除白色对象
   - 清除可与用户程序并发进行
```

## 六、Mutator Assist（辅助 GC）

**问题**：若用户 goroutine 分配内存的速度比 GC 标记的速度更快，堆会无限增长。

**解决**：当 GC 进行中且内存分配超过某个阈值时，用户 goroutine 需要**辅助执行一部分 GC 标记工作**（按照分配量的比例），才能继续分配内存。

这是一种**背压机制**，保证 GC 进度不落后于内存分配速度。

---

## 七、GC 触发条件

| 触发方式 | 说明 |
|---------|------|
| **内存阈值（默认）** | 当堆内存增长到上次 GC 后的 `GOGC%`（默认 100%，即翻倍）时触发 |
| **手动触发** | `runtime.GC()` 强制触发一次完整 GC |
| **定时触发** | sysmon 检测到超过 2 分钟未 GC，强制触发 |

```go
// 调整 GC 目标（GOGC=200 表示内存增长到 3 倍才 GC，减少 GC 频率）
import "runtime/debug"
debug.SetGCPercent(200)

// Go 1.19 引入：设置最大堆内存上限（触发 GC 的绝对阈值）
debug.SetMemoryLimit(4 * 1024 * 1024 * 1024)  // 4GB
```

## 八、面试高频问题

### Q1：Go GC 的核心算法是什么？

**答**：**并发三色标记-清除 + 混合写屏障**。通过三色标记（白/灰/黑）并发地标记可达对象，通过混合写屏障（同时包含插入和删除写屏障）维护三色不变式，保证并发标记的正确性，最终清除白色（不可达）对象。

### Q2：什么是 STW？Go 中 STW 发生在哪些时刻？

**答**：STW（Stop The World）是暂停所有用户 goroutine 的操作。在 Go 1.8+ 中，STW 只发生在两个极短的阶段：
- **Mark Setup 阶段**：开启写屏障、初始化标记工作
- **Mark Termination 阶段**：处理最后的灰色对象、关闭写屏障
两个 STW 阶段的时间都 < 1ms。

### Q3：三色标记中，为什么需要写屏障？

**答**：三色标记与程序并发进行时，程序可能修改对象引用关系。若黑色对象新增了对白色对象的引用（黑引白），同时删除了灰色对象到该白色对象的路径，那么 GC 就会找不到该白色对象，将其误认为垃圾回收，造成悬挂指针。写屏障通过在引用关系变化时自动将相关对象变灰，维护三色不变式，防止对象消失。

### Q4：混合写屏障相比 Go 1.5 的改进是什么？

**答**：Go 1.5 的插入写屏障不覆盖栈上的操作，导致标记结束时需要 STW 重新扫描所有 goroutine 的栈（因为栈上的引用变化未被跟踪）。混合写屏障（Go 1.8）在 GC 开始时直接将所有栈上对象标记为黑色，期间堆上新创建的对象也直接标记为黑色，彻底消除了 STW re-scan 的需要，将 STW 时间降至 < 1ms。

### Q5：GC 频率很高，如何优化？

**答**：
1. **减少内存分配**：复用对象（sync.Pool）、提前预分配（make([]T, 0, n)）、避免在热路径创建临时对象
2. **调大 GOGC**：`GOGC=200` 让堆内存增长更多才触发 GC，降低 GC 频率（以更高内存使用为代价）
3. **设置内存上限**：Go 1.19+ 的 `debug.SetMemoryLimit` 设置绝对上限，在内存允许范围内尽量少触发 GC
4. **避免大量小对象**：大量小对象让 GC 扫描时间增加；可以合并为大结构体
