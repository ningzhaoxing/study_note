# Go 内存逃逸分析

> 逃逸分析决定变量分配在**栈**还是**堆**，直接影响 GC 压力和程序性能。面试重点：**什么是逃逸 → 四种逃逸场景 → 如何判断是否逃逸 → 逃逸的性能影响**。

---

## 一、栈 vs 堆

| 维度 | 栈（Stack） | 堆（Heap） |
|-----|------------|----------|
| 分配方式 | 移动 SP 指针（O(1)，极快） | 内存分配器（mcache → mcentral → mheap） |
| 回收方式 | 函数返回时自动回收（无 GC 参与） | GC 三色标记清除 |
| 生命周期 | 与函数调用帧一致 | 由 GC 管理，与对象引用关系相关 |
| 访问性能 | 快（CPU 缓存友好，局部性好） | 相对较慢（堆内存分散，缓存不友好） |
| 大小限制 | goroutine 初始 2~8 KB（可增长到最大 1GB） | 受可用内存限制 |

**结论**：栈分配远比堆分配快，且不增加 GC 压力。Go 编译器会尽可能把变量分配在栈上，只有在**必须**在函数返回后继续存活时，才"逃逸"到堆上。

---

## 二、什么是逃逸

**内存逃逸**（Memory Escape）：编译器通过**逃逸分析**确定，某个局部变量的生命周期超出了当前函数的栈帧，因此将其分配在**堆**上而不是栈上。

```go
// 不逃逸：x 在函数返回后不再被使用，分配在栈上
func noEscape() int {
    x := 42  // 栈分配
    return x  // 返回值，不是指针
}

// 逃逸：返回 x 的指针，函数返回后外部仍可访问 x
func escape() *int {
    x := 42  // 堆分配！编译器检测到 x 需要在函数外存活
    return &x
}
```

## 三、四种常见逃逸场景

### 3.1 指针逃逸（最常见）

返回局部变量的指针：

```go
func newUser(name string) *User {
    u := User{Name: name}  // u 逃逸到堆
    return &u
}

// 查看逃逸信息：
// go build -gcflags="-m" ./...
// 输出：./main.go:2:3: moved to heap: u
```

**原因**：`u` 在函数返回后仍被调用者持有，不能随函数栈帧销毁。

### 3.2 interface{} / any 导致逃逸

将具体类型赋值给 interface 时，若编译器无法确定具体类型，会逃逸：

```go
func printValue(v interface{}) {
    fmt.Println(v)
}

x := 42
printValue(x)  // x 逃逸！int 被装箱为 interface{}，装箱值存在堆上
```

```go
// 特殊情况：fmt.Println 系列几乎总是引发逃逸
// 因为其参数类型是 ...interface{}
fmt.Println("hello")    // "hello" 逃逸
fmt.Printf("%d", 42)    // 42 逃逸
```

### 3.3 栈空间不足（大对象）

如果变量太大，goroutine 的栈放不下，会逃逸到堆：

```go
// 小数组：栈分配
func smallArray() {
    var arr [100]int  // 800 bytes，栈分配
    _ = arr
}

// 大数组：堆分配（逃逸）
func largeArray() {
    var arr [10000]int  // 80 KB，超出栈空间阈值，逃逸
    _ = arr
}

// 运行时才知道大小的 slice：逃逸
func dynamicSlice(n int) {
    s := make([]int, n)  // n 在运行时确定，编译器保守处理，可能逃逸
    _ = s
}
```

### 3.4 闭包捕获外部变量

```go
func makeCounter() func() int {
    count := 0  // count 逃逸到堆
    return func() int {
        count++
        return count
    }
}

counter := makeCounter()
counter()  // 1
counter()  // 2
// count 在 makeCounter 返回后仍被闭包引用，必须在堆上
```

**闭包捕获规则**：
- 若闭包在函数内部使用但不逃出函数作用域，被捕获的变量可能不逃逸
- 若闭包赋值给外部变量（如函数返回值），被捕获的变量必须逃逸

## 四、如何检测逃逸

### 4.1 编译器逃逸分析报告

```bash
# -m：输出逃逸分析结果
go build -gcflags="-m" ./...

# -m -m：更详细的输出
go build -gcflags="-m -m" ./...

# 只分析某个文件
go tool compile -m main.go
```

**输出解读**：
```
./main.go:5:3: moved to heap: u      → u 逃逸到堆
./main.go:8:13: x escapes to heap    → x 逃逸
./main.go:12:2: does not escape      → 不逃逸，栈分配
./main.go:15:14: inlining call to f  → 函数被内联（内联后局部变量可能不逃逸）
```

### 4.2 通过 pprof 分析堆分配

```go
import _ "net/http/pprof"

// 访问 http://localhost:6060/debug/pprof/heap
// 查看堆上对象的分配情况，找出频繁分配的热点
```

### 4.3 benchmark 验证

```go
func BenchmarkNoEscape(b *testing.B) {
    for i := 0; i < b.N; i++ {
        x := 42    // 栈分配
        _ = x
    }
}

func BenchmarkEscape(b *testing.B) {
    for i := 0; i < b.N; i++ {
        x := new(int)  // 堆分配
        *x = 42
        _ = x
    }
}
// 运行：go test -bench=. -benchmem
// 堆分配版本：allocs/op > 0，ns/op 更高
```

## 五、逃逸对性能的影响

```
逃逸的代价：
1. 分配开销：堆分配需要经过 mcache → mcentral → mheap，比栈分配慢
2. GC 压力：堆上的对象需要 GC 扫描和清除，大量小对象逃逸会频繁触发 GC
3. 缓存 miss：堆对象分散在内存各处，CPU 缓存利用率低
```

### 优化策略

```go
// 1. 使用 sync.Pool 复用频繁分配的对象
var bufPool = sync.Pool{
    New: func() interface{} {
        return make([]byte, 0, 4096)
    },
}

buf := bufPool.Get().([]byte)
defer bufPool.Put(buf[:0])

// 2. 预分配足够容量，避免 append 时多次扩容（每次扩容都可能触发新的堆分配）
s := make([]int, 0, 1000)

// 3. 避免在热路径使用 interface{}
// 差：
func process(v interface{}) { ... }
// 好：
func processInt(v int) { ... }

// 4. 内联小函数（inlining 可以消除部分逃逸）
// go:noinline 可以禁止内联（用于测试对比）
```

## 六、面试高频问题

### Q1：什么情况下变量会逃逸到堆？

**答**：四种主要场景：
1. **指针逃逸**：函数返回局部变量的指针
2. **interface 逃逸**：将值赋给 interface{} / any 类型
3. **大对象**：变量太大，超出栈空间阈值
4. **闭包捕获**：闭包引用的外部变量，且闭包本身逃出了原函数作用域

### Q2：逃逸对程序有什么影响？

**答**：
- **性能下降**：堆分配比栈分配慢，且存在 GC 开销
- **GC 压力增加**：堆上的对象越多，GC 工作量越大，可能增加 STW 或 GC 辅助的时间
- **内存占用增加**：堆对象需要额外的内存管理元数据

### Q3：如何判断一个变量是否逃逸？

**答**：使用 `go build -gcflags="-m"` 查看编译器的逃逸分析输出。`moved to heap: x` 或 `x escapes to heap` 表示逃逸；`does not escape` 表示在栈上分配。

### Q4：函数返回指针一定会逃逸吗？

**答**：通常会，但**内联（inlining）** 可以消除逃逸。如果编译器将函数内联展开到调用方，被内联的局部变量生命周期就等同于调用方的作用域，可能不需要逃逸到堆。可以用 `go build -gcflags="-m"` 查看是否有 `inlining call to xxx` 的提示。

### Q5：slice 一定在堆上吗？

**答**：不一定。slice 由三字段的 header（pointer, len, cap）组成，header 本身可以在栈上。底层数组是否在堆上，取决于：
- `make([]int, 3)` 且大小固定且较小：底层数组可能在**栈**上
- `make([]int, n)`（n 是变量）：编译器无法确定大小，底层数组分配在**堆**上
- 若 slice 从函数返回或赋给外部变量：必然逃逸到堆
