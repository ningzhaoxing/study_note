# 单例模式

> **核心思想**：确保一个类只有**一个实例**，并提供一个全局访问点。

## 解决的问题

某些资源（配置中心、连接池、日志器）如果被多次实例化，会导致：
- 数据不一致（多份配置互相覆盖）
- 资源浪费（建立多个不必要的 DB 连接）
- 竞争条件（多个实例同时写同一状态）

## UML 结构

```plantuml
class Singleton {
  - instance: Singleton
  - Singleton()
  + GetInstance(): Singleton
}

Singleton --> Singleton : 返回唯一实例
```

## 三种实现方式

### 1. 饿汉模式（Eager Initialization）

程序启动时就创建实例，适合初始化轻量、必然被使用的场景。

```go
var instance = &InstanceClass{}

type InstanceClass struct{}

func GetInstance() *InstanceClass {
    return instance
}
```

✅ 优点：无并发问题，实现最简单
❌ 缺点：若该实例从未被使用，会造成内存浪费

---

### 2. 懒汉模式（Lazy Initialization）

需要时才创建，使用 `sync.Once` 保证并发安全。

```go
var (
    instance *InstanceClass
    once     sync.Once
)

type InstanceClass struct{}

func GetInstance() *InstanceClass {
    once.Do(func() {
        instance = &InstanceClass{}
    })
    return instance
}
```

✅ 优点：延迟初始化，节省资源；`sync.Once` 保证线程安全且高效
❌ 缺点：相比饿汉稍复杂

> 💡 **Go 推荐做法**：使用 `sync.Once`，它内部通过原子操作实现，比手写双重检查锁更安全。

---

### 3. 双重检查锁（Double-Checked Locking）

手动实现懒汉的并发安全，多线程竞争时减少锁的获取次数。

```go
var (
    instance *SingleClass
    mu       sync.Mutex
)

type SingleClass struct{}

func GetInstance() *SingleClass {
    if instance == nil {       // 第一次检查（无锁，快速路径）
        mu.Lock()
        defer mu.Unlock()
        if instance == nil {   // 第二次检查（加锁后再确认）
            instance = &SingleClass{}
        }
    }
    return instance
}
```

✅ 优点：减少锁的竞争，性能比每次都加锁好
❌ 缺点：代码复杂，Go 中直接用 `sync.Once` 即可，无需手写

---

## 三种方式对比

| 方式 | 初始化时机 | 并发安全 | 推荐度 |
|------|----------|---------|--------|
| 饿汉 | 程序启动 | ✅ 天然安全 | ⭐⭐⭐ |
| 懒汉（sync.Once） | 首次调用 | ✅ 安全 | ⭐⭐⭐⭐⭐ |
| 双重检查锁 | 首次调用 | ⚠️ 需谨慎 | ⭐⭐（Go 中不推荐）|

## 常见应用场景

| 场景 | 说明 |
|------|------|
| **配置管理** | 全局配置只需读取一次，所有模块共享同一份 |
| **数据库连接池** | `sql.DB` 本身是连接池，应全局单例复用 |
| **日志器** | `zap.Logger` 等初始化较重，全局共享一个实例 |
| **缓存客户端** | Redis/Memcached 客户端，避免重复建立连接 |
| **消息队列 Producer** | Kafka/RabbitMQ Producer 通常全局单例 |
| **服务注册中心客户端** | Consul/etcd 客户端，单例避免重复注册 |

## 实际案例

### Go 标准库中的单例

```go
// http.DefaultClient 就是一个包级别的单例
var DefaultClient = &Client{}

// sync.Pool 的使用也体现了单例思想
var bufPool = sync.Pool{
    New: func() interface{} { return new(bytes.Buffer) },
}
```

### 业务代码中的单例（配置中心）

```go
type Config struct {
    DBAddr   string
    RedisAddr string
}

var (
    cfg  *Config
    once sync.Once
)

func GetConfig() *Config {
    once.Do(func() {
        cfg = &Config{
            DBAddr:    os.Getenv("DB_ADDR"),
            RedisAddr: os.Getenv("REDIS_ADDR"),
        }
    })
    return cfg
}
```

## 注意事项 / 常见陷阱

> ⚠️ **单元测试难题**：单例持有全局状态，测试间可能互相干扰。可通过依赖注入或测试前重置实例来解决。

> ⚠️ **分布式环境**：单例只保证**进程内唯一**，多节点部署时每个进程都有自己的实例，需配合分布式锁实现跨进程唯一。

> ⚠️ **滥用问题**：不要把单例当全局变量滥用，会导致模块间隐式耦合，优先考虑依赖注入。
