## 一、面向对象（OOP）

### 定位
**编程范式** - 一种组织代码的基本思想

### 核心思想
将现实世界抽象为**对象**，通过**封装、继承、多态**来组织代码。

### 四大特性
- **封装**：隐藏内部实现，只暴露接口
- **继承**：代码复用，建立 is-a 关系
- **多态**：同一接口，不同实现
- **抽象**：提取共性，忽略细节

### 在 Go 中的体现
```go
// 封装：struct + 方法
type User struct {
    name string  // 小写，外部不可见
}
func (u *User) GetName() string { return u.name }

// 多态：接口
type Animal interface { Speak() string }
type Dog struct{}
func (d Dog) Speak() string { return "Woof" }

// 组合优于继承（Go 的哲学）
type Logger struct{}
type Service struct {
    Logger  // 嵌入，而非继承
}
```

## 二、面向切面（AOP）

### 定位
**编程范式** - OOP 的补充，解决横切关注点

### 核心思想
将**横切关注点**（日志、事务、权限）从业务逻辑中分离，统一管理。

### 与 OOP 的关系
- OOP 擅长：纵向的业务逻辑分层（User → UserService → UserRepository）
- AOP 擅长：横向的基础设施关注点（所有 Service 都需要日志、事务）

### 实现方式
见 [[综合#四、代理模式 vs 装饰器模式 vs 面向切面编程（AOP）]]

## 三、IOC / 依赖注入（DI）

### 定位
**设计原则** + **设计模式** - 源自 SOLID 的依赖倒置原则（DIP）

### 核心思想
- **IOC（控制反转）**：对象的创建和依赖关系的管理，由**外部容器**控制，而非对象自己控制
- **DI（依赖注入）**：IOC 的一种实现方式，通过**构造函数/方法参数**注入依赖

### 三种注入方式

#### 1. 构造函数注入（推荐）
```go
type UserService struct {
    repo UserRepository  // 依赖接口，而非具体实现
    logger Logger
}

// 依赖由外部注入
func NewUserService(repo UserRepository, logger Logger) *UserService {
    return &UserService{repo: repo, logger: logger}
}
```

#### 2. 方法注入
```go
func (s *UserService) SetLogger(logger Logger) {
    s.logger = logger
}
```

#### 3. 接口注入（Go 中少见）
```go
type LoggerInjector interface {
    InjectLogger(logger Logger)
}
```

### 为什么需要 DI？

**反例（紧耦合）**：
```go
type UserService struct {
    repo *MySQLUserRepo  // 直接依赖具体实现
}
func NewUserService() *UserService {
    return &UserService{
        repo: &MySQLUserRepo{},  // 自己创建依赖
    }
}
// 问题：无法切换到 PostgreSQL，无法 mock 测试
```

**正例（依赖注入）**：
```go
type UserService struct {
    repo UserRepository  // 依赖接口
}
func NewUserService(repo UserRepository) *UserService {
    return &UserService{repo: repo}  // 外部注入
}
// 优势：可以注入 MySQLRepo / PostgresRepo / MockRepo
```

### Go 中的 DI 框架
- **Wire**（Google 官方，编译期代码生成）
- **Fx**（Uber，运行时依赖注入）
- **手动注入**（推荐，简单项目无需框架）

## 四、设计模式

### 定位
**解决方案模板** - 针对常见问题的可复用解决方案

### 与其他概念的关系
- 设计模式是**实现** OOP/AOP/DI 的**具体手段**
- 例如：
  - **工厂模式** → 实现 IOC（对象创建由工厂控制）
  - **装饰器模式** → 实现 AOP（叠加横切关注点）
  - **策略模式** → 实现 OOP 的多态

## 五、事件驱动

### 定位
**架构风格** - 一种系统间通信和解耦的方式

### 核心思想
组件之间通过**发布/订阅事件**进行通信，而非直接调用。

### 架构图
```
┌─────────┐  发布事件   ┌──────────┐  分发   ┌─────────┐
│ 生产者   │ ────────→ │ 事件总线  │ ────→ │ 消费者1  │
│(Publisher)│           │(Event Bus)│ ────→ │ 消费者2  │
└─────────┘           └──────────┘       └─────────┘
```

### 优势
- **解耦**：生产者不知道消费者是谁
- **扩展性**：新增消费者无需修改生产者
- **异步**：生产者发完事件立即返回

### 实现方式

#### 1. 进程内事件总线
```go
type EventBus struct {
    handlers map[string][]func(interface{})
}

func (bus *EventBus) Subscribe(event string, handler func(interface{})) {
    bus.handlers[event] = append(bus.handlers[event], handler)
}

func (bus *EventBus) Publish(event string, data interface{}) {
    for _, handler := range bus.handlers[event] {
        go handler(data)  // 异步执行
    }
}

// 使用
bus.Subscribe("user.created", func(data interface{}) {
    user := data.(*User)
    sendWelcomeEmail(user)
})
bus.Publish("user.created", newUser)
```

#### 2. 分布式事件驱动（消息队列）
- **Kafka**：高吞吐、持久化
- **RabbitMQ**：灵活路由
- **NATS**：轻量、低延迟

### 与其他概念的关系
- 事件驱动是**架构风格**，可以用 OOP 实现事件处理器
- 可以结合 DI 注入事件总线
- 观察者模式是事件驱动的设计模式体现

## 六、五者关系总结

| 概念         | 层次   | 作用              | 关系               |
| ---------- | ---- | --------------- | ---------------- |
| **面向对象**   | 编程范式 | 组织代码的基本思想       | 基础，其他概念建立在此之上    |
| **AOP**    | 编程范式 | OOP 的补充，处理横切关注点 | 与 OOP 互补         |
| **IOC/DI** | 设计原则 | 控制反转，依赖注入       | 实现 OOP 的依赖倒置原则   |
| **设计模式**   | 解决方案 | 可复用的代码模板        | 实现上述思想的具体手段      |
| **事件驱动**   | 架构风格 | 系统间解耦通信         | 更高层次，可用 OOP+模式实现 |

## 七、实际项目中的综合应用

### 场景：电商订单系统

```go
// ===== 1. 面向对象：定义领域模型 =====
type Order struct {
    ID     int64
    UserID int64
    Amount float64
}

// ===== 2. 依赖注入：Service 依赖接口 =====
type OrderService struct {
    repo   OrderRepository  // 接口
    logger Logger           // 接口
    eventBus EventBus       // 接口
}

func NewOrderService(repo OrderRepository, logger Logger, bus EventBus) *OrderService {
    return &OrderService{repo: repo, logger: logger, eventBus: bus}
}

// ===== 3. AOP：用装饰器加日志 =====
type LoggingOrderService struct {
    next OrderService
}
func (s *LoggingOrderService) CreateOrder(order *Order) error {
    log.Println("Creating order...")
    err := s.next.CreateOrder(order)
    log.Println("Order created, err=", err)
    return err
}

// ===== 4. 事件驱动：发布订单创建事件 =====
func (s *OrderService) CreateOrder(order *Order) error {
    if err := s.repo.Save(order); err != nil {
        return err
    }
    // 发布事件，解耦后续流程
    s.eventBus.Publish("order.created", order)
    return nil
}

// ===== 5. 设计模式：观察者模式处理事件 =====
// 库存服务订阅事件
eventBus.Subscribe("order.created", func(data interface{}) {
    order := data.(*Order)
    inventoryService.DeductStock(order)
})

// 通知服务订阅事件
eventBus.Subscribe("order.created", func(data interface{}) {
    order := data.(*Order)
    notificationService.SendOrderConfirmation(order)
})
```

## 八、如何选择？

| 我想… | 用什么 |
|-------|--------|
| 组织代码结构，定义领域模型 | **面向对象** |
| 统一处理日志、事务、权限 | **AOP**（装饰器/中间件） |
| 解耦依赖，方便测试和替换实现 | **IOC/DI** |
| 解决特定设计问题（创建对象、扩展功能） | **设计模式** |
| 系统间异步解耦通信 | **事件驱动** |
