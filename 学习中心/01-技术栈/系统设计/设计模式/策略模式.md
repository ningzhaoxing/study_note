# 策略模式

> **核心思想**：定义一系列算法，将每种算法封装到独立的类中，使它们可以**互相替换**。算法的变化不影响使用算法的客户端。

## 解决的问题

当一个行为有多种实现方式（如：支付方式、排序算法、促销规则），如果用 `if/else` 或 `switch` 堆在一起：
- 新增一种方式 → 必须修改原有代码（违反开闭原则）
- 逻辑越来越复杂 → 难以维护和测试

策略模式把每种算法独立封装，新增只需加新类。

## UML 结构

```plantuml
interface Strategy {
    Execute(data) result
}

class ConcreteStrategyA {
    Execute(data) result
}

class ConcreteStrategyB {
    Execute(data) result
}

ConcreteStrategyA --u|> Strategy
ConcreteStrategyB --u|> Strategy

class Context {
    - strategy: Strategy
    + SetStrategy(s Strategy)
    + ExecuteStrategy(data) result
}

Context o--> Strategy
```

## Go 代码示例

### 示例一：商场促销（原始案例扩展）

```go
// ICashAcceptor 策略接口
type ICashAcceptor interface {
    AcceptCash(money float64) float64
}

// --- 具体策略 ---

// 普通收费
type NormalAcceptCash struct{}
func (n *NormalAcceptCash) AcceptCash(money float64) float64 { return money }

// 打折促销
type RebateAcceptCash struct{ discount float64 }
func (r *RebateAcceptCash) AcceptCash(money float64) float64 {
    return money * (1 - r.discount)
}

// 满减返利
type ReturnAcceptCash struct{ limit, returnCash float64 }
func (r *ReturnAcceptCash) AcceptCash(money float64) float64 {
    if money >= r.limit {
        return money - r.returnCash
    }
    return money
}

// --- Context ---

type CashContext struct {
    strategy ICashAcceptor
}

func NewCashContext(strategy ICashAcceptor) *CashContext {
    return &CashContext{strategy: strategy}
}

func (c *CashContext) GetResult(money float64) float64 {
    return c.strategy.AcceptCash(money)
}

// --- 客户端 ---
func main() {
    ctx := NewCashContext(&RebateAcceptCash{discount: 0.2}) // 8折
    fmt.Println(ctx.GetResult(100)) // 80

    ctx.strategy = &ReturnAcceptCash{limit: 300, returnCash: 100} // 满300减100
    fmt.Println(ctx.GetResult(350)) // 250
}
```

### 示例二：文件导出（多格式输出）

```go
type Exporter interface {
    Export(data []byte) error
}

type JSONExporter struct{}
func (e *JSONExporter) Export(data []byte) error {
    fmt.Println("Exporting as JSON:", string(data))
    return nil
}

type CSVExporter struct{}
func (e *CSVExporter) Export(data []byte) error {
    fmt.Println("Exporting as CSV:", string(data))
    return nil
}

type ExcelExporter struct{}
func (e *ExcelExporter) Export(data []byte) error {
    fmt.Println("Exporting as Excel:", string(data))
    return nil
}

// 注册策略，运行时动态选择
var exporterRegistry = map[string]Exporter{
    "json":  &JSONExporter{},
    "csv":   &CSVExporter{},
    "excel": &ExcelExporter{},
}

func ExportData(format string, data []byte) error {
    exporter, ok := exporterRegistry[format]
    if !ok {
        return fmt.Errorf("unsupported format: %s", format)
    }
    return exporter.Export(data)
}
```

## 优缺点

| | 说明 |
|--|------|
| ✅ 开闭原则 | 新增策略只需新增类，不修改 Context |
| ✅ 消除条件语句 | 替代大量 `if/else switch`，代码更清晰 |
| ✅ 可单独测试 | 每个策略可以独立单元测试 |
| ✅ 运行时切换 | 可以动态替换算法 |
| ❌ 类数量增多 | 每个策略都是一个类，策略多时类爆炸 |
| ❌ 客户端需感知策略 | 客户端必须了解各策略的区别才能选择 |

## 常见应用场景

| 场景 | 说明 |
|------|------|
| **支付方式** | 支付宝、微信支付、银行卡各自一套策略 |
| **排序算法** | 快排、归并、堆排，根据数据量动态选择 |
| **促销规则** | 打折、满减、积分兑换，策略可随活动配置 |
| **数据校验** | 不同业务场景的表单校验规则 |
| **文件导出** | JSON/CSV/Excel/PDF 多格式按需选择 |
| **限流算法** | 令牌桶、漏桶、滑动窗口，按场景切换 |
| **路由策略** | 负载均衡中的轮询、随机、加权轮询、一致性 Hash |
| **压缩算法** | gzip/snappy/zstd，按数据类型选择最优 |

## 实际框架中的体现

```
Go net/http    → ServeMux 路由匹配策略可替换
gRPC           → 负载均衡策略（round_robin / pick_first）可配置
database/sql   → Driver 注册机制本质是策略注册
sort.Slice     → 比较函数 less 就是策略注入
```

## 与模板方法模式的区别

见 [[综合]] 中的详细对比。

简要区别：

| | 策略模式 | 模板方法模式 |
|--|---------|-----------|
| **实现方式** | 组合（接口注入） | 继承（父类定义框架） |
| **变化粒度** | 整个算法可替换 | 只有某些步骤可替换 |
| **运行时切换** | ✅ 可以 | ❌ 编译时确定 |
| **Go 中的体现** | 接口 + 依赖注入 | 接口嵌套模拟继承 |
