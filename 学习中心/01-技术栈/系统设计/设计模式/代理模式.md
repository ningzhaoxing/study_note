# 代理模式

> **核心思想**：为目标对象提供一个**代理对象**，由代理控制对目标的访问，在不修改目标的前提下，增加前置/后置处理逻辑。

## UML 结构

```plantuml
interface Subject {
    Request()
}

class RealSubject {
    Request()
}

class Proxy {
    - realSubject: RealSubject
    beforeRequest()
    Request()
    afterRequest()
}

RealSubject --u|> Subject
Proxy --u|> Subject
Proxy o--> RealSubject

class Client {}
Client --> Subject
```

**关键点**：代理与真实对象实现**同一接口**，客户端感知不到代理的存在，对代理的调用与直接调用真实对象的接口完全一致。

---

## 四种代理类型

### 1. 远程代理（Remote Proxy）

为远程对象提供本地代理，隐藏网络通信细节。

**场景**：gRPC / HTTP 客户端 SDK

```go
// 客户端只需调用本地接口，网络通信由代理封装
type UserServiceProxy struct {
    client pb.UserServiceClient
}

func (p *UserServiceProxy) GetUser(id int64) (*User, error) {
    // 底层是 gRPC 调用，对上层透明
    resp, err := p.client.GetUser(context.Background(), &pb.GetUserReq{Id: id})
    if err != nil {
        return nil, err
    }
    return &User{Name: resp.Name}, nil
}
```

**实际案例**：
- gRPC 生成的客户端 Stub
- Dubbo/Spring Cloud 的服务调用代理
- AWS SDK 中封装 HTTP 请求的各种 Client

---

### 2. 虚拟代理（Virtual Proxy）

延迟创建开销大的对象，首次真正使用时才初始化（懒加载）。

**场景**：大图片/视频的懒加载、ORM 延迟加载关联对象

```go
type Image interface {
    Display()
}

type RealImage struct {
    filename string
}

func (r *RealImage) Display() {
    fmt.Printf("Displaying %s\n", r.filename)
}

// 虚拟代理：延迟加载
type ImageProxy struct {
    filename  string
    realImage *RealImage
}

func (p *ImageProxy) Display() {
    if p.realImage == nil {
        // 首次调用才真正加载
        fmt.Printf("Loading %s from disk...\n", p.filename)
        p.realImage = &RealImage{filename: p.filename}
    }
    p.realImage.Display()
}
```

**实际案例**：
- GORM 的关联关系懒加载（`Preload` vs 延迟查询）
- Elasticsearch 的滚动查询
- 图片懒加载（前端）/ 缩略图代替原图

---

### 3. 安全代理（Protection Proxy）

在访问真实对象前，增加**权限校验**或**身份认证**，对外不暴露真实主题。

**场景**：接口鉴权、RBAC 权限控制

```go
type FileService interface {
    ReadFile(path string) ([]byte, error)
    WriteFile(path string, data []byte) error
}

type RealFileService struct{}

func (r *RealFileService) ReadFile(path string) ([]byte, error) {
    return os.ReadFile(path)
}
func (r *RealFileService) WriteFile(path string, data []byte) error {
    return os.WriteFile(path, data, 0644)
}

// 安全代理：增加权限检查
type SecureFileService struct {
    real FileService
    role string
}

func (s *SecureFileService) ReadFile(path string) ([]byte, error) {
    // 所有角色可读
    return s.real.ReadFile(path)
}

func (s *SecureFileService) WriteFile(path string, data []byte) error {
    if s.role != "admin" {
        return errors.New("permission denied: write requires admin role")
    }
    return s.real.WriteFile(path, data)
}
```

**实际案例**：
- Go 中间件（Gin/Echo middleware）本质就是安全代理
- Spring Security 的方法级权限注解 `@PreAuthorize`
- API Gateway 的鉴权层

---

### 4. 缓存代理（Cache Proxy）

在代理层缓存结果，避免重复访问昂贵资源。

**场景**：接口缓存、DB 查询缓存

```go
type UserRepository interface {
    FindByID(id int64) (*User, error)
}

type DBUserRepository struct{}

func (r *DBUserRepository) FindByID(id int64) (*User, error) {
    // 真实数据库查询
    return queryFromDB(id)
}

// 缓存代理
type CachedUserRepository struct {
    real  UserRepository
    cache map[int64]*User
    mu    sync.RWMutex
}

func (c *CachedUserRepository) FindByID(id int64) (*User, error) {
    c.mu.RLock()
    if user, ok := c.cache[id]; ok {
        c.mu.RUnlock()
        return user, nil // 命中缓存
    }
    c.mu.RUnlock()

    user, err := c.real.FindByID(id) // 穿透到真实 DB
    if err != nil {
        return nil, err
    }
    c.mu.Lock()
    c.cache[id] = user
    c.mu.Unlock()
    return user, nil
}
```

**实际案例**：
- Redis 作为 MySQL 的缓存代理
- CDN 作为源站的缓存代理
- `groupcache` / `bigcache` 等本地缓存库

## 常见应用场景汇总

| 代理类型 | 核心能力 | 典型技术 |
|---------|---------|---------|
| 远程代理 | 封装网络通信 | gRPC Stub、HTTP Client SDK |
| 虚拟代理 | 延迟初始化 | ORM 懒加载、图片懒加载 |
| 安全代理 | 权限/认证控制 | Middleware、API Gateway |
| 缓存代理 | 缓存加速 | Redis、CDN、本地缓存 |

## 与装饰器模式的区别

见 [[综合]] 中的详细对比。

简要区别：

| | 代理模式 | 装饰器模式 |
|--|---------|----------|
| **目的** | 控制访问（可以拒绝、延迟、缓存） | 增强功能（只是叠加能力） |
| **对象关系** | 代理自己创建/持有真实对象 | 装饰器由外部传入被装饰对象 |
| **客户端感知** | 客户端通常不知道有代理 | 客户端通常知道在做装饰 |
| **叠加层数** | 通常一层 | 可以多层叠加 |
