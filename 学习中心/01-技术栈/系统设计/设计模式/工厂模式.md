# 工厂模式

> **核心思想**：将对象的**创建**与**使用**分离，由工厂负责创建，客户端只面向接口编程。

工厂模式有三个变体，复杂度和灵活性递增：

```
简单工厂 → 方法工厂 → 抽象工厂
  简单       灵活        强大
```

---

# 简单工厂模式（Simple Factory）

## 概念

由**一个**工厂类，根据传入参数决定创建哪种产品。

```plantuml
interface Operator {
    GetResult() string
}

class Add {}
class Sub {}
class Mul {}

Add --u|> Operator
Sub --u|> Operator
Mul --u|> Operator

class OperatorFactory {
    CreateOperator(t string) Operator
}

OperatorFactory ..> Operator
```

✅ 优点：集中管理创建逻辑，客户端与具体产品解耦
❌ 缺点：**违反开闭原则**——每增加一种产品，都要修改工厂类的 `switch/if` 分支

## Go 代码示例

```go
type Operator interface {
    GetResult(a, b float64) float64
}

type Add struct{}
func (a *Add) GetResult(x, y float64) float64 { return x + y }

type Sub struct{}
func (s *Sub) GetResult(x, y float64) float64 { return x - y }

type Mul struct{}
func (m *Mul) GetResult(x, y float64) float64 { return x * y }

// 工厂类
func CreateOperator(t string) Operator {
    switch t {
    case "+":
        return &Add{}
    case "-":
        return &Sub{}
    case "*":
        return &Mul{}
    default:
        return nil
    }
}

// 客户端使用
func main() {
    op := CreateOperator("+")
    fmt.Println(op.GetResult(3, 4)) // 7
}
```

## 适用场景

1. 产品种类**固定且数量少**（不会频繁新增）
2. 客户端完全不关心创建细节
3. 快速原型，追求简洁

# 方法工厂模式（Factory Method）

## 概念

定义一个创建产品的**接口**，每个产品对应一个**独立工厂类**来实现该接口。新增产品只需新增一对（产品类 + 工厂类），**不修改已有代码**。

```plantuml
interface Operator {
    GetResult() string
}

class Add {}
class Sub {}
class Mul {}

Add --u|> Operator
Sub --u|> Operator
Mul --u|> Operator

interface IFactory {
    CreateOperator() Operator
}

class AddFactory {}
class SubFactory {}
class MulFactory {}

AddFactory --u|> IFactory
SubFactory --u|> IFactory
MulFactory --u|> IFactory

AddFactory ..> Add
SubFactory ..> Sub
MulFactory ..> Mul
```

✅ 优点：符合**开闭原则**，新增产品无需修改已有工厂
❌ 缺点：客户端需要知道具体工厂类，类的数量随产品增加而膨胀

## Go 代码示例

```go
type IFactory interface {
    CreateOperator() Operator
}

type AddFactory struct{}
func (f *AddFactory) CreateOperator() Operator { return &Add{} }

type SubFactory struct{}
func (f *SubFactory) CreateOperator() Operator { return &Sub{} }

// 客户端使用——需要自行选择工厂
func main() {
    var factory IFactory = &AddFactory{}
    op := factory.CreateOperator()
    fmt.Println(op.GetResult(3, 4)) // 7
}
```

## 适用场景

1. 产品种类会**频繁扩展**
2. 需要在运行时**动态选择**工厂
3. 框架/SDK 中，允许使用者自定义创建逻辑

# 抽象工厂模式（Abstract Factory）

## 概念

创建一组**相关产品族**的接口。适合有多个维度需要组合的场景（如：不同 DB + 不同缓存的基础设施组合）。

```plantuml
interface AbstractFactory {
    CreateDB() IDB
    CreateCache() ICache
}

class MySQLFactory {}
class PostgresFactory {}

MySQLFactory --u|> AbstractFactory
PostgresFactory --u|> AbstractFactory

interface IDB {}
interface ICache {}

class MySQL {}
class Redis {}
class Postgres {}
class Memcached {}

MySQL --u|> IDB
Postgres --u|> IDB
Redis --u|> ICache
Memcached --u|> ICache
```

✅ 优点：保证同一产品族的一致性，切换整套实现很方便
❌ 缺点：新增产品族中的新产品类型时，需要修改所有工厂接口，扩展困难

## 适用场景

1. 系统需要**跨平台/多环境**（开发/测试/生产使用不同基础设施）
2. 产品族之间有约束，需要保证**同族搭配**

---

# 如何选择

> 核心选择标准：根据产品的**变化频率**和项目的**扩展需求**来决定。

| 模式 | 复杂度 | 产品数量 | 是否频繁扩展 | 推荐场景 |
|------|--------|---------|------------|--------|
| 简单工厂 | 低 | 少（<5） | 否 | 快速实现、种类固定 |
| 方法工厂 | 中 | 中/多 | 是 | 需要扩展，符合开闭原则 |
| 抽象工厂 | 高 | 多族 | 否（族内） | 多维度产品族切换 |

---

## 实际应用案例

| 场景 | 模式 | 说明 |
|------|------|------|
| `database/sql` 中的 `sql.Open` | 简单工厂 | 根据 driver 名称创建不同 DB 连接 |
| `log.New` / `zap.New` | 方法工厂 | 不同 logger 工厂 |
| 支付渠道（支付宝/微信/银行卡） | 方法工厂 | 每种支付方式独立工厂 |
| 多云基础设施（AWS/GCP/阿里云） | 抽象工厂 | 一套接口，按云厂商切换整套实现 |
| ORM 中的 Dialect | 方法工厂 | MySQL/Postgres/SQLite 各自方言 |
