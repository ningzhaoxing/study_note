# 设计模式综合对比

本文汇总几个容易混淆的模式的核心区别，以及选型指南。

---

## 一、代理模式 vs 装饰器模式

### 相同点

- 都通过**实现相同接口 + 组合原有对象**的方式工作
- 代码结构高度相似，从 UML 上看几乎一样

### 核心区别：**意图不同**

| 维度        | 代理模式                       | 装饰器模式             |
| --------- | -------------------------- | ----------------- |
| **意图**    | **控制访问**（可以拒绝、延迟、缓存、远程调用）  | **增强功能**（叠加新职责）   |
| **对象来源**  | 代理**自己创建/持有**真实对象，客户端不直接接触 | 装饰器由**外部传入**被装饰对象 |
| **客户端感知** | 客户端通常**不知道**有代理存在          | 客户端通常**知道**在做功能组合 |
| **叠加性**   | 通常只有**一层**代理               | 可以**多层嵌套**叠加      |
| **生命周期**  | 代理管理真实对象的生命周期              | 被装饰对象在装饰器外部存在     |

### 判断口诀

> - 想**限制/控制**访问 → **代理**（鉴权、缓存、远程调用、懒加载）
> - 想**叠加/扩展**能力 → **装饰器**（日志 + 限流 + 监控）

### 代码对比

```go
// ===== 代理模式 =====
// 代理自己持有并管理真实对象
type ServiceProxy struct {
    real *RealService // 代理内部创建或持有
}
func NewServiceProxy() *ServiceProxy {
    return &ServiceProxy{real: &RealService{}} // 代理决定何时创建真实对象
}
func (p *ServiceProxy) Do() {
    if !p.checkAuth() { return } // 控制访问
    p.real.Do()
}

// ===== 装饰器模式 =====
// 被装饰对象由外部注入
type LogDecorator struct {
    next Service // 由调用方传入，装饰器不关心具体类型
}
func NewLogDecorator(next Service) *LogDecorator {
    return &LogDecorator{next: next}
}
func (d *LogDecorator) Do() {
    log.Println("before")
    d.next.Do()  // 一定会调用，只是增强
    log.Println("after")
}
```

## 二、策略模式 vs 模板方法模式

### 相同点

- 都用于处理**算法/行为的变化**
- 都将"不变的部分"与"可变的部分"分离

### 核心区别：**实现方式与变化粒度**

| 维度 | 策略模式 | 模板方法模式 |
|------|---------|-----------|
| **实现方式** | **组合**（接口注入） | **继承**（父类定义框架） |
| **变化粒度** | **整个算法**都可替换 | 只有**某几个步骤**可替换，框架不变 |
| **运行时切换** | ✅ 可以动态切换 | ❌ 编译时确定（子类固定） |
| **代码复用** | 各策略独立，无复用 | 公共步骤在父类复用 |
| **调用方向** | 客户端选择策略注入 | 父类调用子类的方法（好莱坞原则） |
| **Go 推荐** | 接口 + 依赖注入 | 接口嵌套模拟继承 |

### 判断口诀

> - 整个算法要换 → **策略**（支付方式、排序算法、限流策略）
> - 框架固定，只有部分步骤要换 → **模板方法**（报表生成、游戏流程、数据导入）

### 代码对比

```go
// ===== 策略模式 =====
// 整个算法可以被替换
type Sorter interface {
    Sort(data []int) []int
}

type Context struct {
    sorter Sorter  // 可以在运行时换任何排序算法
}
func (c *Context) SetSorter(s Sorter) { c.sorter = s } // 运行时切换
func (c *Context) Execute(data []int) []int {
    return c.sorter.Sort(data)
}

// ===== 模板方法模式 =====
// 框架固定，子类只填充步骤
type DataPipeline interface {
    Extract() []byte   // 步骤可变
    Transform([]byte) []byte // 步骤可变
    Load([]byte)        // 步骤可变
}

type BasePipeline struct{ impl DataPipeline }
// 框架不变，顺序固定
func (p *BasePipeline) Run() {
    data := p.impl.Extract()          // 固定步骤1
    transformed := p.impl.Transform(data) // 固定步骤2
    p.impl.Load(transformed)           // 固定步骤3
}
```

## 三、工厂模式 vs 原型模式

两者都是**创建型模式**，目的都是创建对象：

| 维度 | 工厂模式 | 原型模式 |
|------|---------|---------|
| **创建方式** | `new` + 构造逻辑 | `clone` 已有对象 |
| **适用场景** | 对象创建逻辑复杂、需要隐藏实现 | 对象初始化代价高、需要大量相似对象 |
| **依赖关系** | 需要知道如何构造对象 | 只需要有一个原型对象 |
| **扩展方式** | 新增工厂类 | 注册新原型 |

## 四、代理模式 vs 装饰器模式 vs 面向切面编程（AOP）

### 层次关系

```
面向切面编程（AOP）
    ↓ 编程范式/思想
    ├─ 代理模式 ← 实现手段之一
    └─ 装饰器模式 ← 实现手段之一
```

**核心理解**：
- **AOP** 是一种**编程思想**，用于解决横切关注点（Cross-cutting Concerns）的分离问题
- **代理模式**和**装饰器模式**是实现 AOP 的**具体技术手段**

---

### 什么是 AOP（面向切面编程）

**定义**：将那些与业务逻辑无关、但又在多处重复出现的功能（如日志、事务、权限、监控）抽取出来，统一管理。

**横切关注点**（Cross-cutting Concerns）：
- 日志记录
- 性能监控
- 事务管理
- 权限校验
- 异常处理
- 缓存控制

这些功能**横跨多个模块**，如果散落在业务代码中，会导致：
- 代码重复
- 业务逻辑与基础设施代码耦合
- 难以统一修改

---

### AOP 的实现方式

| 实现方式 | 说明 | 典型技术 |
|---------|------|---------|
| **静态织入** | 编译期/类加载期修改字节码 | AspectJ、编译器插件 |
| **动态代理** | 运行时生成代理对象 | Java 动态代理、CGLIB、Go 装饰器 |
| **中间件/拦截器** | 框架层面的拦截链 | Gin/Echo middleware、gRPC interceptor |

在 Go 中，由于没有反射生成代理的便利性，**AOP 主要通过装饰器模式 + 中间件实现**。

---

### 三者关系对比

| 维度 | AOP | 代理模式 | 装饰器模式 |
|------|-----|---------|----------|
| **层次** | 编程范式/思想 | 设计模式（实现手段） | 设计模式（实现手段） |
| **目的** | 分离横切关注点 | 控制访问 | 叠加功能 |
| **关注点** | 业务逻辑 vs 基础设施 | 访问控制 | 功能增强 |
| **实现** | 可用代理/装饰器/编译器 | 代理对象 | 装饰器链 |
| **典型场景** | 日志、事务、监控 | 鉴权、缓存、远程调用 | HTTP 中间件、IO 流 |

### 代码示例：三者在 Go 中的体现

#### 场景：给 UserService 加日志和权限校验

```go
// ===== 业务接口 =====
type UserService interface {
    GetUser(id int64) (*User, error)
    DeleteUser(id int64) error
}

type RealUserService struct{ db *sql.DB }
func (s *RealUserService) GetUser(id int64) (*User, error)   { /* 查 DB */ }
func (s *RealUserService) DeleteUser(id int64) error         { /* 删除 */ }

// ===== AOP 思想：横切关注点分离 =====
// 日志、权限不应该写在 RealUserService 里，应该独立管理

// ===== 实现方式1：装饰器模式（推荐） =====
// 日志装饰器
type LoggingDecorator struct{ next UserService }
func (d *LoggingDecorator) GetUser(id int64) (*User, error) {
    log.Printf("[LOG] GetUser(%d)", id)
    user, err := d.next.GetUser(id)
    log.Printf("[LOG] GetUser result: err=%v", err)
    return user, err
}
func (d *LoggingDecorator) DeleteUser(id int64) error {
    log.Printf("[LOG] DeleteUser(%d)", id)
    return d.next.DeleteUser(id)
}

// 权限装饰器
type AuthDecorator struct {
    next UserService
    role string
}
func (d *AuthDecorator) GetUser(id int64) (*User, error) {
    // 读操作：所有角色可访问
    return d.next.GetUser(id)
}
func (d *AuthDecorator) DeleteUser(id int64) error {
    if d.role != "admin" {
        return errors.New("permission denied")
    }
    return d.next.DeleteUser(id)
}

// 组装（AOP 织入）
func NewUserService(db *sql.DB, role string) UserService {
    var svc UserService = &RealUserService{db: db}
    svc = &LoggingDecorator{next: svc}      // 织入日志切面
    svc = &AuthDecorator{next: svc, role: role} // 织入权限切面
    return svc
}

// ===== 实现方式2：代理模式 =====
// 如果权限检查需要"控制是否创建真实对象"，用代理更合适
type UserServiceProxy struct {
    real UserService
    role string
}
func NewUserServiceProxy(db *sql.DB, role string) *UserServiceProxy {
    return &UserServiceProxy{
        real: &RealUserService{db: db}, // 代理内部创建真实对象
        role: role,
    }
}
func (p *UserServiceProxy) DeleteUser(id int64) error {
    if p.role != "admin" {
        return errors.New("permission denied") // 直接拒绝，不调用真实对象
    }
    return p.real.DeleteUser(id)
}
```

---

### 判断口诀

> **我在做什么？**
> - 想分离**横切关注点**（日志/事务/监控） → 用 **AOP 思想**
> - 具体实现时，需要**叠加多个功能** → 用 **装饰器模式**
> - 具体实现时，需要**控制访问** → 用 **代理模式**

---

### Go 中的 AOP 最佳实践

| 场景 | 推荐方式 |
|------|---------|
| **HTTP 请求** | Gin/Echo middleware（装饰器链） |
| **gRPC 调用** | Interceptor（装饰器链） |
| **数据库操作** | Repository 装饰器（日志、缓存、事务） |
| **方法级 AOP** | 手写装饰器（Go 无注解，无法自动织入） |
| **编译期织入** | 使用 `go generate` + 代码生成工具 |

---

### 实际框架中的体现

```
Spring AOP     → 基于动态代理（JDK Proxy / CGLIB）
AspectJ        → 编译期字节码织入
Gin Middleware → 装饰器模式（HTTP 层面的 AOP）
gRPC Interceptor → 装饰器模式（RPC 层面的 AOP）
GORM Hooks     → 回调机制（ORM 层面的 AOP）
```

## 五、所有模式速查表

### 按分类

| 分类 | 模式 | 解决的核心问题 |
|------|------|-------------|
| **创建型** | [[单例模式]] | 全局唯一实例 |
| **创建型** | [[工厂模式]] | 封装对象创建，解耦使用与创建 |
| **创建型** | [[原型模式]] | 通过克隆快速创建相似对象 |
| **结构型** | [[代理模式]] | 控制对对象的访问 |
| **结构型** | [[装饰器模式]] | 动态叠加对象的职责 |
| **行为型** | [[策略模式]] | 封装一组可互换的算法 |
| **行为型** | [[模板方法模式]] | 定义算法骨架，子类填充步骤 |

### 按使用场景

| 我想…                      | 用什么模式      |
| ------------------------ | ---------- |
| 确保某个对象全局只有一份             | [[单例模式]]   |
| 根据条件创建不同类型的对象            | [[工厂模式]]   |
| 快速复制一个复杂对象               | [[原型模式]]   |
| 在不修改对象的前提下控制访问（鉴权/缓存/远程） | [[代理模式]]   |
| 动态给对象叠加功能（日志/限流/监控）      | [[装饰器模式]]  |
| 运行时动态切换算法/行为             | [[策略模式]]   |
| 流程固定但步骤实现不同（报表/导入/游戏）    | [[模板方法模式]] |

## 六、面向对象设计原则回顾

| 原则 | 说明 | 对应模式 |
|------|------|---------|
| **单一职责** (SRP) | 每个类只负责一件事 | 装饰器（功能分离叠加） |
| **开闭原则** (OCP) | 对扩展开放，对修改关闭 | 策略、工厂方法 |
| **里氏替换** (LSP) | 子类可以替换父类 | 模板方法 |
| **接口隔离** (ISP) | 依赖最小接口 | 策略（面向接口）|
| **依赖倒置** (DIP) | 依赖抽象，而非具体实现 | 所有模式 |

> 💡 不要为了用设计模式而用设计模式。先有问题，再找模式。**过度设计**比不用模式危害更大。
