# 模板方法模式

> **核心思想**：在父类（抽象类）中定义算法的**骨架（框架）**，将某些步骤的**具体实现**延迟到子类。子类可以重写特定步骤，但不改变算法整体结构。

## 解决的问题

有一类流程，整体步骤相同（如：数据导出 = 连接数据源 → 查询 → 格式化 → 输出），但各步骤的具体实现因场景而异。如果每种场景都完整实现一遍，会导致**大量重复代码**。

模板方法模式将**不变的部分**提到父类，**可变的部分**由子类实现。

---

## UML 结构

```plantuml
abstract AbstractTemplate {
    + TemplateMethod()
    # PrimitiveOperation1()
    # PrimitiveOperation2()
}

class ConcreteClassA {
    # PrimitiveOperation1()
    # PrimitiveOperation2()
}

class ConcreteClassB {
    # PrimitiveOperation1()
    # PrimitiveOperation2()
}

ConcreteClassA --u|> AbstractTemplate
ConcreteClassB --u|> AbstractTemplate
```

- `TemplateMethod()`：**模板方法**，定义算法骨架，通常 `final`（不可重写）
- `PrimitiveOperation()`：**原语操作**，由子类实现的可变步骤

## Go 代码示例

> Go 没有抽象类，通过**接口嵌入 struct** 模拟模板方法模式。

### 示例一：游戏流程（原始案例完善）

```go
// 定义步骤接口
type IGame interface {
    Init()
    Start()
    End()
}

// 模板 struct：定义算法骨架
type Game struct {
    IGame // 嵌入接口，子类通过赋值实现多态
}

// TemplateMethod：定义不变的流程框架
func (g *Game) Play() {
    g.Init()   // 步骤1：初始化
    g.Start()  // 步骤2：开始
    g.End()    // 步骤3：结束
}

// --- 具体子类 ---

type Football struct {
    Game
}

func NewFootball() *Football {
    f := &Football{}
    f.IGame = f // 关键：将自身注入模板
    return f
}

func (f *Football) Init()  { fmt.Println("Football: 布置场地") }
func (f *Football) Start() { fmt.Println("Football: 开球！") }
func (f *Football) End()   { fmt.Println("Football: 颁奖结束") }

type Basketball struct {
    Game
}

func NewBasketball() *Basketball {
    b := &Basketball{}
    b.IGame = b
    return b
}

func (b *Basketball) Init()  { fmt.Println("Basketball: 检查场地和器材") }
func (b *Basketball) Start() { fmt.Println("Basketball: 跳球开始！") }
func (b *Basketball) End()   { fmt.Println("Basketball: 统计数据，离场") }

// 客户端
func main() {
    games := []*Game{
        NewFootball().Game,
        NewBasketball().Game,
    }
    for _, g := range games {
        g.Play() // 相同的调用方式，不同的行为
        fmt.Println("---")
    }
}
```

### 示例二：数据报表生成（更贴近业务）

```go
type ReportGenerator interface {
    FetchData() []map[string]interface{}
    FormatData(data []map[string]interface{}) string
    Output(result string)
}

// 模板骨架
type BaseReport struct {
    impl ReportGenerator
}

// 模板方法：流程不变
func (r *BaseReport) Generate() {
    data := r.impl.FetchData()        // 步骤1：拉数据（子类实现）
    result := r.impl.FormatData(data) // 步骤2：格式化（子类实现）
    r.impl.Output(result)             // 步骤3：输出（子类实现）
}

// 日报
type DailyReport struct{ BaseReport }
func NewDailyReport() *DailyReport {
    r := &DailyReport{}
    r.impl = r
    return r
}
func (r *DailyReport) FetchData() []map[string]interface{} {
    return fetchFromDB("SELECT * FROM orders WHERE date = today()")
}
func (r *DailyReport) FormatData(data []map[string]interface{}) string {
    return formatAsTable(data)
}
func (r *DailyReport) Output(result string) { sendEmail(result) }

// Excel 报表
type ExcelReport struct{ BaseReport }
func NewExcelReport() *ExcelReport {
    r := &ExcelReport{}
    r.impl = r
    return r
}
func (r *ExcelReport) FetchData() []map[string]interface{} {
    return fetchFromDB("SELECT * FROM orders WHERE month = this_month()")
}
func (r *ExcelReport) FormatData(data []map[string]interface{}) string {
    return formatAsExcel(data)
}
func (r *ExcelReport) Output(result string) { saveToFile("report.xlsx", result) }
```

## 优缺点

| | 说明 |
|--|------|
| ✅ 复用公共逻辑 | 不变的步骤只写一次，减少重复代码 |
| ✅ 扩展灵活 | 新场景只需新增子类，不修改框架 |
| ✅ 强制规范 | 子类必须实现规定的步骤，保证流程一致性 |
| ❌ 理解成本 | 需要同时看父类和子类才能理解完整流程 |
| ❌ Go 中实现稍繁琐 | 没有抽象类，需要接口嵌入技巧 |
| ❌ 违反里氏替换 | 子类行为改变可能破坏父类预期 |

---

## 常见应用场景

| 场景 | 说明 |
|------|------|
| **数据导入导出** | 连接 → 读取 → 转换 → 写入，步骤固定，来源/目标各异 |
| **报表生成** | 取数 → 计算 → 格式化 → 输出，各类报表只差具体实现 |
| **测试框架** | `setUp → test → tearDown` 是典型模板方法 |
| **HTTP 请求处理** | 框架定义生命周期，开发者填充 Handler |
| **游戏引擎** | `Init → Update → Render → Destroy` 游戏循环 |
| **ORM 操作** | `BeforeCreate → Create → AfterCreate` 钩子机制 |
| **CI/CD 流水线** | 拉代码 → 构建 → 测试 → 部署，步骤固定，各环境实现不同 |

## 与策略模式的区别

见 [[综合]] 中的详细对比。

| | 模板方法模式 | 策略模式 |
|--|------------|---------|
| **实现方式** | 继承（父类 + 子类） | 组合（接口注入） |
| **变化粒度** | 某几个步骤可变，框架不变 | 整个算法都可替换 |
| **运行时切换** | ❌ 编译时确定子类 | ✅ 运行时可换策略 |
| **代码复用** | 公共步骤在父类中复用 | 无复用，各策略独立 |
| **Go 推荐** | 接口嵌套模拟 | 直接用接口 + 依赖注入 |
