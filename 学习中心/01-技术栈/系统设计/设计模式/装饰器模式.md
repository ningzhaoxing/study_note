# 装饰器模式

> **核心思想**：在**不修改原有类**的情况下，通过**包装**的方式动态地为对象添加额外职责。相比继承，装饰器更灵活，可以**任意组合叠加**。

## 解决的问题

如果想给对象加功能：
- **继承方式**：子类数量爆炸（A + 日志、A + 缓存、A + 日志 + 缓存 = 多个子类）
- **装饰器方式**：独立的装饰类，按需组合，不修改原始类

---

## UML 结构

```plantuml
interface Component {
    Operation()
}

class ConcreteComponent {
    Operation()
}

abstract Decorator {
    - component: Component
    Operation()
}

class ConcreteDecoratorA {
    Operation()
    extraBehaviorA()
}

class ConcreteDecoratorB {
    Operation()
    extraBehaviorB()
}

ConcreteComponent --u|> Component
Decorator --u|> Component
Decorator o--> Component
ConcreteDecoratorA --u|> Decorator
ConcreteDecoratorB --u|> Decorator
```

**关键结构**：
1. 装饰器实现与被装饰对象**相同的接口**
2. 装饰器内部**持有**一个接口类型的引用（组合）
3. 可以**多层嵌套**：`DecoratorB(DecoratorA(ConcreteComponent))`

## Go 代码示例

### 示例一：HTTP 中间件（最典型的装饰器应用）

```go
type Handler interface {
    ServeHTTP(w http.ResponseWriter, r *http.Request)
}

// 真实处理器
type UserHandler struct{}
func (h *UserHandler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintln(w, "user data")
}

// 装饰器1：日志
type LoggingDecorator struct {
    next Handler
}
func (d *LoggingDecorator) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    start := time.Now()
    fmt.Printf("[LOG] %s %s\n", r.Method, r.URL.Path)
    d.next.ServeHTTP(w, r)
    fmt.Printf("[LOG] cost: %v\n", time.Since(start))
}

// 装饰器2：认证
type AuthDecorator struct {
    next Handler
}
func (d *AuthDecorator) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    token := r.Header.Get("Authorization")
    if token != "valid-token" {
        http.Error(w, "Unauthorized", http.StatusUnauthorized)
        return
    }
    d.next.ServeHTTP(w, r)
}

// 装饰器3：限流
type RateLimitDecorator struct {
    next    Handler
    limiter *rate.Limiter
}
func (d *RateLimitDecorator) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    if !d.limiter.Allow() {
        http.Error(w, "Too Many Requests", http.StatusTooManyRequests)
        return
    }
    d.next.ServeHTTP(w, r)
}

// 组合叠加：认证 → 限流 → 日志 → 真实处理器
func main() {
    var h Handler = &UserHandler{}
    h = &LoggingDecorator{next: h}
    h = &RateLimitDecorator{next: h, limiter: rate.NewLimiter(10, 1)}
    h = &AuthDecorator{next: h}
    http.Handle("/user", h)
}
```

### 示例二：IO 流（Go 标准库的经典装饰器）

```go
// Go 的 io.Reader / io.Writer 本身就是装饰器模式的最佳体现

f, _ := os.Open("data.txt")               // 基础 Reader

// 套一层：带缓冲
br := bufio.NewReader(f)

// 再套一层：gzip 解压
gr, _ := gzip.NewReader(br)

// 再套一层：限速读取
lr := io.LimitReader(gr, 1024*1024) // 最多读 1MB

// 每一层都是装饰器，叠加能力，接口不变
```

### 示例三：数据库操作增强

```go
type UserRepo interface {
    Save(user *User) error
    FindByID(id int64) (*User, error)
}

// 真实实现
type MySQLUserRepo struct{ db *sql.DB }
func (r *MySQLUserRepo) Save(user *User) error          { /* ... */ return nil }
func (r *MySQLUserRepo) FindByID(id int64) (*User, error) { /* ... */ return nil, nil }

// 装饰器：加缓存
type CachedUserRepo struct {
    next  UserRepo
    cache sync.Map
}
func (r *CachedUserRepo) Save(user *User) error {
    r.cache.Delete(user.ID) // 写时让缓存失效
    return r.next.Save(user)
}
func (r *CachedUserRepo) FindByID(id int64) (*User, error) {
    if v, ok := r.cache.Load(id); ok {
        return v.(*User), nil
    }
    user, err := r.next.FindByID(id)
    if err == nil {
        r.cache.Store(id, user)
    }
    return user, err
}

// 装饰器：加日志
type LoggedUserRepo struct {
    next UserRepo
}
func (r *LoggedUserRepo) Save(user *User) error {
    log.Printf("saving user %d", user.ID)
    err := r.next.Save(user)
    log.Printf("saved user %d, err=%v", user.ID, err)
    return err
}
func (r *LoggedUserRepo) FindByID(id int64) (*User, error) {
    log.Printf("finding user %d", id)
    return r.next.FindByID(id)
}

// 组合
func NewUserRepo(db *sql.DB) UserRepo {
    var repo UserRepo = &MySQLUserRepo{db: db}
    repo = &CachedUserRepo{next: repo}
    repo = &LoggedUserRepo{next: repo}
    return repo
}
```

## 优缺点

| | 说明 |
|--|------|
| ✅ 单一职责 | 每个装饰器只负责一个功能（日志/缓存/限流分离） |
| ✅ 开闭原则 | 新增功能不修改原有代码，新建装饰器即可 |
| ✅ 灵活组合 | 可任意顺序叠加，功能组合自由 |
| ✅ 替代继承 | 避免继承层次过深 |
| ❌ 顺序依赖 | 装饰顺序影响结果，排查问题时需理清调用链 |
| ❌ 调试困难 | 多层嵌套时堆栈较深，不易追踪 |

## 常见应用场景

| 场景 | 说明 |
|------|------|
| **HTTP 中间件** | Gin/Echo/net-http 的 middleware chain |
| **IO 流处理** | `bufio`、`gzip`、`io.LimitReader` |
| **日志增强** | 给任意对象的方法加日志、耗时监控 |
| **缓存层** | 在 Repository 上叠加缓存装饰器 |
| **权限校验** | 在服务层叠加权限检查 |
| **链路追踪** | 给 HTTP/gRPC Client 注入 trace header |
| **重试机制** | 给 Client 叠加 retry 装饰器 |
| **事务管理** | 给 Repository 操作叠加事务装饰器 |

## 与代理模式的区别

见 [[综合]] 中的详细对比。

| | 装饰器模式 | 代理模式 |
|--|-----------|---------|
| **目的** | 叠加/增强功能 | 控制访问 |
| **对象来源** | 被装饰对象由外部传入 | 代理自己管理真实对象 |
| **叠加性** | 多层叠加，顺序明确 | 通常只有一层 |
| **客户端感知** | 通常知道在组合装饰器 | 通常对客户端透明 |
