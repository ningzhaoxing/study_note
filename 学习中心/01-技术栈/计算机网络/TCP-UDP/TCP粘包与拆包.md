# TCP 粘包与拆包

---

## 一、为什么会有粘包/拆包？

TCP 是**面向字节流**的协议，数据在传输时是一段连续的字节流，**没有消息边界**。

与 UDP 不同（UDP 是面向报文的，一次 send 对应一次 recv），TCP 的接收方无法从字节流中知道哪里是一条消息的开始、哪里是结束。

### 粘包（Packet Sticking）

多条消息被合并成一个 TCP 报文一起发送，接收方一次 recv 读取到了多条消息的数据。

```
发送方：[消息A][消息B]
接收方：[消息A 消息B]  ← 粘在一起了
```

**触发原因**：
- Nagle 算法将多个小消息合并成一个报文发送
- 发送速度快，接收方未及时读取，多条消息在缓冲区堆积

### 拆包（Packet Splitting）

一条消息被拆成多个 TCP 报文分开发送，接收方多次 recv 才能读取完整消息。

```
发送方：[消息A(100字节)]
接收方：[消息A前60字节]  [消息A后40字节]  ← 拆开了
```

**触发原因**：
- 消息大小 > MSS（最大报文段），TCP 自动拆分
- 消息大小 > 发送缓冲区剩余空间

---

## 二、为什么 UDP 没有这个问题？

UDP 是**面向消息（报文）**的协议：
- 每次 `sendto()` 对应一次独立的 UDP 数据报
- 接收方每次 `recvfrom()` 只能读取**一个完整的数据报**
- 操作系统保证 UDP 报文的边界

但 UDP 也有自己的问题：数据报过大时会在网络层被分片，可靠性更低。

---

## 三、解决方案

### 方案 1：固定长度

每条消息固定为 N 字节，不足补零，接收方每次读取固定长度。

```
[消息A: 固定100字节][消息B: 固定100字节]
```

- ✅ 实现简单
- ❌ 浪费空间（消息长度不一时，短消息也要补零）

### 方案 2：分隔符（特殊字符）

消息末尾添加特殊分隔符（如 `\n`、`\r\n`、`\0`），接收方按分隔符切割。

```
[消息A内容\n][消息B内容\n]
```

- ✅ 空间利用率高
- ❌ 消息内容不能包含分隔符（需要转义处理）
- 适合文本协议（如 HTTP、Redis 的 RESP 协议）

### 方案 3：消息头携带长度（推荐）

在消息头部固定字节中写入**消息体的长度**，接收方先读 header 获取长度，再读对应字节数的 body。

```
[Header: 4字节长度 | Body: N字节数据]
```

- ✅ 支持任意内容，无转义问题
- ✅ 效率高，解析简单
- 适合二进制协议（如 Dubbo、Netty 自定义协议）

---

## 四、常见框架的处理方式

| 框架/协议 | 解决方式 |
|-----------|----------|
| **HTTP/1.1** | `Content-Length` 头部指定 body 长度；或 `Transfer-Encoding: chunked` 分块传输 |
| **WebSocket** | 帧头携带 Payload Length |
| **Netty** | 提供 `LengthFieldBasedFrameDecoder`、`LineBasedFrameDecoder` 等解码器 |
| **Redis RESP** | 以 `\r\n` 作为分隔符 |
| **gRPC** | Length-Prefixed Message（5字节头部：1字节压缩标志 + 4字节长度） |
