# HTTP 与 RPC 的区别

## 一、基本概念

| | HTTP | RPC |
|--|------|-----|
| **全称** | HyperText Transfer Protocol（超文本传输协议） | Remote Procedure Call（远程过程调用） |
| **本质** | 应用层通信协议 | 通信**模式/框架**（不是单一协议） |
| **设计目标** | 浏览器与服务器之间传输超文本内容 | 让远程调用像本地函数调用一样简单 |
| **底层协议** | TCP（HTTP/3 用 UDP） | 通常基于 TCP（也可基于 HTTP） |

> 二者都是基于 TCP 定义了不同消息格式的**应用层协议**（或框架）。

---

## 二、主要区别

### 2.1 服务发现方式

**HTTP**：
- 基于 **DNS** 解析域名，获取服务器 IP
- 默认端口 80（HTTPS 443）

**RPC**：
- 依赖**注册中心**（如 etcd、Consul、ZooKeeper）
- 服务启动时向注册中心注册自己的地址和接口信息
- 客户端从注册中心查询目标服务的地址

```
RPC 服务发现：
服务B启动 → 注册到 etcd（"service-b" → "192.168.1.10:8080"）
服务A调用 → 查询 etcd → 直接连接 192.168.1.10:8080
```

### 2.2 数据传输格式

**HTTP**：
- 报文以**文本**格式传输（HTTP/2 用二进制帧，但内容仍是文本 JSON）
- 常用 **JSON** 作为数据格式（可读性强，但体积较大）
- 需要考虑浏览器兼容性，有很多冗余设计

**RPC**：
- 使用更紧凑的**二进制序列化协议**（如 Protobuf、Thrift、Avro）
- 体积更小，解析速度更快
- 不需要考虑浏览器，专为服务间通信优化

```
同一份数据:
JSON（HTTP）:  {"id": 42, "name": "Alice"}  → 约 30 字节
Protobuf（RPC）: 二进制编码                → 约 8 字节
```

### 2.3 接口描述方式

**HTTP REST**：
- 通常用 **OpenAPI（Swagger）** 描述接口
- 接口约定灵活，但没有强制的类型检查

**RPC（如 gRPC）**：
- 使用 **IDL（接口描述语言）** 定义接口，如 `.proto` 文件
- 编译器自动生成各语言的客户端/服务端代码
- **强类型**，编译期就能发现接口不匹配

```protobuf
// gRPC .proto 文件示例
service UserService {
  rpc GetUser(GetUserRequest) returns (UserResponse);
}

message GetUserRequest {
  int64 user_id = 1;
}
```

### 2.4 性能对比

| 维度 | HTTP（JSON） | RPC（Protobuf） |
|------|-------------|----------------|
| **序列化大小** | 较大 | 较小（3-10x 压缩） |
| **序列化速度** | 较慢 | 较快（5-10x） |
| **连接建立** | 每次可能重新连接 | 通常维持长连接，连接池复用 |
| **整体性能** | 适中 | 更高（内部服务调用优选） |

## 三、什么时候用 HTTP，什么时候用 RPC？

### 使用 HTTP（REST） 的场景：
- **对外开放的 API**（需要浏览器、移动端等多种客户端接入）
- 对接第三方服务（通用性强）
- 团队技术栈多样，需要跨语言通信且不想维护 Proto 文件

### 使用 RPC 的场景：
- **微服务内部**调用（如服务 A 调用服务 B、C、D）
- 对**性能要求高**的场景（高 QPS、低延迟）
- 需要强类型约束和代码生成的场景

---

## 四、常见 RPC 框架

| 框架 | 公司 | 序列化协议 | 传输协议 |
|------|------|-----------|---------|
| **gRPC** | Google | Protobuf | HTTP/2 |
| **Dubbo** | 阿里 | Hessian/JSON/Protobuf | TCP |
| **Thrift** | Facebook | Thrift Binary | TCP |
| **brpc** | 百度 | Protobuf | TCP/HTTP |

> 有趣的是，**gRPC 底层就是 HTTP/2**，它利用 HTTP/2 的多路复用来提升性能，同时使用 Protobuf 做高效序列化。

---

## 五、总结

```
HTTP（RESTful）：面向资源，对外暴露，通用性强，开发简单
RPC：面向过程，内部调用，性能高，类型安全，但维护成本略高
```

**大型微服务架构的常见做法**：
- 对外（C 端/第三方）：HTTP RESTful API 或 GraphQL
- 内部服务间：gRPC 或其他 RPC 框架
