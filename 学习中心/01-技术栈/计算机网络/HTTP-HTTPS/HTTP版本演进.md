# HTTP 版本演进

## 时间线

```
1991  HTTP/0.9  —— 极简单行协议，只支持 GET
1996  HTTP/1.0  —— 增加请求方法、Header、状态码
1997  HTTP/1.1  —— 持久连接、管线化、分块传输（至今仍广泛使用）
2015  HTTP/2    —— 多路复用、头部压缩、服务器推送（基于 SPDY）
2022  HTTP/3    —— 基于 QUIC（UDP），彻底解决队头阻塞
```

---

## HTTP/0.9（1991）

- 极度简单，只有一行：`GET /index.html`
- 只支持 GET 方法
- 响应只有 HTML 内容，没有 Header、没有状态码
- 连接用完即关闭

---

## HTTP/1.0（1996）

**相比 0.9 的改进：**
- 增加了 **POST、HEAD** 等请求方法
- 引入了 **Header 头部字段**（Content-Type、Status 等）
- 增加了**状态码**（200、404 等）
- 支持传输 HTML 以外的内容（图片、视频等）

**问题：**
- **短连接**：每次请求都要新建 TCP 连接，请求完毕立即断开，性能极差
- 每次请求都要经历 TCP 三次握手的开销

---

## HTTP/1.1（1997）⭐ 主流版本

**相比 1.0 的改进：**

### 1. 持久连接（Keep-Alive）
- 默认开启 `Connection: keep-alive`
- **复用 TCP 连接**，同一个连接可发多个请求，减少握手开销

### 2. 请求管线化（Pipelining）
- 无需等待响应，可以**连续发送多个请求**
- 但服务器必须按顺序响应，存在**队头阻塞（Head-of-Line Blocking）**

### 3. 分块传输（Chunked Transfer）
- 支持 `Transfer-Encoding: chunked`
- 服务器不需要提前知道内容长度，可以**边生成边发送**

### 4. 增加新方法
- 新增 PUT、DELETE、OPTIONS、PATCH 等方法，支持 **RESTful** 风格

### 5. Host 头部
- 允许在同一 IP 上部署多个虚拟主机

**遗留问题：**
- **队头阻塞**：前一个请求未完成，后续请求必须排队等待
- **Header 冗余**：每个请求都携带大量重复 Header（如 Cookie、User-Agent），且以明文传输
- **无优先级**：不能为重要资源设置传输优先级
- **服务器不能主动推送**：只能被动响应请求

---

## HTTP/2（2015）⭐ 现代主流版本

基于 Google 的 **SPDY** 协议发展而来，在 HTTP/1.1 语义不变的前提下，从根本上重构了传输层。

### 核心改进

#### 1. 二进制分帧（Binary Framing）⭐⭐
- HTTP/1.x 是**文本协议**，HTTP/2 改为**二进制协议**
- 将数据分割为更小的**帧（Frame）**，帧是最小传输单位
- 更高效、更易于解析

#### 2. 多路复用（Multiplexing）⭐⭐
- 在**一个 TCP 连接**上，可以**并行**发送多个请求和响应
- 每个请求/响应是一个独立的**流（Stream）**，互不干扰
- 彻底解决了 HTTP/1.x 的**应用层队头阻塞**问题

```
HTTP/1.x:  [请求1 → 响应1] [请求2 → 响应2]  # 串行
HTTP/2:    [请求1 → 响应1]                   # 并行
           [请求2 → 响应2]
           [请求3 → 响应3]
```

#### 3. 头部压缩（HPACK）⭐
- 客户端和服务器共同维护一个**头部字段索引表**
- 重复的 Header 只发送索引值，不重复发送
- 大幅减少头部传输开销（尤其是 Cookie 等大字段）

#### 4. 服务器推送（Server Push）
- 服务器可以在客户端请求之前，**主动推送**资源
- 如：客户端请求 HTML，服务器顺便推送 CSS 和 JS

#### 5. 请求优先级
- 可以为每个流设置**优先级权重**
- 重要资源（如 HTML、CSS）优先传输

**遗留问题：**
- **TCP 队头阻塞**：HTTP/2 基于 TCP，如果 TCP 层丢包，**所有流**都会被阻塞等待重传
- TCP 连接建立 + TLS 握手延迟（需要 1.5~2 个 RTT）

---

## HTTP/3（2022）⭐ 下一代协议

为彻底解决 TCP 的队头阻塞问题，HTTP/3 基于 **QUIC** 协议，运行在 **UDP** 之上。

### 核心改进

#### 1. 基于 QUIC（Quick UDP Internet Connections）
- QUIC = UDP + 可靠传输 + 拥塞控制 + TLS 1.3
- 在 UDP 之上自己实现了可靠传输，每个流独立管理丢包重传

#### 2. 彻底解决队头阻塞
- HTTP/2 的多个流共享一个 TCP 连接，TCP 丢包会阻塞所有流
- HTTP/3 的每个 QUIC 流独立，一个流丢包**不影响其他流**

#### 3. 更快的连接建立
- QUIC 集成了 TLS 1.3，首次连接只需 **1 个 RTT**（HTTP/2 需要 3 个 RTT）
- 复用连接（0-RTT）时，可以做到**零延迟**连接

#### 4. 连接迁移
- 基于**连接 ID** 而非 IP + 端口标识连接
- 网络切换（如 WiFi → 4G）时，连接不需要重新建立

---

## 版本对比总结

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|----------|--------|--------|
| 传输协议 | TCP | TCP | UDP (QUIC) |
| 传输格式 | 文本 | 二进制帧 | 二进制帧 |
| 多路复用 | ❌ | ✅ | ✅ |
| 头部压缩 | ❌ | ✅ (HPACK) | ✅ (QPACK) |
| 服务器推送 | ❌ | ✅ | ✅ |
| TCP 队头阻塞 | ✅ 有 | ✅ 有 | ❌ 无 |
| 连接建立延迟 | 1~2 RTT | 1~2 RTT | 0~1 RTT |
| 连接迁移 | ❌ | ❌ | ✅ |
