# HTTP 与 HTTPS 的区别

## 一、核心区别总结

| 维度 | HTTP | HTTPS |
|------|------|-------|
| **安全性** | 明文传输，不安全 | 加密传输，安全 |
| **默认端口** | **80** | **443** |
| **握手过程** | TCP 三次握手 | TCP 三次握手 + **SSL/TLS 握手** |
| **证书** | 不需要 | 需要向 **CA** 申请数字证书 |
| **性能** | 较快（无加密开销） | 略慢（有加解密开销，但差距正在缩小） |
| **URL** | `http://` | `https://` |

---

## 二、HTTP 存在的风险

### 🔴 窃听风险（明文传输）
- HTTP 报文以**明文**传输，任何中间节点（代理、路由器）都可以读取内容

### 🔴 篡改风险（无完整性校验）
- 传输过程中，内容可能被中间人**篡改**，且通信双方无法发现

### 🔴 冒充风险（无身份认证）
- 无法验证通信对方的**真实身份**，可能遭受中间人攻击（MITM）

---

## 三、HTTPS 如何解决这三个问题

| 风险 | 解决方案 | 机制 |
|------|----------|------|
| 窃听风险 | **信息加密** | 混合加密（对称 + 非对称） |
| 篡改风险 | **校验机制** | 摘要算法（数字签名） |
| 冒充风险 | **身份证书** | CA 颁发的数字证书 |

### 3.1 信息加密：混合加密

单用对称加密：密钥传输不安全（如何安全地分发密钥？）
单用非对称加密：性能差，速度慢

**混合加密方案**：
1. 用**非对称加密**安全地协商一个"会话密钥"
2. 用**对称加密**（会话密钥）加密实际通信数据

```
非对称加密（RSA/ECDHE）→ 安全协商会话密钥
对称加密（AES/ChaCha20）→ 高效加密数据传输
```

### 3.2 校验机制：摘要算法

使用 **SHA-256** 等摘要算法对数据计算哈希值，连同数据一起传输。
接收方重新计算哈希对比，如果不一致则说明数据被篡改。

```
发送方: 数据 → Hash(数据) = 摘要 → 发送 (数据 + 摘要)
接收方: 收到 (数据 + 摘要) → Hash(数据) 与摘要对比
```

### 3.3 身份证书：数字证书体系

CA（Certificate Authority，证书颁发机构）用自己的私钥对服务器的公钥进行签名，颁发**数字证书**。

```
证书包含：域名、服务器公钥、有效期、CA 签名
浏览器：预装各大 CA 的根证书 → 验证服务器证书签名
```

## 四、SSL/TLS 协议详解

### 基本流程

```
1. 客户端向服务器索要并验证服务器的公钥（通过证书）
2. 双方协商产生会话密钥
3. 双方使用会话密钥进行加密通信
```

### TLS 1.2 握手（4 次通信）

**第 1 步：Client Hello（客户端发起）**
- 客户端支持的 TLS 版本
- 客户端随机数 `Random1`
- 支持的密码套件列表（如 TLS_AES_128_GCM_SHA256）

**第 2 步：Server Hello + 证书（服务端响应）**
- 确认使用的 TLS 版本
- 服务器随机数 `Random2`
- 选定的密码套件
- 服务器的**数字证书**（含服务器公钥）

**第 3 步：客户端验证并发送预主密钥**
1. 用浏览器/操作系统内置的 CA 公钥，验证证书真实性
2. 从证书中取出服务器公钥
3. 生成随机数 `Pre-Master Key`，用**服务器公钥加密**后发送
4. 发送"Change Cipher Spec"通知，之后改用会话密钥通信
5. 发送握手完成通知（含之前所有握手数据的摘要）

**第 4 步：服务器确认**
1. 用私钥解密 `Pre-Master Key`
2. 用 `Random1 + Random2 + Pre-Master Key` 生成**会话密钥（Session Key）**
3. 发送"Change Cipher Spec"通知
4. 发送握手完成通知

此后所有通信都使用 Session Key 对称加密。

```
                 ← 1 RTT TCP 握手 →
                 ← 2 RTT TLS 握手 →   (TLS 1.2)
                 ← 1 RTT TLS 握手 →   (TLS 1.3)
```

### TLS 1.3 的改进

- 握手从 **2 RTT** 缩短为 **1 RTT**（或 0-RTT 复用）
- 移除了不安全的密码套件（RSA 密钥交换）
- 强制使用**前向保密**（ECDHE），即使私钥泄露，历史通信也无法解密

## 五、HTTPS 的性能问题

**曾经的问题**（已基本解决）：
- TLS 握手增加 1~2 RTT 延迟
- 加解密消耗 CPU

**现代优化手段**：
- **TLS Session Resumption**：复用之前的会话密钥，0-RTT 恢复
- **OCSP Stapling**：服务器预先获取证书状态，避免客户端额外查询
- **HTTP/2 + TLS**：多路复用摊薄握手成本
- 硬件加速：AES-NI 指令集大幅提升加解密速度

## 六、面试常见问题

### Q：HTTPS 一定安全吗？
**A**：不是绝对安全。如果用户安装了伪造的根证书，或者 CA 机构被攻破，则 HTTPS 可能被中间人攻击。此外，HTTPS 只保护传输过程，不保护服务端存储安全。

### Q：数字证书的验证链是什么？
**A**：
```
根证书（Root CA，浏览器内置）
  → 中间证书（Intermediate CA）
    → 服务器证书（Domain Certificate）
```
浏览器沿链向上验证，直到匹配内置的根证书为止。

### Q：HTTP 切换到 HTTPS 需要做什么？
**A**：
1. 向 CA 申请 SSL/TLS 数字证书（可用 Let's Encrypt 免费申请）
2. 服务器配置证书和私钥
3. 设置 HTTP → HTTPS 的 301 重定向
4. 配置 HSTS（HTTP Strict Transport Security），强制浏览器只用 HTTPS
