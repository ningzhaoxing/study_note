# 变形与构造：改变或创建树的结构

> 前一篇的题都是在"读"树，这一篇开始"写"树——翻转、展开、从序列中重建、以及找到两个节点的公共祖先。

---

## 翻转二叉树（LeetCode 226）

### 问题

把二叉树的每个节点的左右子树交换，返回翻转后的根节点。

```
原始：          翻转后：
    4               4
   / \             / \
  2   7    →     7   2
 / \ / \         / \ / \
1  3 6  9       9  6 3  1
```

### 前序思维的最简单例子

翻转整棵树，就是对每个节点都做"交换左右子树"这一件事。

这是一个典型的前序操作：**当前节点先动手（交换左右），然后把同样的事情递归交给子树**。没有什么信息需要从子树返回，当前节点需要做的事只依赖它自己。

```go
func invertTree(root *TreeNode) *TreeNode {
    if root == nil { return nil }
    root.Left, root.Right = root.Right, root.Left // 当前节点：交换左右
    invertTree(root.Left)
    invertTree(root.Right)
    return root
}
```

也可以写成后序（先递归子树再交换），但这里前序更自然——因为交换这件事和子树的结果无关，早做晚做都行。用前序是因为逻辑上"当前节点先处理"读起来更直接。

## 二叉树展开为链表（LeetCode 114）

### 问题

将二叉树"原地"展开为一个"链表"——所有节点只使用 `Right` 指针连接，`Left` 全部置为 `nil`，连接顺序和前序遍历的顺序一致。

```
原始：          展开后：
    1           1
   / \           \
  2   5    →     2
 / \   \           \
3   4   6           3
                     \
                      4
                       \
                        5
                         \
                          6
```

### 为什么必须是后序

这道题是理解"后序为什么存在"的好例子。

**展开的结果是前序序列**（根→左→右），但处理的顺序必须是**后序**（子树先处理好，当前节点再操作）。原因是：如果当前节点先动手（把左子树接到右边），左右子树还没被展开，拼接后的结构乱了。

必须先让左右子树各自展开成链，再把两条链拼接到当前节点后面：

```
flatten(root.Left)   → 左子树已经是一条链了
flatten(root.Right)  → 右子树已经是一条链了

此时：  root
       /    \
  [左链头]  [右链头]

操作：把右链接到左链的尾部，再把整条左链接到 root.Right，清空 root.Left
结果：root → 左链 → 右链
```

```go
func flatten(root *TreeNode) {
    if root == nil { return }
    if root.Left == nil && root.Right == nil { return }

    flatten(root.Left)
    flatten(root.Right)

    // 后序位置：左右子树已经展开完毕
    if root.Left != nil {
        // 找到左链的尾部
        leftTail := root.Left
        for leftTail.Right != nil {
            leftTail = leftTail.Right
        }
        // 左链尾部接上右链
        leftTail.Right = root.Right
        // root.Right 接上左链，清空 root.Left
        root.Right = root.Left
        root.Left = nil
    }
}
```

有一个更简洁的写法利用"逆前序"（从后往前）+ 前向指针，但上面这个版本更直接地体现了"为什么要用后序"。

### 树展开为链表：和链表的交汇

这道题的结果本质上是个链表（用 `Right` 指针串起来），这是二叉树笔记里和 [[链表/00-核心思维框架|链表]] 唯一的显式交汇点。树的递归结构帮助我们按正确顺序把节点穿成一串，这恰好利用了树的"结构即顺序"这一特性——前序遍历的顺序，编码在树的形状里，递归自然地还原了它。

## 通过遍历序列构造二叉树

### 核心思路

这三道构造题的逻辑是完全一致的：**遍历序列告诉你根节点是谁，中序序列告诉你左右子树各包含哪些节点**。

每次构造只做这一件事：从序列里找到根节点，用中序序列划分左右子树，然后递归地构造左右子树。

```
找到根节点
    ↓
在中序序列里定位根
    ↓
左边 = 左子树的中序   右边 = 右子树的中序
左边大小 → 从另一个序列里切出左子树的部分
    ↓
递归构造左子树和右子树
```

### 前序 + 中序（LeetCode 105）

前序的第一个元素永远是当前子树的根：

```go
func buildTree(preorder []int, inorder []int) *TreeNode {
    if len(preorder) == 0 { return nil }
    root := &TreeNode{Val: preorder[0]} // 前序首元素是根
    // 在中序里找根，左边是左子树，右边是右子树
    mid := 0
    for i, v := range inorder {
        if v == preorder[0] { mid = i; break }
    }
    // 左子树大小 = mid，据此切割前序序列
    root.Left = buildTree(preorder[1:1+mid], inorder[:mid])
    root.Right = buildTree(preorder[1+mid:], inorder[mid+1:])
    return root
}
```

**为什么前序切割方式是 `[1:1+mid]`？** 左子树大小是 `mid`（中序里根左边的元素个数），前序里根之后 `mid` 个元素属于左子树，剩下的属于右子树。

### 中序 + 后序（LeetCode 106）

后序的最后一个元素永远是当前子树的根，其余逻辑和上面完全一样：

```go
func buildTree(inorder []int, postorder []int) *TreeNode {
    if len(postorder) == 0 { return nil }
    rootVal := postorder[len(postorder)-1] // 后序末元素是根
    root := &TreeNode{Val: rootVal}
    mid := 0
    for i, v := range inorder {
        if v == rootVal { mid = i; break }
    }
    // 左子树大小 = mid，后序里前 mid 个属于左子树
    root.Left = buildTree(inorder[:mid], postorder[:mid])
    root.Right = buildTree(inorder[mid+1:], postorder[mid:len(postorder)-1])
    return root
}
```

### 前序 + 后序（LeetCode 889）

没有中序的帮助，无法唯一确定一棵树（一个子节点的节点，无法判断它是左孩子还是右孩子）。所以这道题只要求返回任意一棵满足条件的树。

约定：`preorder[1]` 是左子树的根（如果左子树存在）。在后序里找到这个值，它左边的所有节点（含自身）构成左子树：

```go
func constructFromPrePost(preorder []int, postorder []int) *TreeNode {
    if len(preorder) == 0 { return nil }
    root := &TreeNode{Val: preorder[0]}
    if len(preorder) == 1 { return root }   // 只有一个节点，直接返回
    leftRootVal := preorder[1]              // 约定左子树的根是前序第二个
    leftSize := 0
    for i, v := range postorder {
        if v == leftRootVal { leftSize = i + 1; break }
    }
    root.Left = constructFromPrePost(preorder[1:1+leftSize], postorder[:leftSize])
    root.Right = constructFromPrePost(preorder[1+leftSize:], postorder[leftSize:len(postorder)-1])
    return root
}
```

### 三道构造题的共同结构

```
定位根节点（前序首 / 后序末 / 前序首）
    ↓
用中序（或后序辅助）计算左子树大小
    ↓
按左子树大小切割序列，递归构造左右子树
```

这三道题本质是遍历的**逆操作**——遍历是从树输出序列，构造是从序列还原树。它们和 [[01-遍历与感知]] 构成了一对互逆关系。

## 二叉树的最近公共祖先（LeetCode 236）

### 问题

给定一棵二叉树和两个节点 `p`、`q`，找到它们的最近公共祖先（LCA）。

```
        3
       / \
      5   1
     / \ / \
    6  2 0  8
      / \
     7   4

p=5, q=1 → LCA = 3
p=5, q=4 → LCA = 5（节点本身也可以是自己的祖先）
```

### 后序的最优雅体现

这道题是后序思维最漂亮的一个例子。思路是：让每棵子树自己汇报，"我的范围内有没有找到 p 或者 q？"

函数的语义：`lca(root, p, q)` 返回在以 `root` 为根的子树里，找到的 `p` 或 `q`（或它们的公共祖先）。

两个节点可能的位置关系只有两种：
1. **p 和 q 分居两侧**：左子树返回了其中一个，右子树也返回了其中一个，那当前节点就是 LCA。
2. **p 和 q 都在同一侧**：另一侧返回 `nil`，LCA 在有结果的那一侧的子树里。

```go
func lowestCommonAncestor(root, p, q *TreeNode) *TreeNode {
    // 遇到空节点，或者找到了 p 或 q，直接返回
    if root == nil || root == p || root == q { return root }

    left := lowestCommonAncestor(root.Left, p, q)
    right := lowestCommonAncestor(root.Right, p, q)

    // 后序位置：左右子树的结果都回来了，当前节点做判断
    if left != nil && right != nil {
        return root  // p 和 q 分居两侧，当前节点就是 LCA
    }
    if left != nil { return left }  // p 和 q 都在左侧
    return right                    // p 和 q 都在右侧（或 right 为 nil）
}
```

这段代码短小精悍，但背后的逻辑密度很高。读懂它的关键是理解函数的返回值语义：**它不只是返回"找没找到"，而是返回"在这棵子树范围内，和 p/q 相关的最有价值的节点"**。

这和 [[01-遍历与感知]] 里的直径很像：函数对外承诺返回一件事（那里是深度，这里是最有价值的节点），但在后序位置做了更丰富的判断。

## 这几道题的共同节奏

```
翻转二叉树  ← 最简单的"前序：当前节点先做，再交给子树"
    ↓
展开为链表  ← 最清晰的"后序：子树先完成，当前节点再综合"
    ↓
构造二叉树  ← 递归的"逆向"：序列 → 树
    ↓
最近公共祖先 ← 后序的最优雅形式：函数返回值承载了复合语义
```

从翻转到 LCA，核心始终是那一个问题：**当前节点在什么时刻做什么**。
