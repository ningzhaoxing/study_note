# 双指针系列：两个指针联动的几道题

> 这几道题有个共同的问题：单靠一个指针游走，你没法直接"看到"你需要的位置。所以需要两个指针配合——它们之间的关系编码了你需要的信息。

---

## [相交链表](https://leetcode.cn/problems/intersection-of-two-linked-lists/)（LeetCode 160）

### 问题

两个链表可能在某个节点开始共用同一段尾部，找到这个第一个公共节点。如果不相交，返回 `nil`。

```
链表 A：  a1 → a2 ──────────┐
                              c1 → c2 → c3
链表 B：  b1 → b2 → b3 ──→─┘
```

### 直接的障碍

两个链表长度不同。如果两个指针同时从各自头部出发，到达交点时走的步数不一样，它们不会同时出现在同一个节点上。

### 解法

让指针 A 走完链表 A 后，跳到链表 B 的头部继续走；让指针 B 走完链表 B 后，跳到链表 A 的头部继续走。

这样两个指针走的总步数都是 `len(A) + len(B)`，交点之后的部分是两条链表共用的，所以它们一定会在交点处同时到达。

如果没有交点，两个指针最终都会走到 `nil`，也是相等的，循环自然结束。

```go
func getIntersectionNode(headA, headB *ListNode) *ListNode {
    a, b := headA, headB
    for a != b {
        if a == nil {
            a = headB
        } else {
            a = a.Next
        }
        if b == nil {
            b = headA
        } else {
            b = b.Next
        }
    }
    return a
}
```

这个解法没有做任何长度计算，靠的是：两个指针走完各自的路再走对方的路，总步数相等，差距自然就消除了。

## 环形链表（[LeetCode 141](https://leetcode.cn/problems/linked-list-cycle/) + [142](https://leetcode.cn/problems/linked-list-cycle-ii/)）

### 判断是否有环（LeetCode 141）

**思路**：让一个指针每次走 1 步（slow），另一个每次走 2 步（fast）。如果链表有环，fast 最终一定会在环里追上 slow——就像两个人在操场跑圈，跑得快的人迟早会套圈追上跑得慢的。

```go
func hasCycle(head *ListNode) bool {
    slow, fast := head, head
    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next
        if slow == fast {
            return true
        }
    }
    return false
}
```

`fast != nil && fast.Next != nil` 是为了避免 fast 走两步时访问到 nil 的 Next 导致 panic。

### 找到环的入口（LeetCode 142）

这道题有一个很漂亮的数学推导，结论是：**slow 和 fast 相遇后，把 slow 放回链表头，然后两个指针都每次走 1 步，再次相遇的地方就是环的入口**。

为什么？画出来推一下。

设：
- `L` = 链表头到环入口的距离
- `C` = 环的周长
- `k` = 相遇时 slow 在环内已经走了多少步

slow 走的总步数是 `L + k`，fast 是 slow 的两倍所以是 `2(L + k)`，但 fast 比 slow 多走了若干圈，即：

```
2(L + k) = L + k + n*C
L + k = n*C
L = n*C - k
```

`n*C - k` 是什么？fast 和 slow 的相遇点距离环入口还有 `C - k` 步（沿环方向），而 `n*C - k` 就是走了若干整圈再走 `C - k` 步，最终也停在环入口。

所以从相遇点出发走 `n*C - k` 步 = 从链表头走 `L` 步，两者都到达环入口。

```go
func detectCycle(head *ListNode) *ListNode {
    slow, fast := head, head
    for fast != nil && fast.Next != nil {
        slow = slow.Next
        fast = fast.Next.Next
        if slow == fast {
            slow = head // slow 回到链表头
            for slow != fast {
                slow = slow.Next
                fast = fast.Next // fast 也改为每次走 1 步
            }
            return slow
        }
    }
    return nil
}
```

这道题的巧妙在于：相遇点本身隐含了足够的信息来定位环的入口，不需要去算环的大小或链表的长度。

## [删除链表的倒数第 N 个节点](https://leetcode.cn/problems/remove-nth-node-from-end-of-list/)（LeetCode 19）

### 问题

删除倒数第 N 个节点，返回链表头节点。

### 思路

倒数第 N 个，从前往后数不好数（不知道链表总长）。但如果让两个指针始终保持 N 个节点的距离同步移动，当前面的指针走到末尾时，后面的指针刚好在倒数第 N 个节点上。

具体来说：让 fast 先走 N+1 步，再让 slow 和 fast 同步走，fast 到 nil 时，slow 停在目标节点的**前一个**（因为删除节点需要找它的前驱）。

为什么是 N+1 步而不是 N 步？因为要让 slow 停在目标的前驱，而不是目标本身。

```
链表：dummy → 1 → 2 → 3 → 4 → 5 → nil，删倒数第 2 个（节点 4）

fast 先走 3 步（N+1=3）：
  slow: dummy
  fast: 3

同步移动直到 fast 到 nil：
  slow: 3
  fast: nil

slow.Next 就是节点 4，执行 slow.Next = slow.Next.Next 完成删除
```

```go
func removeNthFromEnd(head *ListNode, n int) *ListNode {
    dummy := &ListNode{Next: head}
    slow, fast := dummy, dummy

    for i := 0; i <= n; i++ {
        fast = fast.Next
    }
    for fast != nil {
        slow = slow.Next
        fast = fast.Next
    }
    slow.Next = slow.Next.Next

    return dummy.Next
}
```

**为什么需要虚拟头节点？** 如果要删的是头节点本身（倒数第 N 个等于链表长度时），头节点没有前驱，`slow.Next = slow.Next.Next` 会出错。虚拟头节点给头节点提供了一个前驱，让逻辑统一。

## 这几道题的共同点

单个指针游走时，你只能感知"当前在哪"。这几道题的问题都是：

- **相交链表**：不知道两条路从哪开始重合
- **环形链表**：不知道有没有环，更不知道环在哪
- **删除倒数第 N 个**：不知道倒数第 N 个在哪

共同的应对方式是让两个指针以某种关系联动，用它们的相对位置来编码你需要的信息。但每道题联动的具体方式不同，不要强行套路，理解每道题背后的"为什么"比记住套路重要。
