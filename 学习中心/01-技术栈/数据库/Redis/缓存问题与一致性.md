## 一、缓存穿透

### 1.1 问题描述

**定义**：查询一个**数据库中不存在**的 key，缓存中也不存在，每次请求都直接打到数据库。

```
正常请求：Client → Redis（命中）→ 返回数据
穿透请求：Client → Redis（未命中）→ DB（查不到）→ 返回 null
          下次再来：Client → Redis（还是未命中）→ DB → 返回 null（循环）
```

**危害**：大量穿透请求导致数据库压力剧增，甚至打垮数据库（常见于恶意攻击：用大量不存在的 ID 请求接口）。

### 1.2 解决方案

#### 方案一：缓存空值

查询 DB 返回 null 时，将该 key 以空值写入缓存，设置**较短的 TTL**（如 5 分钟）。

```go
func GetUser(ctx context.Context, rdb *redis.Client, db *gorm.DB, userID int64) (*User, error) {
    key := fmt.Sprintf("user:%d", userID)

    // 1. 查缓存
    val, err := rdb.Get(ctx, key).Result()
    if err == nil {
        if val == "NULL" {
            return nil, nil // 命中空值缓存，直接返回
        }
        var user User
        json.Unmarshal([]byte(val), &user)
        return &user, nil
    }

    // 2. 查数据库
    var user User
    result := db.First(&user, userID)
    if errors.Is(result.Error, gorm.ErrRecordNotFound) {
        // 缓存空值，TTL 设短（防止正常数据被长期屏蔽）
        rdb.Set(ctx, key, "NULL", 5*time.Minute)
        return nil, nil
    }

    // 3. 写缓存
    data, _ := json.Marshal(user)
    rdb.Set(ctx, key, data, 30*time.Minute)
    return &user, nil
}
```

**局限**：如果攻击者每次使用不同的 key，空值缓存会占满内存。

#### 方案二：布隆过滤器（Bloom Filter）

在缓存层之前加一个布隆过滤器，**提前过滤掉一定不存在的 key**。

```
Client → 布隆过滤器 → (判断可能存在) → Redis → DB
                  ↓
           (判断一定不存在) → 直接返回，不打 DB
```

```go
// 使用 Redis 的 BloomFilter（需要 RedisBloom 模块）
// 或使用 go 内存布隆过滤器（适合单机）
import "github.com/bits-and-blooms/bloom/v3"

var bf *bloom.BloomFilter

func init() {
    // 预期存储 1000000 个元素，误判率 0.001
    bf = bloom.NewWithEstimates(1000000, 0.001)
    // 启动时从 DB 加载所有合法 ID 到布隆过滤器
    loadAllUserIDs(bf)
}

func GetUserWithBloom(ctx context.Context, rdb *redis.Client, db *gorm.DB, userID int64) (*User, error) {
    key := fmt.Sprintf("user:%d", userID)

    // 布隆过滤器判断：一定不存在则直接返回
    if !bf.Test([]byte(key)) {
        return nil, nil
    }

    // 后续正常走缓存 → DB 流程
    // ...
}
```

**特性**：
- 布隆过滤器**不会漏报**（说不存在就一定不存在）
- **可能误报**（说存在可能实际不存在，误判率可调）
- 不支持删除元素（可用 Counting Bloom Filter 解决）

| 方案 | 优点 | 缺点 |
|------|------|------|
| 缓存空值 | 实现简单 | 内存浪费，需设短 TTL |
| 布隆过滤器 | 内存高效，彻底过滤无效请求 | 有误判率，不支持删除，需维护 |

## 二、缓存击穿（热点 Key 失效）

### 2.1 问题描述

**定义**：一个**热点 key 恰好在过期瞬间**，大量并发请求同时穿透到数据库。

```
正常情况：热点 key 在缓存中 → 所有请求命中缓存
击穿瞬间：热点 key 过期 → 大量请求同时 miss → 全部打到 DB（可能几千 QPS 同时查 DB）
```

**与缓存雪崩的区别**：击穿是**单个热点 key**，雪崩是**大量 key 同时失效**。

### 2.2 解决方案

#### 方案一：互斥锁（Mutex Lock）

只允许一个请求去查 DB 并重建缓存，其他请求等待。

```go
func GetHotData(ctx context.Context, rdb *redis.Client, db *gorm.DB, key string) (string, error) {
    // 1. 查缓存
    val, err := rdb.Get(ctx, key).Result()
    if err == nil {
        return val, nil
    }

    // 2. 缓存 miss，尝试获取分布式锁
    lockKey := "lock:" + key
    locked, err := rdb.SetNX(ctx, lockKey, "1", 10*time.Second).Result()
    if err != nil {
        return "", err
    }

    if !locked {
        // 未抢到锁，等待后重试（等待缓存重建完毕）
        time.Sleep(50 * time.Millisecond)
        return GetHotData(ctx, rdb, db, key) // 递归重试
    }
    defer rdb.Del(ctx, lockKey)

    // 3. 抢到锁的请求查 DB 并重建缓存
    data := queryFromDB(db, key)
    rdb.Set(ctx, key, data, 30*time.Minute)
    return data, nil
}
```

#### 方案二：逻辑过期（Logical Expiration）

缓存中**永不设置 TTL**，但在 value 中存储一个逻辑过期时间字段。查询时判断是否逻辑过期，若过期则异步更新缓存，当前请求返回旧数据。

```go
type CacheValue struct {
    Data      interface{} `json:"data"`
    ExpireAt  int64       `json:"expire_at"` // 逻辑过期时间（Unix 时间戳）
}

func GetHotDataLogical(ctx context.Context, rdb *redis.Client, key string) (interface{}, error) {
    val, err := rdb.Get(ctx, key).Result()
    if err != nil {
        return nil, err
    }

    var cv CacheValue
    json.Unmarshal([]byte(val), &cv)

    // 未逻辑过期，直接返回
    if time.Now().Unix() < cv.ExpireAt {
        return cv.Data, nil
    }

    // 逻辑过期，尝试异步更新（非阻塞）
    lockKey := "lock:" + key
    locked, _ := rdb.SetNX(ctx, lockKey, "1", 10*time.Second).Result()
    if locked {
        // 只有抢到锁的协程去异步重建缓存
        go func() {
            defer rdb.Del(ctx, lockKey)
            newData := queryFromDB(nil, key)
            newCV := CacheValue{Data: newData, ExpireAt: time.Now().Add(30 * time.Minute).Unix()}
            data, _ := json.Marshal(newCV)
            rdb.Set(ctx, key, data, 0) // 不设 TTL
        }()
    }

    // 当前请求直接返回旧数据（可用性优先）
    return cv.Data, nil
}
```

### 2.3 两种方案对比

| 对比维度 | 互斥锁 | 逻辑过期 |
|----------|--------|----------|
| 数据一致性 | **高**（等待缓存重建完成） | 低（可能返回旧数据） |
| 可用性 | 低（等待期间请求阻塞/排队） | **高**（始终有数据返回） |
| 实现复杂度 | 较低 | 较高 |
| 适用场景 | 对数据一致性要求高 | 对可用性要求高（允许短暂旧数据） |
| 额外内存 | 无 | 需存储逻辑过期时间 |

## 三、缓存雪崩

### 3.1 问题描述

**定义**：大量 key **在同一时间段集中过期**，或 **Redis 服务宕机**，导致大量请求直接打到数据库。

**场景一**：电商大促，预热缓存时大量商品数据设置了相同的 TTL，同时失效。
**场景二**：Redis 主节点宕机，所有流量打到数据库。

### 3.2 解决方案

#### 针对大量 key 同时过期

**随机化 TTL**：在基础 TTL 上加随机偏移，将过期时间打散。

```go
func SetWithJitter(ctx context.Context, rdb *redis.Client, key string, value interface{}, baseTTL time.Duration) error {
    // 在 baseTTL 基础上随机增减 10%
    jitter := time.Duration(rand.Int63n(int64(baseTTL / 10)))
    actualTTL := baseTTL + jitter
    data, _ := json.Marshal(value)
    return rdb.Set(ctx, key, data, actualTTL).Err()
}
```

#### 针对 Redis 宕机

1. **Redis 高可用**：哨兵模式（Sentinel）或集群模式（Cluster），自动故障转移
2. **多级缓存**：本地缓存（如 `go-cache`）+ Redis 缓存，Redis 宕机时本地缓存兜底
3. **服务降级 + 熔断**：Redis 不可用时，直接返回默认值或降级数据，防止雪崩传递到 DB

```go
// 多级缓存示例
import gocache "github.com/patrickmn/go-cache"

var localCache = gocache.New(5*time.Minute, 10*time.Minute)

func GetWithMultiLevel(ctx context.Context, rdb *redis.Client, key string) (string, error) {
    // L1: 本地缓存
    if val, found := localCache.Get(key); found {
        return val.(string), nil
    }

    // L2: Redis 缓存
    val, err := rdb.Get(ctx, key).Result()
    if err == nil {
        localCache.Set(key, val, gocache.DefaultExpiration)
        return val, nil
    }

    // L3: 数据库（兜底）
    val = queryFromDB(nil, key)
    localCache.Set(key, val, gocache.DefaultExpiration)
    return val, nil
}
```

## 四、缓存一致性策略

### 4.1 Cache Aside Pattern（旁路缓存）

**最常用的缓存一致性策略**，读写逻辑如下：

**读操作**：
```
1. 查 Redis，命中则返回
2. 未命中则查 DB
3. 将 DB 结果写入 Redis，返回结果
```

**写操作**：
```
1. 更新 DB
2. 删除 Redis 中对应的 key（而非更新）
```

### 4.2 为什么是"删除缓存"而非"更新缓存"？

**直接更新缓存的问题（并发写导致脏数据）**：

```
时序：
线程A 更新 DB（写100）→ 线程B 更新 DB（写200）→ 线程B 更新缓存（200）→ 线程A 更新缓存（100）
结果：DB=200，缓存=100  ← 脏数据！
```

**删除缓存不会有这个问题**：删除是幂等操作，无论哪个线程后执行，结果都是缓存被删除，下次读时从 DB 加载最新值。

### 4.3 先删缓存再更新 DB 的问题

```
时序：
线程A 删缓存 → 线程B 读缓存（miss）→ 线程B 读 DB（旧数据）→ 线程A 更新 DB（新数据）→ 线程B 写缓存（旧数据）
结果：DB=新数据，缓存=旧数据  ← 长期脏数据！
```

**结论**：应该**先更新 DB，再删除缓存**（Cache Aside 标准做法）。

### 4.4 先更新 DB 再删除缓存的小概率问题

```
时序（缓存恰好在查询时失效）：
线程A 读缓存（miss，缓存此刻恰好过期）→ 线程A 读 DB（旧数据）→ 线程B 更新 DB（新数据）→ 线程B 删缓存 → 线程A 写缓存（旧数据）
结果：DB=新数据，缓存=旧数据
```

**出现条件非常苛刻**（需要读操作在写操作之前开始，又在写操作完成之后才写缓存），实际发生概率极低。通过**设置合理 TTL** 可以限制不一致窗口。

### 4.5 延迟双删

详见：[[延迟双删缓存策略]]

**流程**：
```
1. 先删除缓存
2. 更新 DB
3. 延迟 N ms 后再次删除缓存
```

**延迟时间如何确定？**
- 延迟时间需要大于"一次 DB 读 + 写缓存"的时间
- 在主从数据库场景下，延迟时间还需大于**主从同步延迟**（通常几十 ms~几百 ms）
- 实践中通常设置 **500ms~1s**，具体需根据业务压测确定

**局限性**：
- 延迟期间（步骤 1 到步骤 3 之间）仍存在脏数据窗口
- 第二次删除若失败（Redis 故障），依然会有脏数据
- 延迟删除增加了系统复杂度

```go
func UpdateUserWithDelayedDoubleDelete(ctx context.Context, rdb *redis.Client, db *gorm.DB, user *User) error {
    key := fmt.Sprintf("user:%d", user.ID)

    // 1. 先删除缓存
    rdb.Del(ctx, key)

    // 2. 更新数据库
    if err := db.Save(user).Error; err != nil {
        return err
    }

    // 3. 延迟再次删除（异步，不阻塞当前请求）
    go func() {
        time.Sleep(500 * time.Millisecond) // 延迟时间根据业务情况调整
        rdb.Del(ctx, key)
    }()

    return nil
}
```

### 4.6 强一致性方案：binlog 订阅 + MQ

对于对**数据一致性要求极高**的场景（如金融、库存），使用 Canal + MQ 实现缓存与 DB 的最终一致性。

```
写操作流程：
应用 → 更新 DB（MySQL binlog 记录变更）
                ↓
         Canal 订阅 binlog
                ↓
         发送 MQ 消息（变更事件）
                ↓
         消费者删除/更新 Redis 缓存
```

**优点**：
- 缓存更新与业务代码解耦
- 即使应用层缓存删除失败，Canal 兜底保证最终一致
- 支持数据库主从架构（Canal 订阅从库 binlog）

**缺点**：
- 架构复杂，引入 Canal + MQ 组件
- 存在短暂延迟（binlog → MQ → 消费 → 更新缓存）

## 五、缓存策略总体对比

| 场景 | 推荐方案 | 一致性 | 复杂度 |
|------|----------|--------|--------|
| 通用读多写少 | Cache Aside（先更新DB再删缓存） | 高（极低概率脏数据） | 低 |
| 主从DB架构 | 延迟双删 | 较高 | 中 |
| 强一致性要求 | Canal + MQ binlog 订阅 | 最终一致 | 高 |
| 热点数据 | 逻辑过期 or 互斥锁 | 取决于方案 | 中 |
| 防穿透 | 布隆过滤器 + 空值缓存 | - | 中 |
| 防雪崩 | 随机TTL + Redis高可用 + 多级缓存 | - | 中高 |

## 六、面试高频问题

**Q1：缓存穿透、击穿、雪崩有什么区别？**

> - **穿透**：查询不存在的 key，每次都打 DB（key 在 DB 中就没有）
> - **击穿**：单个热点 key 过期瞬间，大量请求同时打 DB
> - **雪崩**：大量 key 同时过期 或 Redis 宕机，导致大批请求打 DB

**Q2：为什么 Cache Aside 要删除缓存而不是更新缓存？**

> 并发写场景下，若两个线程同时更新 DB 后再更新缓存，后写入缓存的值可能是较旧的，导致脏数据。而删除是幂等的，删除后下次读时从 DB 加载最新值，不会有并发写脏数据问题。

**Q3：延迟双删的延迟时间怎么定？有什么缺陷？**

> 延迟时间需要大于"业务读 DB + 写缓存"的时间，在主从场景下还要大于主从同步延迟，实践中通常取 500ms~1s。缺陷是：延迟期间仍有短暂脏数据窗口；第二次删除若失败依然脏数据；增加了系统复杂度。

**Q4：如何保证缓存和 DB 的强一致性？**

> 业务层面很难做到完全强一致（只能做最终一致），如需接近强一致可以用：加读写锁（低并发场景）、或 Canal 订阅 binlog + MQ 异步更新缓存。绝大多数业务通过合理设置 TTL + Cache Aside 就能满足需求。
