# Redis 持久化机制

> 关联笔记：[[Redis 及其优缺点]] | [[缓存问题与一致性]] | [[Redis 主从复制]]

---

## 一、为什么需要持久化？

Redis 是**内存数据库**，进程重启或宕机后内存数据全部丢失。持久化机制将内存数据保存到磁盘，重启时可以恢复数据，保证数据不丢失。

**核心矛盾**：持久化会带来 I/O 开销，需要在**数据安全性**与**性能**之间做权衡。

---

## 二、RDB（Redis Database Backup）— 快照持久化

### 2.1 什么是 RDB？

RDB 是将某一时刻 Redis 内存中的**全量数据**以二进制快照的形式保存到 `.rdb` 文件中。

### 2.2 触发方式

#### 手动触发

| 命令 | 说明 |
|------|------|
| `SAVE` | 在**主进程**中同步执行，会**阻塞**所有客户端请求，直到快照完成 |
| `BGSAVE` | fork 一个**子进程**异步执行快照，主进程继续处理请求（推荐） |

#### 自动触发（配置策略）

在 `redis.conf` 中配置 `save` 指令，满足条件则自动触发 `BGSAVE`：

```conf
# 格式：save <seconds> <changes>
save 900 1      # 900 秒内至少 1 次写操作
save 300 10     # 300 秒内至少 10 次写操作
save 60 10000   # 60 秒内至少 10000 次写操作
```

> 以上三条同时生效，满足任一条件即触发快照。

### 2.3 BGSAVE 原理：fork + Copy-on-Write

```
主进程
  │
  ├── fork() ──→ 子进程（继承主进程的内存页表，共享物理页）
  │                │
  │                └── 遍历内存，将数据写入临时 .rdb 文件
  │                        ↓
  │               原子替换旧的 .rdb 文件
  │
  └── 继续处理客户端请求
       │
       └── 若某内存页被修改 → 触发 Copy-on-Write（操作系统复制该页给子进程）
```

**Copy-on-Write（写时复制）**：
- fork 后父子进程共享同一份物理内存页
- 当父进程（主进程）修改某一内存页时，OS 才将该页复制一份给父进程独用
- 子进程持有 fork 时刻的内存快照，不受主进程后续写操作影响
- **内存开销**：极端情况下（大量写操作），内存使用量会接近翻倍

### 2.4 优缺点

| 维度 | 说明 |
|------|------|
| **优点** | 文件紧凑（二进制），**启动恢复速度快**；适合全量备份与容灾 |
| **优点** | BGSAVE 对主进程影响小，fork 期间有短暂阻塞 |
| **缺点** | **数据可能丢失**：两次快照之间的写操作在宕机时全部丢失 |
| **缺点** | fork 子进程本身有开销，数据量大时 fork 耗时较长（可能阻塞几百 ms） |

## 三、AOF（Append Only File）— 追加日志持久化

### 3.1 什么是 AOF？

AOF 以**追加写**的方式将每条写命令记录到 `.aof` 文件中（文本格式，可读性好）。重启时重放所有命令来恢复数据。

### 3.2 三种同步策略（fsync 频率）

| 配置值 | 行为 | 数据安全 | 性能 |
|--------|------|----------|------|
| `always` | 每条写命令都立即 fsync 到磁盘 | 最高（最多丢 1 条） | 最低（磁盘 I/O 频繁） |
| `everysec`（**默认推荐**） | 每秒 fsync 一次，异步线程执行 | 较高（最多丢 1 秒数据） | 较高 |
| `no` | 由操作系统决定何时 fsync | 最低（丢失数据不可控） | 最高 |

```conf
# redis.conf
appendonly yes
appendfsync everysec   # 推荐配置
```

### 3.3 AOF 重写机制

**问题**：随着时间推移，AOF 文件会越来越大（包含大量冗余命令，如 `INCR key` 执行了 1000 次）。

**解决**：AOF 重写（`BGREWRITEAOF`）——基于当前内存数据生成等价的**最简命令集**，替换旧的 AOF 文件。

```
触发方式：
  手动：BGREWRITEAOF 命令
  自动：配置 auto-aof-rewrite-percentage 和 auto-aof-rewrite-min-size
```

```conf
auto-aof-rewrite-percentage 100   # AOF 文件体积比上次重写后增长 100% 时触发
auto-aof-rewrite-min-size 64mb    # AOF 文件至少 64MB 才触发（防止文件太小频繁重写）
```

**重写过程**（同样使用 fork + COW）：
1. 主进程 fork 子进程
2. 子进程将当前内存快照转换为最简命令写入**新 AOF 文件**
3. 主进程将重写期间的新写命令追加到**AOF 重写缓冲区**
4. 子进程完成后，主进程将重写缓冲区的命令追加到新文件，原子替换旧文件

### 3.4 优缺点

| 维度 | 说明 |
|------|------|
| **优点** | **数据安全性高**，`everysec` 模式最多丢 1 秒数据 |
| **优点** | 文件可读（文本格式），可手动修复误操作（如 `FLUSHALL`） |
| **缺点** | 文件体积**远大于 RDB**（即使重写后） |
| **缺点** | **恢复速度慢**：需要重放所有命令 |
| **缺点** | 写入 I/O 开销比 RDB 大 |

## 四、混合持久化（Redis 4.0+）

### 4.1 原理

混合持久化将 **RDB 快照**和 **AOF 增量日志**结合在一个文件中：

```
AOF 文件结构（混合持久化）：
┌─────────────────────────────────────────────────┐
│  RDB 二进制数据（某时刻的全量快照）                │
│─────────────────────────────────────────────────│
│  AOF 格式命令（快照时刻之后的增量写命令）           │
└─────────────────────────────────────────────────┘
```

**触发时机**：在 AOF 重写时，子进程将内存数据以 RDB 格式写入新文件头部，主进程的重写缓冲区命令以 AOF 格式追加在后面。

### 4.2 配置

```conf
aof-use-rdb-preamble yes   # Redis 4.0+ 开启混合持久化
appendonly yes              # 必须开启 AOF
```

### 4.3 优势

| 对比项 | RDB | AOF | 混合持久化 |
|--------|-----|-----|------------|
| 启动恢复速度 | 快 | 慢 | **快**（RDB 部分加载） |
| 数据安全性 | 低（可能丢失分钟级数据） | 高（最多丢 1 秒） | **高**（AOF 增量部分） |
| 文件大小 | 小 | 大 | 中等 |

## 五、三种持久化方式全面对比

| 对比维度 | RDB | AOF | 混合持久化 |
|----------|-----|-----|------------|
| 持久化方式 | 全量快照 | 增量日志 | RDB + AOF 增量 |
| 数据丢失风险 | 分钟级 | 最多 1 秒 | 最多 1 秒 |
| 恢复速度 | 快 | 慢（重放命令） | 快 |
| 文件大小 | 小（二进制压缩） | 大（文本命令） | 中 |
| 文件可读性 | 不可读 | 可读 | 部分可读 |
| 对性能影响 | fork 时有短暂阻塞 | 持续 I/O 开销 | 综合两者 |
| 适用场景 | 容灾备份、可接受少量丢失 | 数据安全要求高 | **生产环境推荐** |

## 六、生产环境推荐配置

```conf
# redis.conf 生产推荐配置

# === RDB 配置 ===
save 900 1
save 300 10
save 60 10000
rdbcompression yes         # RDB 文件启用 LZF 压缩
rdbchecksum yes            # 文件尾部添加 CRC64 校验
dbfilename dump.rdb
dir /var/lib/redis/

# === AOF 配置 ===
appendonly yes
appendfilename "appendonly.aof"
appendfsync everysec       # 每秒 fsync，兼顾安全与性能
no-appendfsync-on-rewrite yes  # AOF 重写期间暂停 fsync（减少 I/O 竞争）
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb

# === 混合持久化 ===
aof-use-rdb-preamble yes   # Redis 4.0+ 强烈推荐开启
```

## 七、面试高频问题

**Q1：RDB 和 AOF 的核心区别是什么？**

> RDB 是全量快照，适合备份，恢复快但可能丢失较多数据；AOF 是增量命令日志，数据安全性高但文件大、恢复慢。生产环境推荐混合持久化，兼顾两者优点。

**Q2：BGSAVE 的原理是什么？fork 之后父子进程如何隔离？**

> BGSAVE 触发 fork() 创建子进程，子进程继承父进程的内存页表。利用操作系统的 **Copy-on-Write** 机制，fork 后父子进程共享同一份物理内存页，只有当父进程（主进程）修改某页时，OS 才复制该页给父进程独用，子进程始终看到的是 fork 时刻的内存快照。

**Q3：AOF 重写是否会影响主进程性能？**

> 同样使用 fork + COW，子进程负责重写。主进程将重写期间的新写命令缓存在 **AOF 重写缓冲区**，子进程完成后由主进程追加并原子替换文件。主进程仅在 fork 和最终替换时有极短暂的阻塞。

**Q4：为什么混合持久化既能快速恢复又能保证数据安全？**

> 混合持久化在 AOF 重写时，文件头部存储 RDB 格式的内存快照（加载速度快），后面追加快照时刻之后的 AOF 增量命令（数据安全），重启时先快速加载 RDB 部分，再重放少量 AOF 命令，兼顾了速度和安全。

**Q5：如果 Redis 同时开启了 RDB 和 AOF，重启时优先使用哪个？**

> 优先使用 **AOF** 文件恢复，因为 AOF 的数据更完整（丢失数据更少）。只有 AOF 未开启时才使用 RDB。
