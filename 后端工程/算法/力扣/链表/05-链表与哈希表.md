# 链表与哈希表：给链表补上寻址能力

> 链表的核心约束是只能顺序访问，无法直接跳到任意节点。这两道题的共同思路是：用哈希表存下节点的引用，让你能在 O(1) 时间内直接拿到任意节点——相当于给链表装上了随机访问的能力。

---

## [随机链表的复制](https://leetcode.cn/problems/copy-list-with-random-pointer/)（LeetCode 138）

### 问题

每个节点除了 `Next` 之外还有一个 `Random` 指针，可以指向链表中的任意节点或 `nil`。对整个链表做深拷贝。

```go
type Node struct {
    Val    int
    Next   *Node
    Random *Node
}
```

### 卡在哪里

顺序遍历链表，依次创建新节点时，节点 A 的 `Random` 可能指向后面还没创建的节点 B——你还没有 B 的副本，所以无法设置 `newA.Random`。

### 思路

把这件事拆成两次遍历：

**第一次**：只创建新节点，不管指针。同时用哈希表记下 `旧节点 → 新节点` 的映射。

**第二次**：对每个旧节点，通过哈希表找到它的新版本，再通过同样的方式找到 `Next` 和 `Random` 的新版本，把指针设好。

```go
func copyRandomList(head *Node) *Node {
    if head == nil {
        return nil
    }

    nodeMap := make(map[*Node]*Node)

    // 第一次遍历：创建所有新节点
    for curr := head; curr != nil; curr = curr.Next {
        nodeMap[curr] = &Node{Val: curr.Val}
    }

    // 第二次遍历：设置 Next 和 Random 指针
    for curr := head; curr != nil; curr = curr.Next {
        if curr.Next != nil {
            nodeMap[curr].Next = nodeMap[curr.Next]
        }
        if curr.Random != nil {
            nodeMap[curr].Random = nodeMap[curr.Random]
        }
    }

    return nodeMap[head]
}
```

哈希表的作用就是让第二次遍历时，`nodeMap[curr.Random]` 能直接拿到对应的新节点，而不需要重新遍历查找。

## [LRU 缓存](https://leetcode.cn/problems/lru-cache/)（LeetCode 146）

### 问题

设计一个容量为 `capacity` 的 LRU（最近最少使用）缓存，支持 `Get(key)` 和 `Put(key, value)`，两个操作都要求 O(1) 时间复杂度。超出容量时，淘汰最久没被访问的那个 key。

### 为什么需要两种结构

**只用哈希表**：`Get` 是 O(1)，但无法知道哪个 key 最久未被访问——哈希表没有顺序。

**只用链表**：可以维护访问顺序（最新的放头部，最旧的在尾部），但 `Get` 需要从头遍历查找，是 O(n)。

**两者结合**：哈希表存 `key → 节点指针`，`Get` 时 O(1) 定位节点；双向链表维护访问顺序，定位到节点之后可以 O(1) 把它移到头部（因为有 `prev` 指针，不需要遍历找前驱）。

### 为什么是双向链表

单向链表删除一个节点需要找到它的前驱，要 O(n) 遍历。双向链表每个节点都有 `prev`，删除是 O(1)。

### 整体设计

链表维护访问顺序：**头部 = 最近访问，尾部 = 最久未访问**。

- `Get(key)`：哈希表找到节点，把它移到链表头部，返回值。
- `Put(key, value)`：key 已存在就更新值并移到头部；不存在就创建新节点插到头部并写入哈希表。如果超出容量，删掉链表尾节点，同时从哈希表里删掉对应 key。

链表头尾各用一个虚拟节点，让每次插入和删除都不需要判断链表是否为空。

```go
type LRUCache struct {
    cap        int
    cache      map[int]*DLinkedNode
    head, tail *DLinkedNode
}

type DLinkedNode struct {
    key, val   int
    prev, next *DLinkedNode
}

func Constructor(capacity int) LRUCache {
    head := &DLinkedNode{}
    tail := &DLinkedNode{}
    head.next = tail
    tail.prev = head
    return LRUCache{
        cap:   capacity,
        cache: make(map[int]*DLinkedNode),
        head:  head,
        tail:  tail,
    }
}

func (l *LRUCache) Get(key int) int {
    if node, ok := l.cache[key]; ok {
        l.moveToFront(node)
        return node.val
    }
    return -1
}

func (l *LRUCache) Put(key int, value int) {
    if node, ok := l.cache[key]; ok {
        node.val = value
        l.moveToFront(node)
        return
    }
    node := &DLinkedNode{key: key, val: value}
    l.cache[key] = node
    l.addToFront(node)
    if len(l.cache) > l.cap {
        removed := l.removeLast()
        delete(l.cache, removed.key)
    }
}

func (l *LRUCache) addToFront(node *DLinkedNode) {
    node.prev = l.head
    node.next = l.head.next
    l.head.next.prev = node
    l.head.next = node
}

func (l *LRUCache) removeNode(node *DLinkedNode) {
    node.prev.next = node.next
    node.next.prev = node.prev
}

func (l *LRUCache) moveToFront(node *DLinkedNode) {
    l.removeNode(node)
    l.addToFront(node)
}

func (l *LRUCache) removeLast() *DLinkedNode {
    last := l.tail.prev
    l.removeNode(last)
    return last
}
```

`addToFront` 和 `removeNode` 是最容易写错的地方。静下来画一下指针改动的前后顺序，不要凭感觉写——先断开旧连接、再建立新连接，顺序错了会丢节点。

## 两道题的共同模式

两道题都在做同一件事：**用哈希表存节点引用，弥补链表无法直接寻址的缺陷**。

- 138：哈希表存 `旧节点 → 新节点`，让第二次遍历时能直接找到任意旧节点的副本。
- 146：哈希表存 `key → 节点指针`，让 `Get` 时能直接跳到链表中的任意位置并操作它。

链表提供顺序和结构，哈希表提供直达任意节点的能力，两者结合解决了单独用其中一个做不到的问题。
