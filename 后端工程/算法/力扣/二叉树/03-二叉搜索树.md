# 二叉搜索树：有序性藏在结构里

> BST 是二叉树里一个特殊的子集，它多了一个约束：有序性。这个约束使得所有 BST 问题都有一个共同的解题入口。

---

## BST 的本质：把排好序的东西装进树里

二叉搜索树的定义：对树里的每一个节点，它的**左子树所有节点的值都小于它**，**右子树所有节点的值都大于它**。

注意这个约束是**全局**的，不只是父子关系——不是"左孩子 < 根 < 右孩子"，而是"左子树里所有节点都 < 根"。这一点在验证 BST 那道题里会造成常见的错误。

```
合法 BST：            非法 BST（看起来像但不是）：
      5                         10
     / \                       /  \
    3   7          →          5    15
   / \ / \                   / \
  2  4 6  8                 1   12   ← 12 > 10，不能在 10 的左子树里
```

**BST 的核心性质**：对 BST 做**中序遍历**，得到的恰好是一个升序序列。

```
    5
   / \
  3   7
 / \ / \
2  4 6  8

中序遍历：2 → 3 → 4 → 5 → 6 → 7 → 8
```

这一性质是所有 BST 题的解题入口——几乎每道 BST 题都在以某种方式利用这个升序性。

---

## 将有序数组转换为二叉搜索树（LeetCode 108）

### 问题

给一个升序排列的整数数组，将其转换为一棵**高度平衡**的 BST（任意节点的左右子树高度差不超过 1）。

### 把中序的结论反过来用

既然 BST 的中序遍历是升序数组，那**从升序数组构造 BST 就是中序遍历的逆操作**。

要使树高度平衡，就要让左右子树的节点数量尽量相等——取**数组中点**作为根节点，左半边建左子树，右半边建右子树，这样自然地平衡。

```go
func sortedArrayToBST(nums []int) *TreeNode {
    if len(nums) == 0 { return nil }
    mid := len(nums) / 2
    root := &TreeNode{Val: nums[mid]}
    root.Left = sortedArrayToBST(nums[:mid])       // 左半边建左子树
    root.Right = sortedArrayToBST(nums[mid+1:])    // 右半边建右子树
    return root
}
```

这个函数的递归形状和 [[02-变形与构造]] 里的构造题一模一样：定根、划分、递归左右。区别是这里的"划分"直接来自数组的下标，不需要在序列里搜索。

## 验证二叉搜索树（LeetCode 98）

### 问题

判断一棵二叉树是否是合法的 BST。

### 最常见的错误：只看父子

很多人的第一反应是：对每个节点，检查 `node.Left.Val < node.Val < node.Right.Val`。但这是不够的。

```
      10
     /  \
    5    15
   / \
  1  12      ← 12 > 10，它不应该在 10 的左子树里
              但若只看父子关系，12 < 5 是假的，5 就会被判断为非法
              这道题的坑就在这里
```

正确的约束是：每个节点的值，必须在它**所有祖先**共同限定的一个范围内。

### 前序思维：把合法范围向下传递

这是一个典型的前序（自上而下）模式：父节点把合法范围传给子节点。

- 对左孩子：它的值必须在 `(min, current.Val)` 之间
- 对右孩子：它的值必须在 `(current.Val, max)` 之间
- 这个范围在每一层递归时不断收紧

```go
func isValidBST(root *TreeNode) bool {
    return validate(root, math.MinInt64, math.MaxInt64)
}

func validate(node *TreeNode, min, max int) bool {
    if node == nil { return true }
    // 当前节点的值必须严格在 (min, max) 区间内
    if node.Val <= min || node.Val >= max { return false }
    // 向左传：上界收紧为当前值
    // 向右传：下界收紧为当前值
    return validate(node.Left, min, node.Val) &&
           validate(node.Right, node.Val, max)
}
```

注意参数 `min` 和 `max` 会随着递归深度的增加越来越紧——这正是把"全局约束"拆解为"局部参数"的做法。这和链表里的虚拟头节点有类似的精神：为了让边界情况和普通情况用同一套逻辑处理，加一个"外层包装"。

### 中序写法：利用升序性

另一种思路是直接利用"BST 中序遍历是升序"这一性质：中序遍历时，记录前一个访问的值 `prev`，如果当前值 `<= prev`，则不是 BST。

```go
func isValidBST(root *TreeNode) bool {
    prev := math.MinInt64
    var inorder func(*TreeNode) bool
    inorder = func(node *TreeNode) bool {
        if node == nil { return true }
        if !inorder(node.Left) { return false }
        if node.Val <= prev { return false }  // 中序中当前值必须严格大于前一个
        prev = node.Val
        return inorder(node.Right)
    }
    return inorder(root)
}
```

两种写法都对。前序写法更直接地体现了 BST 的定义，中序写法更直接地体现了"中序=升序"这一核心性质。

## 二叉搜索树中第 K 小的元素（LeetCode 230）

### 问题

给定一棵 BST，找到其中第 k 小的元素。

### 中序即升序，数到第 K 个

如果对 BST 做中序遍历，得到的是升序序列。第 k 小的元素，就是这个序列里第 k 个元素。

不需要把所有元素都收集起来，找到就停：

```go
func kthSmallest(root *TreeNode, k int) int {
    result, count := 0, 0
    var inorder func(*TreeNode)
    inorder = func(node *TreeNode) {
        if node == nil || count >= k { return } // 找到后提前终止
        inorder(node.Left)
        count++
        if count == k {
            result = node.Val
            return
        }
        inorder(node.Right)
    }
    inorder(root)
    return result
}
```

这道题是"BST 中序 = 升序"这一性质最直接的应用。它不需要任何额外的技巧，只要在中序遍历里数到第 k 个就停。

## 三道题的共同线索

三道 BST 题背后都有同一条线：

```
有序数组 → BST（108）：利用"中序=升序"的逆操作，中点建根，天然平衡
验证 BST（98）：       BST 的约束是全局的，前序把合法范围向下传递
第 K 小（230）：       中序遍历直接给出升序序列，数到第 K 个即可
```

三道题对应了三个角度：**建树、验证、查询**。它们共同说明了一件事：**BST 的有序性是通过树的形状来维护的，而中序遍历是读取这个有序性的唯一钥匙**。

## 回顾整个二叉树专题

十二道题翻来覆去，核心就是在做两件事：

**① 在正确的遍历位置（前序/中序/后序）处理当前节点**
——前序让信息自上而下流动，后序让信息自下而上汇聚，中序在 BST 里有升序意义。

**② 把"对整棵树的问题"分解为"当前节点做什么 + 递归处理子树"**
——递归不是技巧，而是树的形状在代码中的直接映射。

每道题都是这两件事在不同场景下的组合。比较这九道链表题和这十二道二叉树题，链表的约束是"只能顺序游走"，树的约束是"只能通过 Left/Right 向下走"——链表用指针操控解决问题，树用递归分解解决问题。这是两种数据结构各自给出的解题语言。
