# TCP 流量控制

> **核心目标**：让发送方的发送速率不要太快，让接收方来得及接收（端到端的控制）。
>
> 区别于[[TCP拥塞控制]]（控制整个网络的拥塞），流量控制只关注通信双方之间的速率匹配。

---

## 一、滑动窗口机制

TCP 利用**滑动窗口**实现流量控制。接收方在每个 ACK 报文中通告自己当前的**接收窗口大小（rwnd）**，发送方据此调整发送速率。

### 发送窗口

```
|--- 已发送已确认 ---|--- 已发送未确认 ---|--- 可发送未发送 ---|--- 不可发送 ---|
                   ^                   ^                   ^
                  P1                  P2                  P3
```

- **P1**：发送窗口左沿（后沿），收到新 ACK 后右移
- **P3**：发送窗口右沿（前沿），由接收方通告的 rwnd 决定
- **P2**：下一个待发送字节的指针
- 发送窗口大小 = P3 - P1

### 接收窗口

接收方维护接收缓冲区，未被应用程序读取的数据占用缓冲区，**可用缓冲区大小 = rwnd**。

## 二、零窗口通知与死锁问题

### 场景

1. 接收方缓冲区已满，向发送方发送 **rwnd = 0** 的报文（零窗口通知）
2. 发送方暂停发送，等待接收方通告新窗口
3. 接收方缓冲区有空闲后，发送 **rwnd > 0** 的更新报文
4. 若该更新报文**丢失** → 发送方永久等待，接收方也不再发送 → **死锁**

### 解决方案：持续计时器（Persist Timer）

- 发送方收到零窗口通知后，启动**持续计时器**
- 计时器超时，发送方主动发送**零窗口探测报文**（仅携带 1 字节数据）
- 接收方收到探测报文后，回复当前 rwnd 值
- 若零窗口探测报文也丢失 → 设置**零窗口重传计时器**重复发送

## 三、糊涂窗口综合症（Silly Window Syndrome）

### 问题

接收方每次只能接收少量数据，却频繁通告小窗口，导致发送方持续发送大量小包，网络效率极低。

### 解决方案

**接收方**：Clark 解决方案
- 接收方在缓冲区空闲空间 < min(MSS, 缓冲区一半) 时，向发送方通告 **rwnd = 0**
- 直到缓冲区空闲达到足够大小，才通告真实窗口值

**发送方**：Nagle 算法（见[[TCP性能优化]]）

---

## 四、流量控制 vs 拥塞控制

| 对比项 | 流量控制 | 拥塞控制 |
|--------|----------|----------|
| **控制对象** | 发送方与接收方之间 | 整个网络 |
| **触发方式** | 接收方通告 rwnd | 发送方感知网络状态（丢包/超时） |
| **核心变量** | 接收窗口 rwnd | 拥塞窗口 cwnd |
| **发送窗口** | swnd = min(rwnd, cwnd) | swnd = min(rwnd, cwnd) |
| **目的** | 防止接收方缓冲区溢出 | 防止网络链路过载 |

> 💡 **发送窗口 = min(接收窗口, 拥塞窗口)**，两者共同约束发送方的实际发送速率。
