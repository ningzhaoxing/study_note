# TCP 性能优化

---

## 一、Nagle 算法

### 问题背景

应用程序可能频繁发送很小的数据（如 telnet 逐字符发送），每个字节的数据都封装成一个 TCP 报文，产生大量**小包**（小于 MSS），造成网络拥挤。

### 算法规则

只有满足以下**任意一个条件**，才允许发送报文：
1. 数据大小 ≥ MSS（最大报文段大小）
2. 已收到之前所有数据的 ACK

否则将小数据缓冲，等待凑够足够大的数据再发送。

### 效果

- 减少网络中小包数量，提升带宽利用率
- 代价：引入发送延迟

### 何时关闭

对于**实时性要求高**的场景（如在线游戏、鼠标操作同步），需关闭 Nagle 算法：

```java
socket.setTcpNoDelay(true);   // Java
```

## 二、延迟 ACK（Delayed ACK）

### 规则

接收方**不立即**回复 ACK，而是：
- 等待最多 **200ms**，若期间有数据要发送，则将 ACK 与数据一起捎带发送（**捎带确认**）
- 或者收到**第 2 个**数据报文时，立即发送 ACK

### 目的

减少纯 ACK 报文的数量，提升网络效率（每个 ACK 都有 40 字节首部开销）。

---

## 三、Nagle 算法 + 延迟 ACK 的冲突

这是一个经典的性能陷阱：

```
发送方（开启 Nagle）          接收方（开启延迟 ACK）
  │ 发送小包 A ──────────────▶ │
  │ 等待 ACK 才能发 B          │ 等 200ms 再回 ACK
  │                            │（等待是否有数据可以捎带）
  │                200ms 后    │
  │ ◀─────────────── ACK ───── │
  │ 发送 B                     │
```

**结果**：每次交互都有 200ms 的延迟，严重影响性能（如 HTTP 请求-响应模型）。

**解决方案**：关闭 Nagle 算法（`TCP_NODELAY`）或关闭延迟 ACK（`TCP_QUICKACK`）。

## 四、TCP Fast Open（TFO）

### 背景

标准 TCP 三次握手需要 1.5 个 RTT 才能开始发送数据，对于短连接（如 HTTP）浪费严重。

### TFO 原理

**第一次连接**（正常三次握手，同时获取 Cookie）：
1. 客户端发送 SYN + `TFO` 选项（请求 Cookie）
2. 服务端回复 SYN+ACK + **TFO Cookie**（服务端根据客户端 IP 加密生成）
3. 客户端保存 Cookie，完成三次握手

**后续连接**（直接携带数据，节省 1 个 RTT）：
1. 客户端发送 SYN + **TFO Cookie + HTTP 请求数据**
2. 服务端验证 Cookie 合法，立即处理数据，回复 SYN+ACK + 响应数据
3. 数据在三次握手完成前就已开始处理

### 效果

- 对重复连接减少 1 个 RTT 的延迟
- 需要内核和应用层支持

## 五、TCP KeepAlive（保活机制）

### 问题背景

连接建立后，若一方长时间无数据传输，但连接并未正常关闭（如对端宕机、网络中断），另一方无法感知，连接成为**僵尸连接**，浪费资源。

### 工作原理

TCP KeepAlive 在连接空闲一段时间后，自动发送**探测报文（Keep-Alive Probe）**：

| 参数 | 默认值 | 含义 |
|------|--------|------|
| `tcp_keepalive_time` | 7200s（2小时） | 连接空闲多久后开始发探测包 |
| `tcp_keepalive_intvl` | 75s | 每次探测包的间隔 |
| `tcp_keepalive_probes` | 9 次 | 探测失败多少次后关闭连接 |

若连续 9 次探测均无响应 → 判定连接死亡 → 关闭连接，释放资源。

### 应用层 vs TCP 层保活

- TCP KeepAlive 是**内核级别**，2小时才触发，粒度太粗
- 生产中通常在**应用层**实现心跳机制（如每 30 秒发送心跳包），更灵活可控

## 六、RST 报文

### 什么是 RST

RST（Reset）标志位置 1 的报文，表示**异常关闭连接**，与 FIN 的正常四次挥手关闭不同。

### 触发 RST 的场景

| 场景 | 说明 |
|------|------|
| **连接不存在** | 收到一个报文，但本地没有对应连接（如端口未监听）|
| **强制关闭** | 调用 `socket.setLinger(true, 0)` 关闭 socket，未发送完的数据被丢弃 |
| **TIME_WAIT 被侵占** | 新连接的 SYN 序列号在旧连接残留数据范围内 |
| **中间件异常** | 防火墙、负载均衡强制断开连接 |
| **对端程序崩溃** | 进程直接被 kill -9，未来得及发送 FIN |

### RST vs FIN 区别

| | FIN | RST |
|---|---|---|
| **含义** | 正常关闭，半关闭 | 异常中止，立即关闭 |
| **数据保证** | 等待数据发送完毕 | 丢弃发送缓冲区中的数据 |
| **对方感知** | 收到 FIN 后可以继续发送数据 | 收到 RST 后连接立即终止 |
| **TIME_WAIT** | 主动关闭方需等待 2MSL | 不需要 TIME_WAIT |
