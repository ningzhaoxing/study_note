# 拥塞控制

避免发送方过度发包导致网络拥塞。
## 过程
四个量：发送窗口(=min(接收窗口,拥塞窗口))、接收窗口、拥塞窗口、慢开始门限。

1. 初始：
	进行**慢开始**算法：
	拥塞窗口为1，拥塞窗口成倍增加。

2. 当拥塞窗口>慢开始门限：
	进行**拥塞避免**算法：
	拥塞窗口按线性增加

3. 当发送方收到三个重复的ACK
	进行**快重传**、**快恢复**算法：
	发送方将慢开始门限和拥塞窗口调整问当前窗口的一半，开始执行拥塞避免算法。
	***作用***：表明轻度拥塞，TCP通效率和稳定性。过**快速恢复**（窗口减半 + 拥塞避免）平衡

5. 发生超时重传(发送方报文段丢失)：***TCP重置***
	将**慢开始门限**更新为发生**拥塞时拥塞窗口**的一半；
	将**拥塞窗口**减少为1，并重新执行**慢开始算法**
	*作用*：超时说明当前网络拥塞比较严重，可以确保发送方在重新开始发送数据时，不会因发送速率过快而再次导致拥塞。这种做法有助于网络逐步恢复，提高网络的性能和可靠性
##  其他
1. 拥塞控制解决了什么问题？
	**问题**：在没有拥塞控制的情况下，当网络中数据流量过大时，网络中的路由器和交换机等设备的缓冲区会被迅速填满。
	一旦填满，后续到达的数据包就会丢弃，导致大量数据丢失，导致引发网络中的重传机制，进一步加剧网络拥塞，最终导致网络瘫痪。
	**解决方式**：tcp通过拥塞控制算法来动态调整发送方的发送速率，根据网络拥塞情况来控制发送到网络中的数据量。


# 三次握手（四元组：源ip、目的ip、源端口号、目的端口号）

![[TCP三次握手.png]]

## 理解框架：双向验证

握手的本质目标是：**双方都要确认「我能发 + 对方能收」这件事成立**。

把它拆成两个信道的逐步验证：

| 次数 | 报文 | 验证了什么 |
|------|------|-----------|
| 第1次（客→服 SYN） | 客户端发出去了 | 但还不知道服务端收没收到 |
| 第2次（服→客 SYN+ACK） | 服务端收到了 | 客→服 ✅；客户端收到回复，服→客 ✅ |
| 第3次（客→服 ACK） | 服务端收到确认 | 服务端也知道了 服→客 ✅ |

- **为什么不是 2 次**：第 2 次后客户端已经确认双向通畅，但**服务端还不知道「服→客」方向是否通畅**，贸然建立连接会产生无效连接
- **为什么不是 4 次**：第 2 次可以把「ACK 客户端的 SYN」和「自己的 SYN」合并，3 次是建立可靠连接的最少次数

> 💡 记忆口诀：**3次 = 双方各发 1 个 SYN + 对方各确认 1 次，最少需要 3 个来回**

## 完整握手流程

```
客户端                                        服务端
CLOSED                                        LISTEN
  │                                             │
  │──── SYN=1, seq=x ──────────────────────────▶│  SYN_RCVD
  │     (第一次握手：客户端发起连接请求)          │
  │                                             │
  │◀─── SYN=1, ACK=1, seq=y, ack=x+1 ──────────│
  │     (第二次握手：服务端确认，并发起自己的SYN) │
SYN_SENT→ESTABLISHED                           │
  │                                             │
  │──── ACK=1, seq=x+1, ack=y+1 ───────────────▶│  ESTABLISHED
  │     (第三次握手：客户端确认服务端的SYN)       │
```

> **SYN 和 FIN 报文虽然不携带数据，但都会消耗一个序列号**（接收方需要对其确认）。

## 握手时协商了哪些参数？

三次握手不只是建立连接，双方还通过 TCP Options 字段协商传输参数：

| 选项 | 说明 |
|------|------|
| **MSS**（最大报文段大小） | 双方通告自己能接受的最大数据段大小，取两者较小值作为实际 MSS |
| **Window Scale**（窗口扩大因子） | 将窗口大小字段左移 N 位，支持超过 65535 字节的大窗口（高带宽网络必需） |
| **SACK-permitted** | 声明自己支持选择性确认，后续传输可使用 SACK |
| **Timestamps** | 用于精确计算 RTT 和防止序号回绕（PAWS 机制） |

## ISN（初始序列号）为什么要随机？

每次建立连接时，双方各自随机生成初始序列号（ISN），而不是从固定值（如 0）开始。

**原因：防止历史报文被新连接接收**

若 ISN 固定，旧连接因网络延迟残留的报文，在四元组相同的新连接建立后到达，其序号可能恰好落在新连接的接收窗口内，被错误接收。

ISN 随机化后，旧连接残留报文的序号几乎不可能落在新连接的接收窗口内，可有效避免数据错乱。

> Linux 的 ISN 生成算法：`ISN = M + F(src_ip, src_port, dst_ip, dst_port, secret_key)`，其中 M 是时钟计数器，每隔一段时间自增。

## 为什么是三次，而不是两次？

为了防止旧的重复无效连接

**场景：**
客户端向服务端发送 SYN 报文后，客户端宕机了，因为网络拥塞导致 SYN 被滞留在网络中。客户端重启后重新向服务端建立连接，旧的 SYN 此时才到达服务端。

**两次握手的问题：**
服务端收到旧 SYN 后直接建立连接（进入 ESTABLISHED），但客户端对这个旧连接毫不知情，不会发送数据，却占用了服务端资源。

**三次握手的解决：**
服务端回复 SYN+ACK 后，客户端收到该响应，能识别出这是对旧 SYN 的回复（序号对不上），于是发送 RST 报文通知服务端中止连接，避免建立无效连接。

## 为什么不是四次握手？

因为服务端可以将 **SYN 和 ACK 合并在同一个报文**中发送（第二次握手），没有必要拆成两次。三次已经是能够建立可靠连接的最少次数。

## 若第二次握手后，客户端宕机
### 此时会浪费的资源

1. 端口号被占用
2. 内存资源
3. CPU资源
4. 网络带宽

### 处理措施
1. **超时重传机制**
	服务端重传 SYN+ACK，当重传次数超过 `tcp_synack_retries`（默认 5 次）后，服务端主动丢弃并关闭连接。
2. **半连接队列超时**
	操作系统维护半连接队列，若半连接在超时时间内未完成三次握手，会被自动丢弃。

> 相关内容：[[TCP安全-SYN攻击]] — 半连接队列被打满的 SYN Flood 攻击及 SYN Cookie 防御
# 四次挥手

![[秋招/计网面经/传输层/TCP.UDP/assets/TCP/file-20250517171239935.png]]

## 理解框架：全双工独立关闭

TCP 是**全双工**的，可以理解成**两条独立的单行道**：

```
客→服 方向：  ══════════════════════════▶
服→客 方向：  ◀══════════════════════════
```

关闭连接 = **两条路各自独立关闭**，每条路的关闭都需要 FIN + ACK，合起来就是 4 次：

```
客→服 关闭：  客发 FIN（第1次）→ 服回 ACK（第2次）
服→客 关闭：  服发 FIN（第3次）→ 客回 ACK（第4次）
```

**为什么第 2 次和第 3 次不能合并**：服务端收到客户端 FIN 时，**可能还有数据没发完**。必须先 ACK 说"收到了"，等把剩余数据发完后，再发自己的 FIN——这中间有时间差，无法合并。

> 💡 特殊情况：如果服务端刚好没有数据要发了，第 2 次 ACK 和第 3 次 FIN **可以合并**，变成三次挥手完成关闭。

## 完整挥手流程

```
客户端（主动关闭）                              服务端（被动关闭）
ESTABLISHED                                   ESTABLISHED
  │                                              │
  │──── FIN=1, seq=u ───────────────────────────▶│
  │     (第一次挥手：客户端通知不再发送数据)        │  CLOSE_WAIT
FIN_WAIT_1                                       │
  │◀─── ACK=1, ack=u+1 ─────────────────────────│
  │     (第二次挥手：服务端确认，但可继续发数据)    │
FIN_WAIT_2                                      │
  │             [服务端处理剩余数据]               │
  │◀─── FIN=1, ACK=1, seq=w, ack=u+1 ───────────│
  │     (第三次挥手：服务端也准备好关闭了)          │  LAST_ACK
TIME_WAIT                                       │
  │──── ACK=1,n ack=w+1 ─────────────────────────▶│
  │     (第四次挥手：客户端确认服务端的FIN)         │  CLOSED
  │（等待 2MSL）                                  │
CLOSED
```

## 为什么是四次，而不是三次？

握手时服务端可以将 ACK 和 SYN 合并，但挥手时**服务端的 ACK 和 FIN 通常不能合并**。

原因是 TCP 支持**半关闭（Half-Close）** 状态：

- 客户端发送 FIN，表示"我不再发送数据了"，但**并不意味着服务端也要立刻停止发送**
- 服务端先回 ACK 确认，此时进入 `CLOSE_WAIT` 状态，可以继续向客户端发送未传完的数据
- 直到服务端应用程序调用 `close()` 处理完剩余数据，才发送 FIN（第三次挥手）

因此 ACK 和 FIN 之间存在时间间隔，必须分开发送，所以需要四次。

> **特殊情况（三次挥手）**：若服务端在收到 FIN 时，应用层也恰好没有数据要发了，可以将第二次 ACK 和第三次 FIN 合并成一个报文，变成"三次挥手"完成关闭。

## 各次挥手丢失的处理

> 💡 **一个规则记全部**：只有**消耗序列号的报文（FIN/SYN/数据）** 才需要被确认，才会触发重传；**纯 ACK 不重传**。
>
> 因此四次挥手中，真正会被重传的只有第 1 次（客户端的 FIN）和第 3 次（服务端的 FIN）。
> 第 2、4 次是纯 ACK，不会被重传——但丢失后，对端会重传 FIN，从而间接触发重新确认。

### 第一次挥手丢失

客户端重传 FIN，每次重传等待时间是上一次的 2 倍（指数退避）。超过 `tcp_orphan_retries`（默认 5 次）后，客户端直接断开连接。

### 第二次挥手丢失

客户端处于 `FIN_WAIT_1` 状态，收不到 ACK 会触发 FIN 重传。超过最大重传次数后，客户端断开连接。

### 第三次挥手丢失

服务端处于 `LAST_ACK` → 重传 FIN。超过最大重传次数后，服务端主动断开。
同时客户端的 `FIN_WAIT_2` 状态有时间限制（Linux 默认 `tcp_fin_timeout` = 60 秒），超时后客户端也会断开。

### 第四次挥手丢失

客户端进入 `TIME_WAIT`，等待 2MSL。若第四次 ACK 丢失，服务端会重传 FIN，客户端重新发 ACK 并重置 2MSL 计时器。服务端超过最大重传次数后关闭；客户端等待 2MSL 后关闭。

## 为什么要有 `TIME_WAIT` 状态

### 1. 防止历史连接数据被新连接错误接收

若没有 `TIME_WAIT`，旧连接关闭后立即使用相同四元组建立新连接，网络中延迟的旧报文可能到达新连接，若序号恰好在接收窗口内，会被错误接收，导致数据污染。

`TIME_WAIT` 持续 2MSL，确保网络中所有旧报文已自然消亡后，才允许相同四元组的连接出现。

### 2. 保证被动关闭方能被正确关闭

若没有 `TIME_WAIT`，客户端发完第四次 ACK 直接进入 `CLOSED`。若该 ACK 丢失，服务端重传 FIN 时，客户端已关闭，会回复 RST 导致服务端异常中止。

`TIME_WAIT` 期间客户端仍存活，可以对服务端重传的 FIN 重新发 ACK，保证连接正常关闭。

## 为什么 `TIME_WAIT` 等待时间是 2MSL

- `MSL`（Maximum Segment Lifetime）：报文在网络中存在的最长时间（Linux 默认 30s，因此 2MSL = 60s）
- `TTL`：报文经过的路由跳数（与 MSL 不同，MSL 是时间维度）

2MSL 覆盖了报文在网络中一个来回的最大生存时间：
- 第 1 个 MSL：等待第四次 ACK 在网络中消亡（若丢失，服务端在此期间会重传 FIN）
- 第 2 个 MSL：等待服务端重传的 FIN 到达客户端并被处理

## TIME_WAIT 过多的问题与解决

### 问题场景

高并发短连接服务（如 HTTP/1.0 每次请求都新建连接），主动关闭方会积累大量 `TIME_WAIT` 状态，导致**本地端口耗尽**（端口范围 1024~65535，约 6 万个），无法创建新连接。

### 解决方案

| 方案 | 说明 | 风险 |
|------|------|------|
| **`SO_REUSEADDR`** | 允许 TIME_WAIT 状态的端口被新连接复用（需序列号验证） | 低，推荐 |
| **`tcp_tw_reuse`** | 允许将 TIME_WAIT 状态的连接重用于新的出站连接（需开启时间戳） | 低，仅用于客户端 |
| **长连接（Keep-Alive）** | HTTP/1.1 默认开启，复用连接，减少握手/挥手次数 | 无，从根本上减少短连接 |
| **连接池** | 应用层维护连接池，复用已建立的 TCP 连接 | 无，推荐 |
| ~~`tcp_tw_recycle`~~ | ~~快速回收 TIME_WAIT~~，**已在 Linux 4.12 移除** | 高，NAT 环境下有 bug |

# 理解框架：状态机存在的意义

> 状态机存在的意义只有一句话：**「我现在在等什么，收到什么报文该怎么处理」**

没有状态机会发生什么？举个例子：

> 服务端收到一个 SYN 报文，它怎么知道这是**新连接请求**还是**历史遗留的旧 SYN**？

靠状态来判断：
- 当前是 `LISTEN` → 这是新连接，正常处理
- 当前是 `ESTABLISHED` → 这是旧的重复 SYN，发 RST 拒绝

**状态 = 当前上下文 + 期望收到的下一个报文**，有了它，TCP 才能在丢包、重传、乱序等任何异常情况下做出正确判断。

## 关键路径速记

```
建立连接（客户端）：  CLOSED → SYN_SENT → ESTABLISHED
建立连接（服务端）：  LISTEN → SYN_RCVD → ESTABLISHED

关闭连接（主动方）：  ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
关闭连接（被动方）：  ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
```

> 💡 主动关闭方经历 **3 个等待状态**，被动关闭方只有 **2 个**。
> 原因：主动方需要等对方两个方向都关完，还要保证最后一个 ACK 能到达（TIME_WAIT）。

详细状态转换及 CLOSE_WAIT 堆积问题见：[[TCP连接状态机]]
