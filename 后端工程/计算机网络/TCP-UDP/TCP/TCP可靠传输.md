# TCP 可靠传输

> TCP 通过以下机制共同保证数据的**无差错、不丢失、不重复、按序到达**：
> 校验和 · 序列号 · 确认应答 · 重传机制 · SACK

---

## 一、序列号与确认号

### 序列号（Seq）

- TCP 将发送的数据流中的**每一个字节**都编上序号
- 本报文段首部中的序列号 = 该报文段数据载荷**第一个字节**的编号
- 序号空间为 32 位，循环使用（用完后回到 0）

### 确认号（Ack）

- 表示接收方**期望收到的下一个字节**的序号
- 隐含含义：确认号 N 之前的所有字节（0 ~ N-1）已全部正确接收
- 即**累计确认**：一个 ACK 确认了它之前所有的数据

### 示例

```
发送方 → [Seq=1, Len=100] → 接收方
发送方 ← [Ack=101]        ← 接收方  （表示已收到 1~100，期望下一个从 101 开始）
```

## 二、ARQ 协议（自动重传请求）

### 停止等待 ARQ（Stop-and-Wait ARQ）

- 发送方发送一个分组后，必须等待 ACK 才能发下一个
- 缺点：信道利用率极低

### 连续 ARQ（Continuous ARQ）

发送方可以连续发送多个分组，不需要等待逐一确认，利用**滑动窗口**控制在途报文数量。

#### Go-Back-N（回退 N 帧）

- 接收方只接受**按序到达**的报文，失序报文直接丢弃
- 若第 N 个报文丢失，发送方需**重传从第 N 帧开始的所有后续帧**
- 缺点：网络状况差时大量重传，浪费带宽

#### Selective Repeat（选择重传）

- 接收方可以**缓存失序报文**，等缺失的报文到达后再按序交付
- 发送方**只重传丢失的报文**，不重传已正确接收的
- 需要 SACK（选择性确认）支持

## 三、超时重传时间（RTO）的计算

超时重传时间（Retransmission Timeout）过大或过小都有问题：
- **RTO 过大**：等待时间长，重传慢，效率低
- **RTO 过小**：频繁触发不必要的重传，加重网络负担

### RTT 采样与平滑

TCP 动态测量往返时延 RTT，并用**加权移动平均**得到平滑值：

```
SRTT = (1 - α) × SRTT + α × RTT_sample    （α ≈ 0.125）
RTTVAR = (1 - β) × RTTVAR + β × |SRTT - RTT_sample|  （β ≈ 0.25）

RTO = SRTT + 4 × RTTVAR
```

### 指数退避

每次重传超时后，RTO **翻倍**（指数退避），直到重传成功或达到最大次数。

## 四、SACK（选择性确认）

### 问题背景

累计确认无法告知发送方哪些报文段已经收到，哪些没有收到。
例如：收到了 1-500、601-1000，丢失了 501-600，ACK=501 无法表达"601-1000 已收到"。

### SACK 机制

在 TCP 选项字段携带 **SACK 块**，每块指定一段已收到数据的范围：

```
ACK=501, SACK=[601-1000]
→ 告诉发送方：501 之前都收到了，601-1000 也收到了，只需重传 501-600
```

- 最多携带 **4 个 SACK 块**（选项字段最大 40 字节）
- 需要双方在握手阶段通过 `SACK-permitted` 选项协商开启
- 配合**选择重传**策略，大幅减少不必要的重传

## 五、超时重传 vs 快重传

| 对比项 | 超时重传 | 快重传 |
|--------|----------|--------|
| **触发条件** | 超时计时器到期，未收到 ACK | 收到 3 个重复 ACK |
| **反应速度** | 慢（需等待超时） | 快（立即重传） |
| **网络判断** | 认为网络严重拥塞 | 认为是轻度拥塞（丢单包） |
| **拥塞窗口处理** | cwnd 重置为 1，执行慢开始 | cwnd 减半，执行快恢复 |
| **触发概率** | 相对较少 | 更常见 |

> 💡 收到**3 个重复 ACK** 而不是 1 个，是为了区分"报文乱序到达"（通常 1~2 个重复 ACK）和"报文丢失"（持续收到重复 ACK）。
