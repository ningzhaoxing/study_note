# HTTP/1.1 vs HTTP/2 vs HTTP/3 深度对比

> 整体演进的时间线和背景见 [[HTTP版本演进]]，本文聚焦三个版本的技术细节对比。

---

## HTTP/1.1 的痛点

### 问题一：队头阻塞（Head-of-Line Blocking）

HTTP/1.1 虽然支持管线化（Pipelining），但服务器**必须按顺序响应**。如果第一个请求很慢，后续请求全部被卡住。

```
客户端:  请求1 → 请求2 → 请求3   (可以连续发)
服务端:  响应1 ————→ 响应2 → 响应3  (必须按顺序)
             ↑
        如果响应1慢，2和3都阻塞在这里
```

**浏览器的变通方案**：对同一域名开多个 TCP 连接（Chrome 最多 6 个），但这带来更多握手开销。

### 问题二：Header 冗余

每个请求都要携带完整 Header，大量重复信息（如 Cookie、User-Agent）以**明文**重复传输，浪费带宽。

```http
# 每个请求都带这些重复的头
GET /image1.png HTTP/1.1
Host: example.com
User-Agent: Mozilla/5.0 ...（很长）
Cookie: session=abc123; user=xxx; ...（很长）
Accept-Encoding: gzip, deflate, br
...
```

## HTTP/2 的解决方案

### 多路复用（核心改进）

**一个 TCP 连接，并行传输多个请求/响应**，每个请求/响应是一个独立的"流（Stream）"。

```
           TCP 连接（一条）
    ┌─────────────────────────────┐
    │  Stream 1: GET /index.html  │
    │  Stream 2: GET /style.css   │ ← 同时在传
    │  Stream 3: GET /script.js   │
    └─────────────────────────────┘
```

流之间互不干扰，解决了**应用层的队头阻塞**。

### 二进制分帧

HTTP/2 把所有传输内容拆成小的**帧（Frame）**，帧有类型标识（HEADERS 帧、DATA 帧等），可以任意交错发送，接收端按 Stream ID 重组。

```
HTTP/1.1（文本，按顺序）:
[请求1完整报文] → [请求2完整报文]

HTTP/2（二进制帧，可交错）:
[1:HEADERS][2:HEADERS][1:DATA][2:DATA][3:HEADERS][1:DATA]...
```

### 头部压缩（HPACK）

**静态表**：预定义 61 个常用头部字段，用索引代替字符串。
**动态表**：运行时动态更新，记录已传输过的头部字段。
**Huffman 编码**：对字符串进行熵编码压缩。

```
第一次请求：发送完整头部，同时写入动态表
后续请求：相同的头部只发一个索引号（1-2 字节）
```

### 服务器推送（Server Push）

服务器可以在客户端请求 HTML 时，**主动推送** CSS、JS 等资源，无需客户端解析 HTML 再发请求。

```
客户端: GET /index.html
服务端: 响应 index.html
       + 主动推送 style.css
       + 主动推送 app.js
```

> 实践中 Server Push 因难以精确控制，已被部分浏览器弃用（Chrome 104 起移除）。

### HTTP/2 的遗留问题：TCP 队头阻塞

HTTP/2 解决了**应用层**的队头阻塞，但没有解决 **TCP 层**的队头阻塞：

```
TCP 丢包场景：
流1:  [帧A] [帧B] [帧C 丢失] [帧D]
流2:  [帧E] [帧F]
流3:  [帧G]

TCP 必须等待帧C重传成功后，才能处理后续所有帧
→ 流2和流3的帧E/F/G全部被阻塞，尽管它们已经到达
```

在高丢包网络（如移动网络）中，HTTP/2 的性能甚至可能比 HTTP/1.1 更差。

## HTTP/3 的解决方案

### 抛弃 TCP，使用 QUIC

**QUIC（Quick UDP Internet Connections）** = UDP + 在应用层实现可靠传输

```
HTTP/1.1 & HTTP/2:   HTTP → TLS → TCP → IP
HTTP/3:              HTTP → QUIC(含TLS) → UDP → IP
```

### 彻底解决队头阻塞

QUIC 中每个流有**独立的流量控制**，一个流丢包，只重传该流，不影响其他流。

```
QUIC 丢包场景：
流1: [帧A] [帧B] [帧C 丢失] [帧D]
流2: [帧E] [帧F]

流1 等待帧C重传
流2 的帧E/F 继续正常处理 ✅
```

### 更快的连接建立

| | HTTP/2 (TCP+TLS1.3) | HTTP/3 (QUIC) |
|--|---------------------|---------------|
| **首次连接** | TCP(1RTT) + TLS(1RTT) = **2 RTT** | **1 RTT** |
| **复用连接** | 0-RTT (TLS Session) | **0 RTT** |

QUIC 将 TLS 1.3 握手**内嵌**到连接建立过程中，两件事合并为一件。

### 连接迁移

TCP 连接由 **源IP + 源端口 + 目标IP + 目标端口** 四元组标识。
QUIC 连接由 **64位连接ID** 标识。

```
切换网络（WiFi → 4G，IP 变了）：
TCP: 连接断开，需要重新握手
QUIC: 连接不中断，用新 IP 继续传输 ✅
```

## 三版本综合对比

| 维度 | HTTP/1.1 | HTTP/2 | HTTP/3 |
|------|:--------:|:------:|:------:|
| **发布年份** | 1997 | 2015 | 2022 |
| **底层协议** | TCP | TCP | UDP+QUIC |
| **传输格式** | 文本 | 二进制 | 二进制 |
| **连接复用** | ❌ (多TCP) | ✅ (一TCP) | ✅ (QUIC) |
| **多路复用** | ❌ | ✅ | ✅ |
| **头部压缩** | ❌ | ✅ HPACK | ✅ QPACK |
| **服务器推送** | ❌ | ✅ | ✅ |
| **应用层队头阻塞** | ✅ 有 | ❌ 无 | ❌ 无 |
| **传输层队头阻塞** | ✅ 有 | ✅ 有 | ❌ 无 |
| **连接建立延迟** | 1~2 RTT | 2 RTT | 0~1 RTT |
| **连接迁移** | ❌ | ❌ | ✅ |
| **加密** | 可选(HTTPS) | 实际必须 | 必须(内置) |

## 面试常见问题

### Q：HTTP/2 的多路复用和 HTTP/1.1 的管线化有什么区别？
**A**：
- 管线化：客户端可以连续发请求，但服务端**必须按顺序响应**，仍有队头阻塞
- 多路复用：基于二进制帧，请求和响应可以**任意交错**，互不阻塞

### Q：HTTP/2 一定比 HTTP/1.1 快吗？
**A**：不一定。在高丢包网络中，HTTP/2 的所有流共享一个 TCP 连接，一旦 TCP 丢包，所有流都被阻塞，可能比 HTTP/1.1 开多个连接更差。

### Q：HTTP/3 为什么选择 UDP？
**A**：TCP 队头阻塞是内核层面的问题，无法在应用层修复。选择 UDP 是为了**在用户态自己控制传输**，实现更灵活的可靠传输机制（QUIC），同时内置 TLS，减少握手延迟。
